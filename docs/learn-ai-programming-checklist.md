# 学 AI 编程：Firefly 博客实践执行清单

> 目标：通过搭建、修改和部署 Firefly 博客，学会一套可重复的 AI 编程与问题解决方法。
>
> 使用方法：一次只完成一个未勾选项。完成后勾选，并在“结果记录”中留下命令、链接、截图或遇到的问题。
>
> 说明：参考文章的部署部分主要介绍 Cloudflare、Vercel 和 EdgeOne；本清单按照“部署到 GitHub Pages，访问 `github.io`”的目标进行了调整。

## 一、完成标准

- [ ] 能用一句话说明自己要解决的问题：搭建一个可写文章的 Firefly 博客，并部署到 GitHub Pages。
- [ ] 能在 ChatGPT 桌面端中清晰描述问题、请求拆分任务和分析报错。
- [ ] 已安装并能使用 ChatGPT、Node.js、pnpm、Git、GitHub Desktop 和 VS Code。
- [ ] 已 Fork Firefly 官方仓库，并将自己的仓库克隆到本地。
- [ ] 能在本地启动 Firefly，并在浏览器打开首页。
- [ ] 已修改至少一项站点配置。
- [ ] 已新增至少一篇 Markdown 文章。
- [ ] 已将“从安装到部署”的过程拆成多篇文章并加入网站。
- [ ] 已将代码推送到 GitHub，并通过 GitHub Actions 部署到 `github.io`。
- [ ] 能打开线上网址，并确认首页、文章页、静态资源和文章链接正常。

## 二、自己的问题解决系统

### 2.1 先把模糊问题说清楚

每次开始前，先填写下面的五句话：

- [ ] 我想完成的结果是：
- [ ] 当前已知条件是：
- [ ] 我手上的输入是：
- [ ] 我希望得到的输出是：
- [ ] 完成的判断标准是：

例如，不写“我要学 AI 编程”，改写成：

> 我想使用 AI 辅助，把 Firefly 博客在本地运行起来，新增文章，最后部署到自己的 GitHub Pages 网站。

### 2.2 识别问题的元素

对每个问题，逐项写出：

- [ ] 目标：最终要得到什么。
- [ ] 输入：已有文件、账号、软件、资料或数据。
- [ ] 输出：文件、命令结果、网页、文章或部署地址。
- [ ] 约束：操作系统、版本、目录、时间、权限或平台限制。
- [ ] 依赖：必须先完成的安装、账号、配置或前置知识。
- [ ] 风险：可能报错、丢失内容、版本不匹配或权限不足的地方。
- [ ] 验证：用什么现象或命令证明这一步完成了。

### 2.3 分层与分布

把一个大问题分成四层：

- [ ] 目标层：我要做成什么。
- [ ] 阶段层：完成目标需要经过哪些阶段。
- [ ] 任务层：每个阶段要完成哪些任务。
- [ ] 最小执行单位：下一步只做一个动作，并且能马上验证。

把最小执行单位写成“动作 + 对象 + 结果”：

- [ ] 例：打开 Node.js 下载页，并确认能看到 LTS 下载选项。
- [ ] 例：在终端执行 `node --version`，并确认版本号大于等于 `22`。
- [ ] 例：在浏览器打开 `http://localhost:4321/`，并确认看到博客首页。

### 2.4 使用 AI 的固定流程

- [ ] 先把当前问题、环境、已尝试的命令和完整报错发给 AI。
- [ ] 要求 AI 先解释问题，再给出一个最小可执行步骤。
- [ ] 一次只执行 AI 给出的一个步骤。
- [ ] 执行后把实际输出反馈给 AI。
- [ ] 不直接复制不了解的命令，先问清楚命令的作用和风险。
- [ ] 要求 AI 给出“完成标准”和“失败时的下一步”。
- [ ] 把有效的命令、解释和解决结果记录到学习笔记。

推荐提问模板：

```text
我的目标是：
我的系统是：
我当前所在目录是：
我已经完成：
我执行的命令是：
完整报错是：
请先判断问题属于哪一层，再给我一个最小执行单位，并说明验证方法。
不要一次给我多个未解释的操作。
```

### 2.5 搜索与判断信息是否有效

- [ ] 先确认自己缺少的具体信息，而不是泛泛搜索“怎么学编程”。
- [ ] 优先查看官方文档、官方仓库和项目当前版本的说明。
- [ ] 检查文章的发布日期和适用版本。
- [ ] 对照本地 `package.json`、锁文件和配置文件确认版本。
- [ ] 至少用两个可靠来源交叉核对关键步骤。
- [ ] 在执行可能修改文件或删除数据的命令前，先让 AI 解释风险。
- [ ] 用一个最小例子验证搜索到的方法。
- [ ] 记录信息来源、适用条件和验证结果。
- [ ] 如果信息与本地项目冲突，以本地项目实际配置和官方文档为准，并记录差异。

### 2.6 重构与复盘

- [ ] 把一次成功解决的问题整理成可复用步骤。
- [ ] 把重复命令整理成自己的命令清单。
- [ ] 把反复出现的报错整理成“现象—原因—处理—验证”表格。
- [ ] 删除已经失效或不适用的旧步骤。
- [ ] 每完成一个阶段，写下“哪里卡住、为什么卡住、下次如何更快判断”。
- [ ] 把最终流程重构成多篇文章，而不是只保留零散聊天记录。

## 五、阶段 2：安装开发工具

### 5.1 安装 Node.js

- [ ] 打开 [Node.js 官方下载页](https://nodejs.org/en/download)。
- [ ] 选择 LTS 稳定版。
- [ ] 确认 Node.js 主版本大于等于 `22`。
- [ ] 运行安装程序。
- [ ] 保持默认安装选项完成安装。
- [ ] 关闭已经打开的终端窗口。
- [ ] 重新打开 PowerShell 或 VS Code 终端。
- [ ] 执行 `node --version`。
- [ ] 确认输出的主版本号大于等于 `22`。
- [ ] 执行 `npm --version`。
- [ ] 确认 npm 能输出版本号。
- [ ] 将 Node.js 版本和 npm 版本记录到学习记录。

### 5.2 安装 pnpm

项目使用 pnpm，并在当前项目的 `package.json` 中声明了包管理器版本。

- [ ] 在终端执行 `npm install --global pnpm@9.14.4`。
- [ ] 等待安装命令执行完成。
- [ ] 执行 `pnpm --version`。
- [ ] 确认输出版本为 `9.14.4` 或与项目声明的版本兼容。
- [ ] 记录 pnpm 安装失败时的完整报错。
- [ ] 如果终端找不到 pnpm，关闭并重新打开终端。
- [ ] 再次执行 `pnpm --version`。

### 5.3 安装 Git

- [ ] 打开 [Git 官方下载页](https://git-scm.com/downloads)。
- [ ] 下载 Windows 版本 Git。
- [ ] 使用默认选项完成安装。
- [ ] 关闭并重新打开终端。
- [ ] 执行 `git --version`。
- [ ] 确认终端能输出 Git 版本号。
- [ ] 执行 `git config --global user.name "你的名字"`。
- [ ] 执行 `git config --global user.email "你的邮箱"`。
- [ ] 执行 `git config --global --list`。
- [ ] 确认 `user.name` 和 `user.email` 已配置正确。

### 5.4 安装 GitHub Desktop

- [ ] 打开 [GitHub Desktop 官方下载页](https://desktop.github.com/download/)。
- [ ] 下载并安装 GitHub Desktop。
- [ ] 启动 GitHub Desktop。
- [ ] 登录自己的 GitHub 账号。
- [ ] 确认 GitHub Desktop 显示自己的账号信息。
- [ ] 打开 GitHub Desktop 的 Git 配置页面。
- [ ] 确认 GitHub Desktop 使用已安装的 Git。
- [ ] 记住 GitHub Desktop 中“Clone repository”和“Push origin”的位置。

### 5.5 安装 VS Code

- [ ] 打开 [VS Code 官方下载页](https://code.visualstudio.com/download)。
- [ ] 下载 Windows 版本 VS Code。

## 七、阶段 4：克隆仓库并本地搭建

### 7.1 使用 GitHub Desktop 克隆

- [ ] 打开 GitHub Desktop。
- [ ] 选择 **File → Clone repository**。
- [ ] 选择 **GitHub.com** 选项。
- [ ] 搜索自己的 `Firefly` 仓库。
- [ ] 确认仓库地址属于自己的账号，而不是 `CuteLeaf/Firefly`。
- [ ] 选择本地存放目录。
- [ ] 点击 **Clone**。
- [ ] 等待克隆完成。
- [ ] 在文件管理器中打开克隆后的目录。
- [ ] 确认目录中能看到 `package.json` 和 `pnpm-lock.yaml`。
- [ ] 确认目录中能看到 `.github`、`src` 和 `public`。

### 7.2 检查远程仓库

- [ ] 在项目目录打开 PowerShell。
- [ ] 执行 `git remote -v`。
- [ ] 确认 `origin` 指向自己的 Fork 仓库。
- [ ] 记录 `origin` 的完整地址。
- [ ] 可选：执行 `git remote add upstream https://github.com/CuteLeaf/Firefly.git`。
- [ ] 可选：执行 `git remote -v` 确认 `upstream` 指向官方仓库。

### 7.3 用 VS Code 打开项目

- [ ] 在项目目录执行 `code .`，或使用 VS Code 的“打开文件夹”。
- [ ] 确认 VS Code 打开的是 Firefly 项目根目录，而不是上一级目录。
- [ ] 在 VS Code 中打开集成终端。
- [ ] 执行 `Get-Location`。
- [ ] 确认当前目录是 Firefly 项目根目录。

### 7.4 安装依赖

- [ ] 在项目根目录执行 `pnpm install`。
- [ ] 等待依赖安装完成。
- [ ] 确认没有未处理的安装错误。
- [ ] 确认项目目录出现 `node_modules`。
- [ ] 不要手动删除或修改 `pnpm-lock.yaml`。
- [ ] 将安装耗时和异常信息记录下来。

### 7.5 启动本地开发服务器

- [ ] 在项目根目录执行 `pnpm dev`。
- [ ] 等待终端显示本地访问地址。
- [ ] 在浏览器打开 `http://localhost:4321/`。
- [ ] 确认浏览器能看到 Firefly 首页。
- [ ] 点击一个导航链接。
- [ ] 打开一篇已有文章。
- [ ] 返回终端按 `Ctrl+C` 停止开发服务器。
- [ ] 再次执行 `pnpm dev`，确认可以重复启动。
