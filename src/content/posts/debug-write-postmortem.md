> 报错修好了，不代表这次调试结束了。
> 真正有价值的一步，是把这次问题整理成一条可复用经验，让下次遇到类似情况时，你能更快定位、更稳修复。

---

上一篇我们讲了如何读懂 `git status`。

当项目改了很多文件时，Git 会告诉你哪些文件被修改、哪些文件是新加的、哪些文件还没有被跟踪。

这能帮你在提交前保持清醒。

但调试篇还差最后一块：

> **修好问题以后，怎么把经验留下来？**

很多新手调试完会立刻松一口气：

```text
终于好了，继续做下一个功能。
```

这当然可以。

但如果每次都这样，下一次遇到类似报错，你可能还是会慌。

更好的做法是：修好以后，用几分钟做一次 Debug 复盘。

---

## 为什么要写 Debug 复盘

Debug 复盘不是写长篇报告。

它的作用很简单：

> **把一次混乱的问题，整理成下次能快速使用的经验。**

比如你这次遇到：

- 新文章构建失败
- 页面白屏但构建通过
- 表格把正文撑宽
- `git status` 里有很多 `??`

如果你只是修好了，很快就会忘。

但如果你记录：

1. 当时的现象
2. 关键错误信息
3. 真正原因
4. 修改了哪个文件
5. 最后怎么验证

下次再遇到类似问题，你就不需要从零开始。

这就是调试经验的复利。

---

## 一次 Debug 复盘应该写什么

Debug 复盘不需要复杂。

可以固定成 6 个问题：

| 问题 | 作用 |
|------|------|
| 现象是什么 | 记录用户能看到的问题 |
| 错误信息是什么 | 保留终端或 Console 的关键线索 |
| 最近改了什么 | 把问题和改动联系起来 |
| 根因是什么 | 找到真正导致问题的原因 |
| 怎么修的 | 记录最小修改方案 |
| 怎么验证的 | 确认问题真的解决 |

这 6 个问题回答完，一次复盘就够用了。

它不需要很文学。

它只需要对未来的你有帮助。

---

## 复盘模板

你可以直接使用这个模板：

```md
## Debug 复盘：[问题标题]

### 现象
[页面白屏 / 构建失败 / 样式溢出 / Git 状态混乱]

### 错误信息
```text
[粘贴关键错误]
```

### 最近改动
- [改了哪个文件]
- [新增了什么内容]
- [调整了什么逻辑]

### 根因
[用一两句话说明真正原因]

### 修复
- [修改了哪个文件]
- [改了什么]

### 验证
- [运行了什么命令]
- [打开了哪个 URL]
- [检查了哪个页面]

### 下次避免
[以后遇到类似问题先检查什么]
```

这个模板不需要每次都写很满。

但至少要写清楚：

> 发生了什么，为什么发生，怎么确认已经修好。

## 示例一：新增文章后构建失败

假设你遇到的错误是：

```text
Article "new-note" uses unknown stage "调试".
```

可以这样复盘：

```md
## Debug 复盘：新增文章后 stage 名称不匹配

### 现象
新增 Markdown 文章后，运行 `npm run build` 失败。

### 错误信息
```text
Article "new-note" uses unknown stage "调试".
```

### 最近改动
- 新增 `src/content/articles/new-note.md`
- 修改 `src/data/articles.ts` 注册文章

### 根因
文章 frontmatter 里的 `stage` 写成了 `调试`，但 `taxonomy.ts` 里的合法阶段名称是 `调试篇`。

### 修复
把 Markdown frontmatter 改成：

```yaml
stage: 调试篇
```

### 验证
- 运行 `npm run build` 通过
- 打开 `/?article=new-note` 能显示文章

### 下次避免
新增文章时，先对照 `src/data/taxonomy.ts` 检查 `stage` 和 `tools` 名称。
```

这就是一条很有价值的经验。

下次再看到 `unknown stage`，你会立刻知道去哪里查。

## 示例二：页面白屏但构建通过

假设错误是：

```text
TypeError: Cannot read properties of undefined (reading 'title')
```

复盘可以这样写：

```md
## Debug 复盘：上一篇 / 下一篇读取 undefined

### 现象
`npm run build` 可以通过，但打开最后一篇文章时页面白屏。

### 错误信息
```text
TypeError: Cannot read properties of undefined (reading 'title')
    at ArticlePager
```

### 最近改动
- 给文章页增加了上一篇 / 下一篇按钮
- 修改了 `src/components/ArticleReader.tsx`

### 根因
最后一篇文章没有下一篇，但组件仍然读取了 `nextArticle.title`。

### 修复
在渲染下一篇按钮前判断 `nextArticle` 是否存在。
不存在时显示禁用状态。

### 验证
- 打开第一篇文章，上一篇禁用
- 打开最后一篇文章，下一篇禁用
- 打开中间文章，前后篇都能跳转
- Console 没有红色错误
- `npm run build` 通过

### 下次避免
数组取前后项时，都要考虑边界情况。
```

这个复盘的重点不是记录所有代码。

重点是记住：

> 前后篇导航要处理第一篇和最后一篇。

## 示例三：正文被表格撑宽

样式问题也可以复盘。

比如：

```md
## Debug 复盘：Markdown 表格撑破正文宽度

### 现象
某篇文章页面没有报错，但正文卡片变宽，页面出现横向滚动。

### 错误信息
无 Console 错误。

### 最近改动
- 新增了一篇包含长表格的 Markdown 文章

### 根因
表格单元格里有长路径，默认不换行，导致 `table` 撑破 `.article-body`。

### 修复
给表格单元格增加换行规则：

```css
.article-body th,
.article-body td {
  white-space: normal;
  overflow-wrap: anywhere;
}
```

同时确认父容器有：

```css
.article-card {
  min-width: 0;
  overflow: hidden;
}
```

### 验证
- 当前文章不再横向滚动
- 有代码块的文章仍然正常
- 首页布局没有变化
- `npm run build` 通过

### 下次避免
新增包含表格、长链接、长路径的文章后，要检查正文宽度。
```

这类复盘以后会特别有用。

因为样式问题通常没有红色错误，最容易忘记原因。

## 让 AI 帮你写复盘

你可以让 AI 根据这次调试过程帮你生成复盘。

提示词可以这样写：

```text
请根据这次调试过程，帮我整理一份 Debug 复盘。

要求包含：
1. 现象
2. 错误信息
3. 最近改动
4. 根因
5. 修复方式
6. 验证方式
7. 下次如何避免

请写得简洁，适合放进我的学习笔记。
```

如果你有终端错误、Console 错误或 Git diff，也可以一起给 AI。

它会帮你从杂乱信息里整理出结构。

但你要自己确认复盘是否准确。

不要让 AI 编一个听起来合理、但不符合真实过程的原因。

## 复盘放在哪里

早期项目里，复盘不需要复杂系统。

可以放在几个地方：

| 位置 | 适合什么 |
|------|----------|
| 当前文章 | 适合公开给别人看的调试经验 |
| 私人笔记 | 适合记录自己的具体踩坑 |
| PRD 或任务文档 | 适合记录项目决策 |
| commit message | 适合记录这次提交修了什么 |
| issue | 适合团队协作或后续追踪 |

对于本站来说，调试篇文章本身就是一种复盘。

我们把具体问题整理成文章：

- 新增文章后构建失败
- 页面白屏但构建通过
- 布局被表格和代码块撑坏
- Git 状态看不懂

这就是把项目里的真实问题变成学习内容。

---

## 复盘不是责备

Debug 复盘不是为了证明自己哪里做错了。

它是为了让你下次更快。

所以写复盘时，不要写成：

```text
我又把 stage 写错了，太粗心了。
```

更好的写法是：

```text
问题发生在 frontmatter 的 stage 名称和 taxonomy 不一致。
下次新增文章时，先对照 taxonomy 检查 stage 和 tools。
```

前一种是情绪。

后一种是经验。

调试里最有价值的是经验。

---

## 一个轻量 Debug 知识库

当复盘变多以后，可以整理成一个轻量知识库。

比如：

| 问题类型 | 先查什么 |
|----------|----------|
| `Cannot find module` | 文件路径、文件名、import |
| `invalid frontmatter` | title、stage、tools、updatedAt |
| `unknown stage` | taxonomy 里的阶段名称 |
| 页面白屏 | 浏览器 Console 第一条红色错误 |
| 正文撑宽 | table、pre、code、img、长链接 |
| Git 状态混乱 | `git status --short --branch` 和 `git diff` |

这个表不需要一开始很完整。

每修好一个问题，就加一条。

慢慢地，它会变成你自己的调试地图。

## 常见问题

### 1. 每个小错误都要复盘吗？

不需要。

简单拼写错误可以不写。

但下面几类值得写：

- 花了你比较久的问题
- 下次可能重复遇到的问题
- 涉及项目结构的问题
- 涉及构建、路由、数据、样式边界的问题

### 2. 复盘要写多长？

不用长。

能让未来的你快速看懂就行。

有时候 6 行就够。

### 3. 复盘可以直接变成文章吗？

可以。

本站调试篇就是这样。

把真实踩坑整理成文章，既帮助自己，也帮助后来的人。

## 小结

这一篇是调试篇的收尾方法。

修好报错以后，不要只停在"好了"。

你可以用一次简短复盘把经验留下来：

1. 现象是什么
2. 错误信息是什么
3. 最近改了什么
4. 根因是什么
5. 怎么修的
6. 怎么验证的
7. 下次如何避免

这样调试就不只是救火。

它会变成学习。

> **每一次 Debug，都是在给未来的自己写路标。**
