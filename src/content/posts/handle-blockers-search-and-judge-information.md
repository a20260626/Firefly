完成 [使用 GitHub Pages 部署 Firefly](/posts/deploy-static-site/) 后，遇到错误是正常的。真正需要训练的能力，不是记住所有命令，而是知道卡住时如何把大问题变成一个可以搜索、判断和验证的小问题。

本篇只解决一个主题：当本地运行、文章构建、GitHub Actions 或线上页面出现异常时，怎样找到有效信息，并把解决过程记录成下一次可以复用的方法。

## 问题是什么

“项目报错了”“网站没有更新”“命令不能运行”都太模糊，不能直接作为搜索问题。一个可处理的卡点，至少要包含以下信息：

| 要素 | 需要记录的内容 | Firefly 示例 |
| --- | --- | --- |
| 目标 | 原本想完成什么 | 让新增文章出现在本地文章列表 |
| 现象 | 实际看到了什么 | 页面能打开，但文章标题没有出现 |
| 时间点 | 哪一步开始异常 | 执行 `pnpm dev` 后刷新首页 |
| 环境 | 系统、项目和工具版本 | Windows、Node.js 22、pnpm、Firefly |
| 原文 | 完整错误或命令输出 | 终端中从错误类型到最后一行的完整内容 |
| 验收 | 什么结果算解决 | 文章页、首页列表和线上链接都能打开 |

把卡点写成一句完整的问题：

```text
在 Windows 的 Firefly 项目根目录中，Node.js 22 和 pnpm 已安装。
执行 pnpm check 时出现具体错误：……
我希望保留现有项目结构，并让检查命令通过。
应该先检查哪个文件或配置？
```

这句话比“帮我修一下项目”更容易获得有效回答，因为它同时说明了目标、现状、环境、约束、证据和期望输出。

卡点也要先分类。分类的作用是缩小搜索范围，而不是马上猜答案：

| 类型 | 常见现象 | 第一检查位置 |
| --- | --- | --- |
| 环境问题 | 命令不存在、版本不符合 | `node --version`、`pnpm --version`、`git --version` |
| 依赖问题 | 安装失败、模块找不到 | `package.json`、锁文件、安装输出 |
| 内容问题 | Markdown 文章不显示、构建失败 | frontmatter、文件路径、文章正文 |
| 配置问题 | 样式或功能没有变化 | `src/config`、组件引用、浏览器缓存 |
| Git 问题 | 推送失败、分支不一致 | `git status`、`git remote -v`、分支名称 |
| 部署问题 | Actions 失败、线上仍是旧内容 | workflow 日志、Pages 设置、线上地址 |

## 卡住不等于失败

编程中的“卡住”，通常只是当前掌握的信息不足以决定下一步。它不一定表示代码很差，也不意味着必须马上找别人给答案。更准确的说法是：当前现象与预期之间存在差异，而你还没有找到能够解释这个差异的证据。

面对卡点时，可以把思考顺序固定成四个问题：

1. 我原本预期看到什么？
2. 实际看到什么？
3. 两者第一次出现差异是在哪一步？
4. 哪一条最小实验能够增加新证据？

例如“文章没有上线”看起来是一个问题，实际可能位于完全不同的层级：

~~~text
Markdown 文件未保存
  ↓
文件已保存但 frontmatter 无效
  ↓
本地构建成功但没有 commit
  ↓
已经 commit 但没有 push
  ↓
Actions build 失败
  ↓
deploy 失败
  ↓
部署成功但访问了错误网址
  ↓
网址正确但浏览器显示缓存内容
~~~

如果还没有确定差异发生在哪一层，直接搜索“文章没有上线”会同时得到多类答案。先定位层级，搜索空间会迅速缩小。

## 先区分事实、推测和操作

排查记录中最容易混淆的是这三类信息：

| 类型 | 含义 | 示例 |
| --- | --- | --- |
| 事实 | 可以从命令、文件或页面直接观察 | `pnpm build` 返回退出码 1 |
| 推测 | 对事实的暂时解释 | 可能是文章 frontmatter 写错了 |
| 操作 | 为验证推测而执行的动作 | 运行 `pnpm check` 并查看第一条错误 |

“部署坏了”不是事实，因为它没有描述可观察现象。“GitHub Actions 的 build 任务在 Install, build, and upload 步骤失败”才是事实。

推荐在笔记中明确标记：

~~~text
事实：本地 pnpm build 成功；Actions build 失败。
推测：云端与本地使用的依赖或环境不同。
验证：比较 Node.js、pnpm、锁文件和 Actions 日志中的安装步骤。
结果：等待实验后填写。
~~~

这样即使推测错误，也不会丢失已经确认的事实。

## 适用环境是什么

本篇适用于以下流程：

- 在 Windows 本地使用 VS Code、PowerShell 或 GitHub Desktop 修改 Firefly；
- 使用 Node.js 22 或更高版本的 LTS 版本；
- 使用 pnpm 安装依赖、运行检查和构建命令；
- 使用 GitHub Actions 和 GitHub Pages 发布网站；
- 使用 ChatGPT 或其他 AI 工具辅助理解错误和制定排查步骤。

搜索前先确认自己所在的目录是 Firefly 项目根目录。可以用下面的命令确认：

```powershell
Get-Location
Get-ChildItem package.json, pnpm-lock.yaml
```

如果命令能够列出 `package.json` 和 `pnpm-lock.yaml`，说明当前目录基本正确。不要在错误目录中反复安装依赖或修改文件，否则会把无关问题混进排查过程。

向 AI 提问时，要一并提供以下上下文：

```text
项目：Astro + Svelte 的 Firefly 博客
系统：Windows
目录：Firefly 项目根目录
版本：Node.js 22、pnpm 9
刚执行的命令：pnpm check
完整输出：粘贴错误原文
已尝试：只做过一次检查，没有修改配置
目标：定位原因，并给出一个最小修改和验证命令
```

不要直接粘贴密码、访问令牌、私有仓库地址或 `.env` 文件内容。遇到权限错误时，只保留错误类型、命令和脱敏后的路径。

## 具体怎么做

### 1. 先停止盲目操作

看到错误后，先不要连续执行多条网上找到的命令，也不要一次修改多个文件。先保存当前状态：

```powershell
git status --short
git branch --show-current
node --version
pnpm --version
```

如果刚刚已经修改过文件，再查看差异：

```powershell
git diff --stat
git diff -- src/content/posts/你的文章.md
```

这一步的最小结果是：知道当前分支、工具版本、未提交文件和最近一次修改范围。

### 2. 复制完整错误，提取关键词

不要只复制最后一句“命令失败”。从第一次出现 `Error`、`ERR_`、`Cannot`、`failed` 或具体文件名的位置开始，向下复制到命令结束。

然后把错误拆成三个关键词：

```text
工具或框架 + 错误类型 + 具体对象
```

例如：

```text
Astro + frontmatter validation + published
pnpm + ERR_PNPM_NO_MATCHING_VERSION + package version
GitHub Actions + Pages deployment + permissions
```

关键词越接近错误原文，搜索结果越容易对应到同一个问题。不要先把错误翻译成自己的猜测，例如把“frontmatter validation failed”直接写成“文章内容有问题”。

### 3. 先查项目内信息

项目中的配置和脚本通常比通用教程更接近真实原因。按下面顺序搜索：

```powershell
rg -n "published|updated|frontmatter|ASTRO_BASE|deploy.yml" src .github package.json astro.config.*
```

再检查与错误直接相关的文件：

```powershell
Get-Content package.json
Get-Content .github/workflows/deploy.yml
Get-Content astro.config.mjs
```

如果问题是“文章没有显示”，先检查文件是否在 `src/content/posts/`、frontmatter 是否完整、`draft` 是否为 `false`，再检查文章 URL 是否使用了正确的 slug。不要先修改全局路由。

如果问题是“线上页面没有更新”，先检查本地构建是否成功，再检查提交是否已经推送，最后查看 Actions 日志。把本地、GitHub 和线上三个状态分开，才能知道问题发生在哪一层。

### 4. 按来源优先级搜索

建议使用下面的搜索顺序：

1. 项目自己的 README、`AGENTS.md`、配置文件和 workflow；
2. Astro、pnpm、GitHub Actions、GitHub Pages 等官方文档；
3. 官方仓库的 issue、release note 和讨论区；
4. 有完整复现步骤的技术文章；
5. 论坛、问答和搜索摘要，仅作为线索。

搜索词要描述事实和环境。例如：

```text
Astro content collection frontmatter validation published date
pnpm lockfile frozen lockfile GitHub Actions
GitHub Pages project site Astro base path
```

不要只搜索“Firefly 报错怎么办”。这种搜索缺少错误类型，通常会返回大量与当前版本无关的内容。

### 5. 判断搜索结果是否有效

找到一条信息后，不要因为它“看起来能解决”就立即执行。用下面五个问题检查：

| 判断项 | 检查问题 |
| --- | --- |
| 来源 | 是项目文档、官方文档，还是没有出处的转载？ |
| 版本 | 文章描述的 Astro、Node.js、pnpm 版本是否接近当前项目？ |
| 环境 | 它针对 Windows、macOS、Linux，还是与当前系统不同？ |
| 证据 | 是否解释了原因，并提供可复现的命令或配置？ |
| 风险 | 是否会删除文件、覆盖配置、泄露密钥或改变项目结构？ |

满足“来源可靠、版本相近、环境相同、步骤可验证、风险可控”的信息，才适合进入下一步实验。只有搜索摘要或一句“重新安装就好了”的回答，只能作为线索，不能直接当作结论。

### 6. 一次只验证一个假设

把候选答案改写成一个假设：

```text
假设：线上文章没有更新，是因为当前提交没有推送到 master 分支。
验证：执行 git status、git log -1 --oneline、git push origin master。
预期：远程分支出现当前提交，并触发 deploy.yml。
```

然后只做与这个假设有关的最小动作。验证完成后，再执行对应的检查：

```powershell
pnpm check
pnpm build
git status
```

如果是浏览器问题，再打开本地或线上 URL，确认标题、文章正文和资源路径。每次验证都记录“做了什么、看到什么、下一步是什么”，不要只记录最后的成功命令。

### 7. 把结论沉淀成可复用记录

解决后，用下面的模板写入笔记或文章：

```markdown
## 卡点名称

### 现象
- 在什么环境中执行了什么命令？
- 完整错误是什么？

### 原因
- 哪个配置、版本、路径或权限导致了问题？

### 解决
1. 执行了什么最小操作？
2. 修改了哪个文件？
3. 没有修改哪些无关内容？

### 验证
- 哪个命令通过了？
- 哪个页面或日志证明完成了？

### 可复用关键词
- `关键词 1`
- `关键词 2`
```

一条合格的记录，应该让未来的自己可以从“现象”直接找到“验证方法”，而不是只留下“最后修好了”。当同类卡点重复出现时，就可以把记录重构成新的文章或检查清单。

## 怎样稳定复现问题

搜索之前先确认问题能否重复出现。把复现过程缩成尽可能少的步骤：

~~~text
1. 在 Firefly 项目根目录打开 PowerShell；
2. 执行 pnpm check；
3. 每次都在同一个文件和行号出现同一条错误。
~~~

一个好的复现步骤应满足：

- 从明确的起点开始；
- 每一步只有一个动作；
- 不依赖“随便点一下”之类模糊描述；
- 别人或未来的自己能够重复；
- 能说明错误是必现、偶发，还是只在特定环境出现。

如果错误偶尔出现，额外记录触发条件：首次启动还是热更新、只在某篇文章出现、只在线上出现、刷新后消失，或网络变化后出现。偶发问题同样可以调查，但不能假装它是稳定复现。

如果项目刚才还能工作，先用只读命令记录最后一个正常状态：

~~~powershell
git log -5 --oneline
git reflog -5
~~~

这些命令能帮助找到最近提交和 HEAD 的移动记录。此时不要立刻执行 `git reset --hard`、删除锁文件或清空整个目录；先确认哪些内容属于自己的未提交工作。

还要避免同时升级 Node.js、更新所有依赖、重装编辑器、修改配置和删除缓存。一次改变五个变量，即使问题消失，也无法知道真正原因，甚至可能产生新的问题。

## 怎样读懂一段错误信息

一段错误通常包含多个层次：

~~~text
[工具名称] Astro
[错误类别] InvalidContentEntryDataError
[具体对象] src/content/posts/example.md
[位置] 第 4 行
[原因] published: Expected type date
[调用栈] 后续多行内部函数
[退出结果] Command failed with exit code 1
~~~

最有搜索价值的通常是“错误类别 + 具体原因 + 工具名称”。`Process completed with exit code 1` 只说明进程失败，不能解释失败原因；真正原因往往出现在它上方。

搜索前先做脱敏：

- 把 `C:\Users\你的名字\...` 改成 `<项目目录>\...`；
- 删除 token、cookie、Authorization 头和 `.env` 值；
- 私有仓库地址改成 `<私有仓库>`；
- 删除随机临时目录、时间戳和无关调用栈；
- 保留错误类型、相对文件路径、版本和关键配置名。

错误中的文件名和行号可以帮助本地定位，但不一定适合放进搜索词。别人很可能没有名为 `my-first-post.md` 的文件，却遇到了相同的 frontmatter 错误。

## 怎样让搜索词逐轮收敛

搜索可以分三轮，而不是一次粘贴整段日志：

| 轮次 | 搜索目标 | 示例 |
| --- | --- | --- |
| 第一轮 | 找到错误所属概念 | `Astro InvalidContentEntryDataError` |
| 第二轮 | 加入具体字段或对象 | `Astro InvalidContentEntryDataError published date` |
| 第三轮 | 加入版本、环境或部署场景 | `Astro 7 content collection published date GitHub Actions` |

如果结果太少，删除本地特有文件名和无关版本；如果结果太多，加入原始错误短语、框架主版本和具体对象。每轮只调整一两个关键词，才能判断哪部分让结果更准确。

常用搜索结构：

~~~text
"错误原文中的稳定短语" 工具名 版本
工具名 具体功能 预期行为 实际现象
site:docs.astro.build 错误关键词
site:github.com/withastro/astro 错误类别
~~~

引号适合搜索固定错误短语，但不要把包含用户名、路径和版本哈希的整行全部放进引号，否则可能匹配不到任何页面。

项目内搜索还可以寻找已有范例。比如文章 frontmatter 报错，可以打开一篇正常文章，对比字段名称、日期格式和缩进：

~~~powershell
Get-Content -TotalCount 20 src/content/posts/一篇正常文章.md
Get-Content -TotalCount 20 src/content/posts/出错文章.md
~~~

对比时只关注与错误相关的最小区域。正常文章中存在的图片、标签数量或正文结构，不一定都是出错文章必须复制的条件。

## 怎样判断证据强弱

不同来源能够支持的结论不同：

| 内容类型 | 能说明什么 | 不能直接说明什么 |
| --- | --- | --- |
| 官方参考文档 | 当前功能和参数如何使用 | 你的项目一定是这个原因 |
| 官方 issue | 某版本确实出现过相似问题 | 你的环境与报告者完全相同 |
| 博客教程 | 作者在特定环境跑通过流程 | 步骤适用于当前新版本 |
| 搜索摘要 | 可能存在相关页面 | 摘要准确代表完整原文 |
| AI 回答 | 帮助整理假设和检查步骤 | 命令一定正确且安全 |

判断时间也很重要。查看发布日期、最后更新时间、讨论中提到的版本，以及问题是否已在后续版本修复。旧文章可以用于理解稳定原理，但不宜不加验证地复制版本号、action 名称和配置字段。

如果建议会升级依赖、修改部署权限、改变路由结构或删除文件，至少再寻找一个独立证据：

- 当前项目配置是否支持这个结论；
- 官方文档是否描述了相同参数；
- 错误日志是否指向相同组件；
- 能否先通过只读命令验证；
- 是否存在更小、更可逆的实验。

三个网页复制同一段内容不一定是三个独立证据。更可靠的组合是：官方参数说明、本地配置和实际命令输出共同指向同一原因。

## 怎样设计最小验证实验

推荐把排查过程记录成实验表：

| 编号 | 假设 | 最小操作 | 预期 | 实际 | 结论 |
| --- | --- | --- | --- | --- | --- |
| 1 | 文件未被内容集合读取 | 运行 `pnpm check` | 显示内容诊断 | 无内容错误 | 暂不支持 |
| 2 | 文章仍是草稿 | 检查 `draft` | 值为 `true` | 值为 `false` | 排除 |
| 3 | commit 未推送 | 比较本地与远程哈希 | 两者不同 | 两者不同 | 支持，继续验证 |

“支持”不等于完全证明，只表示当前证据更符合这个解释。修复并完成最终验证后，才能把原因写成结论。

一个最小实验应该具备四个要素：

1. 只验证一个假设；
2. 操作影响范围明确；
3. 执行前写出预期结果；
4. 无论结果是否符合预期，都能获得新信息。

例如，为了确认“文章未显示是因为 `draft: true`”，只需读取该字段或把它改为 `false` 后重新运行相同检查。此时不应顺便升级依赖、重写路由和清理所有缓存。

## 什么时候需要最小复现

当项目很大、错误调用栈很长，或者无法判断是 Firefly 配置还是第三方工具问题时，可以建立最小复现：

1. 保留能够触发错误的最少文件和配置；
2. 移除与问题无关的主题、图片和文章；
3. 固定 Node.js、pnpm 和依赖版本；
4. 写出从安装到触发错误的完整命令；
5. 确认最小项目仍能稳定复现。

最小复现应在单独目录或临时分支中完成，不能为了“精简”直接删除当前工作目录的重要内容。涉及私有项目时，不要把源码、配置或数据发布到公共仓库。

对于简单的 Markdown 拼写错误，不需要创建新项目。只有当最小复现能显著隔离变量、帮助维护者或他人重现时，它才值得投入时间。

## 怎样向 AI 提出高质量问题

AI 适合帮助整理证据、生成候选假设、解释日志和设计验证步骤。它不应该在缺少上下文时替你猜测整个项目。

可以直接使用下面的模板：

~~~text
我正在处理一个 Astro 7 + Svelte 的 Firefly 博客项目。

目标：新增文章后让本地检查和生产构建通过。
环境：Windows，Node.js 22，pnpm 9，当前位于项目根目录。
执行命令：pnpm check
实际结果：粘贴从第一条具体错误开始的脱敏日志。
预期结果：检查退出码为 0，文章可以生成。
最近改动：只修改了 src/content/posts/example.md。
已确认：package.json 和 pnpm-lock.yaml 存在，其他文章正常。
约束：保留现有结构，不删除锁文件，不修改无关文件。

请先：
1. 区分日志中的事实和可能原因；
2. 给出最多三个候选假设，并按可能性排序；
3. 为第一个假设提供只读检查命令；
4. 说明预期输出以及不同输出分别代表什么；
5. 在证据不足时不要直接修改文件。
~~~

AI 给出回答后，可以继续追问：

- 这个判断对应日志中的哪一行？
- 命令会修改哪些文件？
- 如何在执行前查看当前差异？
- 如果预期输出没有出现，下一步应收集什么证据？
- 这个建议适用于当前主版本吗？

不要用“你确定吗”代替验证。真正有效的确认，是让建议对应到项目文件、官方说明或可重复的实验结果。

## 设置停止规则

下面任一情况出现时，应暂停当前方向并重新整理证据：

- 连续两次修改都没有让错误信息发生任何变化；
- 新建议要求删除大量文件，却无法解释原因；
- 当前搜索结果针对不同框架版本或操作系统；
- 已经忘记最初问题，开始同时修复多个无关警告；
- 无法说清执行下一条命令的预期结果；
- 操作可能覆盖未提交工作或泄露敏感信息。

暂停不等于放弃。此时可以恢复到已知状态、重新执行最小复现、把事实和推测分开，再决定继续搜索、阅读源码、向 AI 提问，还是向维护者提交 issue。

## 什么时候应该提交 issue

满足以下条件时，向项目或依赖维护者报告问题通常更有价值：

- 使用受支持版本，且已经阅读相关文档；
- 在最新兼容版本中仍能复现；
- 能提供最小复现仓库或明确步骤；
- 实际行为与文档描述明显不一致；
- 日志中没有敏感信息；
- 搜索过现有 issue，确认不是已知重复问题。

issue 应包含标题、环境、复现步骤、预期行为、实际行为、最小复现和完整脱敏日志。不要把“请帮我看看项目”当作 bug 报告，也不要同时在多个无关仓库重复发布。

## 如何验证完成

完成本篇后，不需要把所有错误都解决，而要确认自己已经跑通一次“记录—搜索—判断—验证—沉淀”的闭环：

- [ ] 能把模糊的“出错了”改写成包含环境和完整证据的问题；
- [ ] 能确认当前目录、分支和工具版本；
- [ ] 能从终端或 Actions 中复制完整错误；
- [ ] 能把错误拆成工具、类型和对象三个关键词；
- [ ] 能先使用 `rg` 检查项目内的配置和脚本；
- [ ] 能优先查看官方文档和项目实际版本；
- [ ] 能使用来源、版本、环境、证据和风险五项判断搜索结果；
- [ ] 能一次只验证一个假设；
- [ ] 能用 `pnpm check` 或 `pnpm build` 验证代码和内容；
- [ ] 能用浏览器或 Actions 页面验证最终结果；
- [ ] 能把原因、操作和验证结果写成可复用记录。

可以用一个具体场景自测：新建一篇 Markdown 文章后，故意检查它是否出现在本地页面。如果没有出现，按照本篇顺序完成一次排查，并把过程记录下来。只要下一次遇到相同问题时不必重新从头猜，就说明这次学习已经产生了可复用的系统。

## 失败时先检查什么

### 搜索结果很多，但没有一条能解决

- 检查搜索词是否包含错误原文，而不是自己的猜测；
- 加上框架、工具和版本，例如 `Astro`、`pnpm`、`Node.js 22`；
- 回到项目内搜索，确认问题是否只是路径或配置拼写错误；
- 优先打开原始文档，不要只依据搜索摘要下结论。

### AI 给出的命令看起来危险

- 暂停执行删除、覆盖、重置或修改权限的命令；
- 要求 AI 说明命令影响的文件、原因和回滚方法；
- 先用只读命令确认状态，例如 `git status`、`git diff` 和 `Get-Content`；
- 涉及密钥、账号和远程仓库权限时，先移除敏感信息再继续提问。

### 修改后仍然失败

- 检查是否真的保存了文件，以及修改的是当前项目目录中的文件；
- 重新运行同一条验证命令，确认错误是否已经变化；
- 查看 `git diff`，确认修改没有超出当前假设；
- 如果错误变化了，记录新错误，不要继续沿用旧结论；
- 如果连续两次尝试都没有新证据，就退回上一个可工作的状态，重新缩小问题范围。

### 本地成功，但线上失败

- 确认本地改动已经提交并推送到正确的分支；
- 查看 GitHub Actions 的具体失败步骤和日志；
- 检查 `ASTRO_SITE`、`ASTRO_BASE` 以及项目站点路径；
- 用线上完整 URL 验证，不要只打开域名根目录；
- 检查浏览器缓存和部署完成时间，区分“还没部署”和“部署后内容有误”。

### 同一个错误有很多不同答案

- 比较回答针对的工具版本、操作系统和项目类型；
- 检查多个回答是否只是转载同一份上游信息；
- 优先选择影响范围小、可回滚并有明确验证方法的方案；
- 不要把多个答案的命令混合成一套未经验证的新流程；
- 用当前项目的配置和日志淘汰不匹配的答案。

### 搜索不到完整错误

- 缩短引号内的内容，只保留稳定的错误类型和关键短语；
- 删除用户名、绝对路径、哈希、时间戳和项目特有文件名；
- 分别搜索工具名、功能名和错误类别；
- 检查错误是否来自项目自定义脚本，并在仓库内使用 `rg`；
- 如果确实是新问题，准备最小复现，再查官方 issue 或提交报告。

### AI 一直重复相同建议

- 明确列出已经执行的命令和实际输出；
- 要求它不要重复已排除的假设；
- 让它说明下一步实验能区分哪两个原因；
- 保留关键事实，删除已经证伪的猜测；
- 如果缺少必要日志，先停止对话并回到项目收集证据。

### 不知道该不该执行某条命令

先判断它属于只读检查、可逆修改、生成文件、网络操作，还是破坏性操作。对于不理解的命令，要求逐段解释参数，并先寻找只读替代方案。

以下操作默认需要格外谨慎：递归删除、`git reset --hard`、强制推送、修改远程权限、执行来源不明的脚本、把密钥写入命令行，以及全局升级大量依赖。不能说明影响范围时，就不要执行。

排查卡点的最终目标不是得到一条神奇命令，而是获得一条有证据的因果链：我看到了什么，为什么判断它是原因，做了哪个最小修改，怎样证明结果已经改变。掌握这条链，换成新的项目、工具或错误类型时，仍然能够继续学习和解决问题。
