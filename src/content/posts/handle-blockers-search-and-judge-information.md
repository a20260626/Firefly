完成 [使用 GitHub Pages 部署 Firefly](/posts/deploy-static-site/) 后，遇到错误是正常的。真正需要训练的能力，不是记住所有命令，而是知道卡住时如何把大问题变成一个可以搜索、判断和验证的小问题。

本篇只解决一个主题：当本地运行、文章构建、GitHub Actions 或线上页面出现异常时，怎样找到有效信息，并把解决过程记录成下一次可以复用的方法。

## 问题是什么

“项目报错了”“网站没有更新”“命令不能运行”都太模糊，不能直接作为搜索问题。一个可处理的卡点，至少要包含以下信息：

| 要素 | 需要记录的内容 | Firefly 示例 |
| --- | --- | --- |
| 目标 | 原本想完成什么 | 让新增文章出现在本地文章列表 |
| 现象 | 实际看到了什么 | 页面能打开，但文章标题没有出现 |
| 时间点 | 哪一步开始异常 | 执行 `pnpm dev` 后刷新首页 |
| 环境 | 系统、项目和工具版本 | Windows、Node.js 22、pnpm、Firefly |
| 原文 | 完整错误或命令输出 | 终端中从错误类型到最后一行的完整内容 |
| 验收 | 什么结果算解决 | 文章页、首页列表和线上链接都能打开 |

把卡点写成一句完整的问题：

```text
在 Windows 的 Firefly 项目根目录中，Node.js 22 和 pnpm 已安装。
执行 pnpm check 时出现具体错误：……
我希望保留现有项目结构，并让检查命令通过。
应该先检查哪个文件或配置？
```

这句话比“帮我修一下项目”更容易获得有效回答，因为它同时说明了目标、现状、环境、约束、证据和期望输出。

卡点也要先分类。分类的作用是缩小搜索范围，而不是马上猜答案：

| 类型 | 常见现象 | 第一检查位置 |
| --- | --- | --- |
| 环境问题 | 命令不存在、版本不符合 | `node --version`、`pnpm --version`、`git --version` |
| 依赖问题 | 安装失败、模块找不到 | `package.json`、锁文件、安装输出 |
| 内容问题 | Markdown 文章不显示、构建失败 | frontmatter、文件路径、文章正文 |
| 配置问题 | 样式或功能没有变化 | `src/config`、组件引用、浏览器缓存 |
| Git 问题 | 推送失败、分支不一致 | `git status`、`git remote -v`、分支名称 |
| 部署问题 | Actions 失败、线上仍是旧内容 | workflow 日志、Pages 设置、线上地址 |

## 适用环境是什么

本篇适用于以下流程：

- 在 Windows 本地使用 VS Code、PowerShell 或 GitHub Desktop 修改 Firefly；
- 使用 Node.js 22 或更高版本的 LTS 版本；
- 使用 pnpm 安装依赖、运行检查和构建命令；
- 使用 GitHub Actions 和 GitHub Pages 发布网站；
- 使用 ChatGPT 或其他 AI 工具辅助理解错误和制定排查步骤。

搜索前先确认自己所在的目录是 Firefly 项目根目录。可以用下面的命令确认：

```powershell
Get-Location
Get-ChildItem package.json, pnpm-lock.yaml
```

如果命令能够列出 `package.json` 和 `pnpm-lock.yaml`，说明当前目录基本正确。不要在错误目录中反复安装依赖或修改文件，否则会把无关问题混进排查过程。

向 AI 提问时，要一并提供以下上下文：

```text
项目：Astro + Svelte 的 Firefly 博客
系统：Windows
目录：Firefly 项目根目录
版本：Node.js 22、pnpm 9
刚执行的命令：pnpm check
完整输出：粘贴错误原文
已尝试：只做过一次检查，没有修改配置
目标：定位原因，并给出一个最小修改和验证命令
```

不要直接粘贴密码、访问令牌、私有仓库地址或 `.env` 文件内容。遇到权限错误时，只保留错误类型、命令和脱敏后的路径。

## 具体怎么做

### 1. 先停止盲目操作

看到错误后，先不要连续执行多条网上找到的命令，也不要一次修改多个文件。先保存当前状态：

```powershell
git status --short
git branch --show-current
node --version
pnpm --version
```

如果刚刚已经修改过文件，再查看差异：

```powershell
git diff --stat
git diff -- src/content/posts/你的文章.md
```

这一步的最小结果是：知道当前分支、工具版本、未提交文件和最近一次修改范围。

### 2. 复制完整错误，提取关键词

不要只复制最后一句“命令失败”。从第一次出现 `Error`、`ERR_`、`Cannot`、`failed` 或具体文件名的位置开始，向下复制到命令结束。

然后把错误拆成三个关键词：

```text
工具或框架 + 错误类型 + 具体对象
```

例如：

```text
Astro + frontmatter validation + published
pnpm + ERR_PNPM_NO_MATCHING_VERSION + package version
GitHub Actions + Pages deployment + permissions
```

关键词越接近错误原文，搜索结果越容易对应到同一个问题。不要先把错误翻译成自己的猜测，例如把“frontmatter validation failed”直接写成“文章内容有问题”。

### 3. 先查项目内信息

项目中的配置和脚本通常比通用教程更接近真实原因。按下面顺序搜索：

```powershell
rg -n "published|updated|frontmatter|ASTRO_BASE|deploy.yml" src .github package.json astro.config.*
```

再检查与错误直接相关的文件：

```powershell
Get-Content package.json
Get-Content .github/workflows/deploy.yml
Get-Content astro.config.mjs
```

如果问题是“文章没有显示”，先检查文件是否在 `src/content/posts/`、frontmatter 是否完整、`draft` 是否为 `false`，再检查文章 URL 是否使用了正确的 slug。不要先修改全局路由。

如果问题是“线上页面没有更新”，先检查本地构建是否成功，再检查提交是否已经推送，最后查看 Actions 日志。把本地、GitHub 和线上三个状态分开，才能知道问题发生在哪一层。

### 4. 按来源优先级搜索

建议使用下面的搜索顺序：

1. 项目自己的 README、`AGENTS.md`、配置文件和 workflow；
2. Astro、pnpm、GitHub Actions、GitHub Pages 等官方文档；
3. 官方仓库的 issue、release note 和讨论区；
4. 有完整复现步骤的技术文章；
5. 论坛、问答和搜索摘要，仅作为线索。

搜索词要描述事实和环境。例如：

```text
Astro content collection frontmatter validation published date
pnpm lockfile frozen lockfile GitHub Actions
GitHub Pages project site Astro base path
```

不要只搜索“Firefly 报错怎么办”。这种搜索缺少错误类型，通常会返回大量与当前版本无关的内容。

### 5. 判断搜索结果是否有效

找到一条信息后，不要因为它“看起来能解决”就立即执行。用下面五个问题检查：

| 判断项 | 检查问题 |
| --- | --- |
| 来源 | 是项目文档、官方文档，还是没有出处的转载？ |
| 版本 | 文章描述的 Astro、Node.js、pnpm 版本是否接近当前项目？ |
| 环境 | 它针对 Windows、macOS、Linux，还是与当前系统不同？ |
| 证据 | 是否解释了原因，并提供可复现的命令或配置？ |
| 风险 | 是否会删除文件、覆盖配置、泄露密钥或改变项目结构？ |

满足“来源可靠、版本相近、环境相同、步骤可验证、风险可控”的信息，才适合进入下一步实验。只有搜索摘要或一句“重新安装就好了”的回答，只能作为线索，不能直接当作结论。

### 6. 一次只验证一个假设

把候选答案改写成一个假设：

```text
假设：线上文章没有更新，是因为当前提交没有推送到 master 分支。
验证：执行 git status、git log -1 --oneline、git push origin master。
预期：远程分支出现当前提交，并触发 deploy.yml。
```

然后只做与这个假设有关的最小动作。验证完成后，再执行对应的检查：

```powershell
pnpm check
pnpm build
git status
```

如果是浏览器问题，再打开本地或线上 URL，确认标题、文章正文和资源路径。每次验证都记录“做了什么、看到什么、下一步是什么”，不要只记录最后的成功命令。

### 7. 把结论沉淀成可复用记录

解决后，用下面的模板写入笔记或文章：

```markdown
## 卡点名称

### 现象
- 在什么环境中执行了什么命令？
- 完整错误是什么？

### 原因
- 哪个配置、版本、路径或权限导致了问题？

### 解决
1. 执行了什么最小操作？
2. 修改了哪个文件？
3. 没有修改哪些无关内容？

### 验证
- 哪个命令通过了？
- 哪个页面或日志证明完成了？

### 可复用关键词
- `关键词 1`
- `关键词 2`
```

一条合格的记录，应该让未来的自己可以从“现象”直接找到“验证方法”，而不是只留下“最后修好了”。当同类卡点重复出现时，就可以把记录重构成新的文章或检查清单。

## 如何验证完成

完成本篇后，不需要把所有错误都解决，而要确认自己已经跑通一次“记录—搜索—判断—验证—沉淀”的闭环：

- [ ] 能把模糊的“出错了”改写成包含环境和完整证据的问题；
- [ ] 能确认当前目录、分支和工具版本；
- [ ] 能从终端或 Actions 中复制完整错误；
- [ ] 能把错误拆成工具、类型和对象三个关键词；
- [ ] 能先使用 `rg` 检查项目内的配置和脚本；
- [ ] 能优先查看官方文档和项目实际版本；
- [ ] 能使用来源、版本、环境、证据和风险五项判断搜索结果；
- [ ] 能一次只验证一个假设；
- [ ] 能用 `pnpm check` 或 `pnpm build` 验证代码和内容；
- [ ] 能用浏览器或 Actions 页面验证最终结果；
- [ ] 能把原因、操作和验证结果写成可复用记录。

可以用一个具体场景自测：新建一篇 Markdown 文章后，故意检查它是否出现在本地页面。如果没有出现，按照本篇顺序完成一次排查，并把过程记录下来。只要下一次遇到相同问题时不必重新从头猜，就说明这次学习已经产生了可复用的系统。

## 失败时先检查什么

### 搜索结果很多，但没有一条能解决

- 检查搜索词是否包含错误原文，而不是自己的猜测；
- 加上框架、工具和版本，例如 `Astro`、`pnpm`、`Node.js 22`；
- 回到项目内搜索，确认问题是否只是路径或配置拼写错误；
- 优先打开原始文档，不要只依据搜索摘要下结论。

### AI 给出的命令看起来危险

- 暂停执行删除、覆盖、重置或修改权限的命令；
- 要求 AI 说明命令影响的文件、原因和回滚方法；
- 先用只读命令确认状态，例如 `git status`、`git diff` 和 `Get-Content`；
- 涉及密钥、账号和远程仓库权限时，先移除敏感信息再继续提问。

### 修改后仍然失败

- 检查是否真的保存了文件，以及修改的是当前项目目录中的文件；
- 重新运行同一条验证命令，确认错误是否已经变化；
- 查看 `git diff`，确认修改没有超出当前假设；
- 如果错误变化了，记录新错误，不要继续沿用旧结论；
- 如果连续两次尝试都没有新证据，就退回上一个可工作的状态，重新缩小问题范围。

### 本地成功，但线上失败

- 确认本地改动已经提交并推送到正确的分支；
- 查看 GitHub Actions 的具体失败步骤和日志；
- 检查 `ASTRO_SITE`、`ASTRO_BASE` 以及项目站点路径；
- 用线上完整 URL 验证，不要只打开域名根目录；
- 检查浏览器缓存和部署完成时间，区分“还没部署”和“部署后内容有误”。

排查卡点的最终目标不是得到一条神奇命令，而是获得一条有证据的因果链：我看到了什么，为什么判断它是原因，做了哪个最小修改，怎样证明结果已经改变。掌握这条链，换成新的项目、工具或错误类型时，仍然能够继续学习和解决问题。
