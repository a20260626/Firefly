> 文章页已经能读了，但一个内容站不能只让用户读完一篇就停住。
> 这一篇要做的是站内导航：让 Learning Stages、Tool Index、最近文章、上一篇和下一篇一起工作，把文章串成一条可继续走的路线。

---

上一篇我们做了文章阅读页。

文章可以通过 URL 打开，Markdown 能渲染，文章目录也能从标题自动生成。

这意味着网站已经具备了"阅读一篇文章"的能力。

但内容站还需要更进一步：

> **读完这一篇之后，用户应该知道下一步去哪。**

如果用户读完文章后只能返回首页，体验就会断掉。

如果侧栏里有阶段、工具、最近文章和前后篇导航，用户就能自然地继续探索。

所以这一篇要做的，就是阶段四里的站内导航。

---

## 为什么导航很重要

很多人做网站时，会把导航理解成顶部菜单。

但对内容站来说，导航不只是顶部几个链接。

它包括所有帮助用户移动的入口：

- 首页文章标题
- 继续阅读
- Learning Stages
- Tool Index
- 最近文章
- 文章目录
- 上一篇 / 下一篇
- 返回首页

这些入口共同决定一件事：

> **用户能不能在不迷路的情况下继续阅读。**

一个好的内容站，不应该让用户每读完一篇就重新思考"我现在去哪"。

它应该在合适的位置，把下一步放在用户眼前。

---

## 本站需要哪些导航路径

本站有两类阅读方式。

第一类是按顺序学习。

比如用户从认知篇开始，一路读到发布篇。

第二类是按问题查找。

比如用户现在只想看 Git、部署、AI 编程助手或者前端基础。

所以导航系统也要支持两条路径：

| 路径 | 对应模块 |
|------|----------|
| 按学习顺序读 | Learning Stages、上一篇 / 下一篇 |
| 按工具问题查 | Tool Index、最近文章 |

这两条路径不是互相替代。

它们服务的是不同场景。

新手第一次来，可以按 Learning Stages 读。

遇到具体问题时，可以从 Tool Index 查。

想看看网站最近写了什么，可以看最近文章。

---

## Learning Stages：主线导航

Learning Stages 是本站最重要的导航模块。

因为它对应的是学习路线。

6 个阶段分别是：

1. 认知篇
2. 起步篇
3. 对话篇
4. 实战篇
5. 调试篇
6. 发布篇

每个阶段下面显示属于这个阶段的文章。

用户看到它时，会得到两个信息：

1. 当前网站的整体学习顺序是什么
2. 每个阶段已经有哪些文章可以读

这比普通分类更有指导性。

普通分类只告诉你"文章放在哪里"。

Learning Stages 还告诉你"应该先读什么，再读什么"。

所以它应该在首页和文章页都出现。

用户从首页进入文章后，右侧仍然能看到 Learning Stages，阅读过程就不会和整体路线断开。

---

## Tool Index：问题导航

Tool Index 负责另一种移动方式：

> 我不是按顺序读，我是带着问题来找内容。

比如：

- 我想看 AI 编程助手怎么用
- 我想看 VS Code / Cursor 属于什么
- 我想看 Git 怎么保存版本
- 我想看 HTML / CSS / JavaScript 的基础
- 我想看怎么部署上线

这时用户不一定关心文章属于哪个阶段。

他更关心文章能不能解决眼前的问题。

所以 Tool Index 按 5 类工具组织文章：

| 工具分类 | 作用 |
|----------|------|
| ai编程助手 | 找和 Codex、Claude、GPT 对话写代码相关的内容 |
| ai IDE/编辑器 | 找和写代码环境相关的内容 |
| 版本控制 | 找和 Git、GitHub、提交、回退相关的内容 |
| 前端基础 | 找和 HTML、CSS、JavaScript 相关的内容 |
| 部署发布 | 找和 GitHub Pages、Vercel、Netlify 相关的内容 |

一篇文章可以出现在多个工具分类里。

这不是重复，而是交叉索引。

因为真实问题本来就会跨分类。

例如"做文章阅读页"这件事，既涉及前端基础，也涉及 AI 编程助手和编辑器环境。

---

## 最近文章：更新导航

最近文章模块解决的是另一个问题：

> 网站最近更新了什么？

对持续写作的网站来说，这个模块很重要。

它让老用户不用重新扫完整个学习路线，就能快速看到新增内容。

它也让首页和文章页都有一点"现在进行时"。

最近文章可以按文章注册顺序，也可以按 `updatedAt` 排序。

第一版不必做得很复杂。

只要能稳定显示最近几篇文章，并且点击能跳转，就已经足够。

后面文章数量多了，再考虑：

- 只显示最近 5 篇
- 按更新时间排序
- 加上阶段标签
- 加上更完整的归档页

先把入口做稳，比一开始做复杂归档更重要。

---

## 上一篇 / 下一篇：顺序导航

Learning Stages 解决的是全局路线。

上一篇 / 下一篇解决的是局部连续。

用户读完一篇文章后，最自然的问题是：

> 下一篇是什么？

如果底部直接有"下一篇"，用户就不需要回首页找。

这对阶段四尤其有用。

因为阶段四是一组实战记录：

1. 写简单 PRD
2. 复盘本站 PRD
3. 拆成任务
4. 搭项目基础
5. 做信息架构
6. 做首页体验
7. 做文章阅读页
8. 做站内导航

这些内容天然适合按顺序读。

上一篇 / 下一篇就是把这条顺序变成可点击路径。

它不需要手动写在 Markdown 里。

只要文章注册表有顺序，就可以自动计算。

---

## 标题也应该能点击

首页文章卡片里，标题本身应该是链接。

这看起来很小，但很符合用户直觉。

在博客和技术网站里，用户看到文章标题时，往往会直接点标题。

如果标题不能点，只能点"继续阅读"，会有一点别扭。

所以本站统一了这些入口：

- 首页文章标题能点
- 继续阅读能点
- Learning Stages 里的文章标题能点
- Tool Index 里的文章标题能点
- 最近文章里的文章标题能点
- 上一篇 / 下一篇能点

这套规则越统一，用户越不用思考。

他看到文章标题，就知道可以进入文章。

---

## 所有跳转都应该走同一个函数

实现导航时，很容易到处写跳转逻辑。

比如一个地方写：

```ts
window.location.href = `?article=${slug}`;
```

另一个地方写：

```ts
window.history.pushState(null, '', `/?article=${slug}`);
```

再另一个地方单独设置状态。

这样做很容易出问题。

更稳的方式是把打开文章的行为集中起来。

例如：

```ts
function openArticle(slug: string) {
  setSelectedSlug(slug);
  window.history.pushState(null, '', `?article=${slug}`);
  window.scrollTo({ top: 0 });
}
```

然后所有文章链接都调用它。

这样有几个好处：

1. URL 会同步更新
2. 页面状态会同步更新
3. 点击新文章后会回到顶部
4. 后续要改跳转方式，只改一个地方

导航入口可以很多，但底层跳转逻辑应该尽量统一。

---

## 导航要保持上下文

用户进入文章页后，不能只看到孤立的正文。

他还需要上下文。

本站文章页保留了这些模块：

| 模块 | 提供的上下文 |
|------|--------------|
| 学习档案 | 这个站点整体有多少内容 |
| 文章目录 | 当前文章内部结构 |
| Learning Stages | 当前文章在学习路线中的位置 |
| Tool Index | 当前文章关联哪些工具问题 |
| 最近文章 | 还有哪些新内容可以继续看 |
| 上一篇 / 下一篇 | 这篇文章前后是什么 |

这些模块一起构成阅读上下文。

它们让用户不需要记住整个网站结构。

页面会一直提醒他：

> 你在这里，还有这些方向可以继续走。

---

## 桌面端和移动端的导航不一样

桌面端屏幕宽，可以使用三栏布局。

左侧放学习档案和文章目录。

中间放正文。

右侧放 Learning Stages、Tool Index 和最近文章。

但手机屏幕不能这样硬塞。

移动端更适合单列：

1. 先显示主要内容
2. 再显示导航模块
3. 模块之间保持足够间距
4. 不强行 sticky

导航设计不是固定一种布局。

它要根据屏幕大小调整。

桌面端强调并列浏览。

移动端强调顺序阅读。

## 如何让 AI 帮你做站内导航

你可以这样给 AI 任务：

```text
请完成 M5：站内导航。

目标：
让首页和文章页里的文章入口都能统一跳转，并让用户能通过阶段、工具、最近文章、上一篇和下一篇在站内移动。

要求：
1. 首页文章标题和继续阅读都能跳转到文章页
2. Learning Stages 下的文章链接能跳转
3. Tool Index 下的文章链接能跳转
4. 最近文章里的文章链接能跳转
5. 文章页顶部和底部都有上一篇 / 下一篇导航
6. 所有文章跳转使用统一的 openArticle 逻辑
7. 点击新文章后页面滚动到顶部
8. URL query 参数 article 要同步更新
9. 首页和文章页保留一致的导航模块
10. 完成后运行 npm run build 验证
```

这段任务的关键，是把"能点"说清楚。

不要只说：

```text
优化导航。
```

这太抽象。

要明确哪些地方能点、点击后去哪、URL 是否更新、页面是否回到顶部。

这才是可执行的任务。

## 验收站内导航

站内导航完成后，可以这样检查：

1. 首页文章标题是否能打开文章？
2. 首页继续阅读是否能打开同一篇文章？
3. Learning Stages 里的每篇文章是否能跳转？
4. Tool Index 里的每篇文章是否能跳转？
5. 最近文章是否能跳转？
6. 文章页上一篇 / 下一篇是否正确？
7. 第一篇文章的上一篇是否禁用？
8. 最后一篇文章的下一篇是否禁用？
9. 点击文章后 URL 是否变成 `?article=xxx`？
10. 点击文章后页面是否回到顶部？
11. 刷新文章页后是否仍然显示当前文章？
12. `npm run build` 是否通过？

这些检查会帮你发现很多小问题。

比如：

- 只有首页能跳，侧栏不能跳
- URL 更新了，但正文没变
- 正文变了，但页面没有回到顶部
- 最后一篇仍然显示可点击的下一篇

导航问题通常不是大错误，但很影响体验。

所以要一个入口一个入口地检查。

---

## 常见问题

### 1. 导航链接要用 button 还是 a？

如果是站内状态切换，第一版用 button 也可以。

但要注意样式要像链接，并且交互要统一。

如果后续改成正式路由，很多地方可以逐步替换成真正的链接。

第一版最重要的是行为稳定。

### 2. 上一篇 / 下一篇应该按什么排序？

第一版按文章注册表顺序。

因为这个顺序通常就是作者希望用户阅读的顺序。

后续如果需要更复杂，可以加 `order` 字段。

但现在没有必要过早复杂化。

### 3. 最近文章和下一篇会不会重复？

可能会。

但它们的语义不同。

下一篇代表顺序。

最近文章代表更新。

即使出现重复，也不影响用户理解。

---

## 小结

这一篇做的是站内导航。

它把首页、文章页、阶段索引、工具索引和前后篇连接成一个整体。

你需要记住这条线：

1. Learning Stages 负责学习主线
2. Tool Index 负责问题查找
3. 最近文章负责更新入口
4. 上一篇 / 下一篇负责连续阅读
5. 首页标题和继续阅读都应该能点
6. 所有文章跳转应该使用统一逻辑
7. URL、页面状态和滚动位置要同步

做到这里，网站就不再是一组孤立页面。

它变成了一个能让用户持续移动的内容系统。

> **好的导航不是让用户看到更多链接，而是让用户不需要停下来想下一步去哪。**
