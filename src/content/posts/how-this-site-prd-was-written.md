> 上一篇我们讲了"如何写一个简单 PRD"。
> 这一篇不讲抽象方法，直接拆开本站的 PRD，看看它是怎么从一个模糊想法变成可执行开发任务的。

---

如果说上一篇是在讲"PRD 应该怎么写"，那么这一篇就是现场复盘：

> **这个 AI 编程学习站的 PRD，到底是怎么写出来的？**

这件事很值得单独讲。

因为很多人以为 PRD 是一开始就完整写好的。

其实不是。

本站的 PRD 也是一步一步长出来的：先有一个模糊方向，然后不断追问目标用户、内容结构、页面模块、参考风格、开发顺序，最后才变成可以交给 AI 执行的开发说明。

这正是 AI 编程里非常真实的一幕：

> 你不是一开始就知道全部答案，而是在和 AI 对话的过程中，把需求越想越清楚。

---

## 最开始的想法：做一个 AI 编程学习站

最初的想法其实很简单：

> 做一个帮助零基础用户学习 AI 编程的网站。

这句话听起来方向明确，但还不能直接开发。

因为它没有回答几个关键问题：

- 零基础用户具体是谁？
- 这个网站是课程、博客、工具站，还是作品集？
- 首页要展示什么？
- 文章怎么组织？
- 学习路径从哪里开始，到哪里结束？
- 视觉上应该像什么？
- 第一版做到什么程度就算完成？

如果直接把这句话丢给 AI：

```text
帮我做一个帮助零基础用户学习 AI 编程的网站。
```

AI 很可能会生成一个通用落地页：

- 顶部导航
- 大标题
- 几个功能卡片
- 一个按钮
- 一点营销文案

它未必错，但它不是我们真正想要的东西。

我们想要的是一个可以持续写文章、持续扩展、持续迭代的学习笔记站。

所以 PRD 的第一步，就是把这个模糊想法拆细。

---

## 第一次定方向：它不是 landing page，而是内容站

很多网站的第一屏会做成很大的 hero：

- 一个醒目的标题
- 一句价值主张
- 一个开始按钮
- 一张装饰图

但这个项目不适合那样做。

因为本站的核心不是"宣传一个产品"，而是"让用户读文章、按阶段学习"。

所以 PRD 里第一条重要判断是：

> **这个网站不是 landing page，而是学习内容站。**

这会直接改变页面设计。

如果是 landing page，首页重点是转化。

如果是内容站，首页重点是：

1. 让用户知道有哪些文章
2. 让用户知道应该按什么顺序读
3. 让用户能从工具维度快速查找
4. 让用户点击标题或继续阅读进入文章页

于是首页的核心不再是一个大 hero，而是文章流、学习阶段和工具索引。

这个判断，是整个 PRD 的地基。

---

## 第二次定结构：6 个学习阶段

确定这是内容站之后，下一个问题是：

> 这些内容应该怎么组织？

一开始可能会想到很多分类：

- AI 工具
- HTML
- CSS
- JavaScript
- Git
- 部署
- 提示词
- 调试

这些都对，但如果直接堆在一起，用户会迷路。

所以 PRD 里把内容整理成一条学习曲线：

| 阶段 | 主题 | 作用 |
|------|------|------|
| 认知篇 | ai编程能做什么 | 建立心智模型，消除恐惧 |
| 起步篇 | 环境从0到1 | 装好编辑器、终端、Git、Node.js |
| 对话篇 | 用ai写第一行代码 | 学会怎么和 AI 提需求 |
| 实战篇 | 从网页开始 | 用 AI 辅助做出这个网站 |
| 调试篇 | 翻车与修复 | 学会排查报错和修复问题 |
| 发布篇 | 让世界看到 | 部署上线，持续迭代 |

这 6 个阶段不是随便排的。

它们对应的是一个新手真正会经历的路线：

1. 先不害怕
2. 再装工具
3. 再学会对话
4. 再做项目
5. 再处理出错
6. 最后发布出去

这就是 PRD 里的学习路径。

---

## 第三次定索引：5 类工具按用户视角分组

有了学习阶段，还不够。

因为用户不一定每次都按阶段读。

有时候他会想：

- 我现在只想查 Git
- 我想知道 Cursor 和 VS Code 属于哪类
- 我想找部署相关内容
- 我想回看 AI 编程助手怎么用

所以 PRD 里又增加了第二条组织方式：工具索引。

最后定成 5 类：

| 工具分类 | 示例 | 用户理解方式 |
|----------|------|--------------|
| ai编程助手 | Codex / Claude / GPT | 跟 AI 对话，让它生成代码 |
| ai IDE/编辑器 | Cursor / VS Code | 写代码和管理项目的地方 |
| 版本控制 | Git / GitHub | 保存历史、协作和回退 |
| 前端基础 | HTML / CSS / JavaScript | 网页本身要学的东西 |
| 部署发布 | GitHub Pages / Vercel / Netlify | 让别人看到你的作品 |

注意，这里不是按技术复杂度分组，而是按用户视角分组。

对新手来说，"构建工具、运行时、包管理器"这些说法太抽象。

但"写代码的环境"、"保存历史的工具"、"发布给别人看的地方"就更容易理解。

这就是 PRD 写作里的一个重要原则：

> **分类不是给作者看的，是给用户看的。**

---

## 第四次定页面：首页和文章页

分类确定后，PRD 开始进入页面层。

第一版只需要两个页面状态：

1. 首页
2. 文章页

首页负责展示全局结构。

文章页负责阅读正文。

最终首页模块是这样定的：

| 区域 | 模块 |
|------|------|
| 左侧 | 学习档案 |
| 中间 | 文章预览卡片、学习路线 |
| 右侧 | Learning Stages、Tool Index、最近文章 |

文章页模块是这样定的：

| 区域 | 模块 |
|------|------|
| 左侧 | 学习档案、文章目录 |
| 中间 | 文章正文 |
| 右侧 | Learning Stages、Tool Index、最近文章 |

这里有一个后面才变清楚的要求：

> **首页和文章页要保持同样的三栏宽度。**

这样用户从首页点击文章进入阅读页时，不会感觉整个站突然换了一个布局。

这就是 PRD 里很容易漏掉、但体验上很重要的细节。

---

## 第五次定内容系统：文章要自动归类

如果只做三五篇文章，手动写链接也可以。

但这个网站会持续增加内容。

所以 PRD 里必须提前考虑：

> 后续新增文章时，能不能少改代码？

最后的方案是：每篇文章用 Markdown 写正文，用 frontmatter 写元数据。

例如：

```yaml
---
title: 用 AI 写第一行代码
stage: 对话篇
tools:
  - ai编程助手
  - ai IDE/编辑器
  - 前端基础
updatedAt: 2026-07-03
---
```

这样一篇文章就同时拥有了三个信息：

1. 它叫什么标题
2. 它属于哪个学习阶段
3. 它关联哪些工具

网站读取这些信息后，就能自动把文章放到：

- 首页文章流
- Learning Stages
- Tool Index
- 最近文章

这一步让网站从"写死的页面"变成了"可维护的内容系统"。

哪怕它很简单，也已经具备内容站的骨架。

---

## 第六次定视觉：参考 Sukka's Blog

PRD 里还有一个很关键的部分：视觉参考。

我们没有只写：

> 做得好看一点。

而是明确参考：

> Sukka's Blog 的清爽技术博客风格。

并且把它拆成具体特征：

- 浅灰背景
- 白色卡片
- 细边框
- 轻阴影
- 蓝色链接
- 三栏布局
- 中间正文宽度克制
- 侧栏提供目录、分类、最近文章

这些描述比"好看"有用得多。

因为 AI 可以根据具体特征改代码。

比如当我们说：

> 左侧学习档案、中间正文、右侧 Learning Stages 要参考 Sukka's Blog 的宽度。

AI 就可以把布局调成：

- 左侧约 320px
- 中间约 820px
- 右侧约 320px
- 三栏间距统一

这就是视觉 PRD 的价值：把感觉变成可执行的参数。

---

## 第七次定交互：标题、继续阅读和侧边链接都能跳转

内容站最核心的交互就是阅读路径。

所以 PRD 里逐渐补上了这些交互要求：

1. 首页文章卡片要显示摘要
2. 点击"继续阅读"进入文章页
3. 点击首页文章标题也进入文章页
4. Learning Stages 下的文章标题能跳转
5. Tool Index 下的文章标题能跳转
6. 最近文章里的标题能跳转
7. 文章页左侧目录能跳到对应小节

这听起来很细，但它决定了网站是否顺手。

如果只有一个入口能进文章，用户会觉得不自然。

参考成熟博客时你会发现：文章标题本身通常就是链接。

所以 PRD 里后面补了一条：

> 首页正文中的文章标题也要可点击跳转。

这就是从参考网站中提取交互细节。

---

## 第八次定开发顺序：先 M1，再逐步迭代

PRD 不只是写页面长什么样。

它还要帮助你决定：

> 先做什么，后做什么？

本站最后拆成了类似这样的顺序：

| 里程碑 | 目标 |
|--------|------|
| M1 | 搭项目基础，跑通 Vite + React + TypeScript |
| M2 | 建立阶段、工具、文章数据结构 |
| M3 | 做首页文章流和学习路线 |
| M4 | 做文章阅读页和 Markdown 渲染 |
| M5 | 调整侧栏、目录、最近文章和跳转交互 |
| M6 | 根据参考站优化宽度和视觉细节 |

这个顺序很重要。

如果一开始就追求最终视觉，很容易卡住。

更稳的做法是：

1. 先让项目跑起来
2. 再让数据通起来
3. 再让页面能跳转
4. 再让文章能阅读
5. 最后调视觉和体验

这就是编程思维里的"顺序"。

---

## 这个 PRD 是怎么被 AI 使用的

写好 PRD 后，不是让 AI 一口气做完所有东西。

更好的方式是分段给 AI：

```text
根据这个 PRD，第一步应该做什么？
```

AI 会先建议搭项目基础。

然后你可以继续说：

```text
做 M1：项目基础。
```

完成后再提交，再问：

```text
下一步做什么？
```

这个过程看起来慢，但很稳。

它有三个好处：

1. 每一步的目标清楚
2. 出错时容易定位
3. 每完成一段都可以提交保存

这比一次性让 AI 生成一个完整项目可靠得多。

## 这个 PRD 后来发生了哪些变化

一个真实 PRD 一定会变。

本站也是这样。

后续迭代中，我们不断补充了新的要求：

- 6 个学习阶段重新命名
- 5 类工具索引按用户视角重组
- 首页不再直接显示完整文章，而是显示文章预览
- 点击继续阅读跳转到新页面
- 文章页保留学习档案
- 文章页保留 Learning Stages
- 首页和文章页使用同样的三栏宽度
- 首页文章标题也支持点击跳转

这些变化不是推翻 PRD，而是让 PRD 更准确。

所以你可以把 PRD 想成一个活文档。

它不是一次写完后就锁死，而是在项目推进中不断修正。

## 最终得到的 PRD 骨架

如果把本站 PRD 压缩成一个版本，大概是这样：

```text
# AI 编程学习笔记站 PRD

## 项目目标
做一个帮助零基础用户学习 AI 编程的内容站，从认知、起步、对话、实战、调试到发布，形成完整学习路径。

## 目标用户
没有系统编程经验，但想用 AI 做网页、工具或个人项目的人。

## 核心结构
- 首页：学习档案、文章预览、6 个阶段、5 类工具索引、最近文章
- 文章页：学习档案、文章目录、正文、Learning Stages、Tool Index、最近文章

## 学习阶段
认知篇、起步篇、对话篇、实战篇、调试篇、发布篇。

## 工具索引
ai编程助手、ai IDE/编辑器、版本控制、前端基础、部署发布。

## 内容系统
文章用 Markdown 编写，通过 frontmatter 标记 title、stage、tools、updatedAt，自动进入首页、阶段索引和工具索引。

## 视觉风格
参考 Sukka's Blog：浅灰背景、白色卡片、细边框、轻阴影、蓝色链接、三栏布局、正文宽度克制。

## 第一版完成标准
首页能展示文章和索引；点击标题或继续阅读能进入文章页；文章页能阅读正文、查看目录，并通过侧栏跳转到其他文章。
```

这就是本站 PRD 的核心。

它不复杂，但足够指导开发。

## 你可以学到什么

从本站 PRD 的写法里，你可以学到 5 件事：

1. **先判断网站类型**：是内容站、工具站、作品集，还是落地页？
2. **先定用户路径**：用户从哪里进来，要去哪里？
3. **先定内容结构**：文章、阶段、工具、标签怎么组织？
4. **视觉参考要具体**：不要只说好看，要说布局、颜色、间距、模块关系
5. **PRD 可以迭代**：每次发现新需求，都把它补进结构里

这套方法不只适用于本站。

你做个人博客、作品集、课程站、工具导航页，都可以这样写。

---

## 小结

本站的 PRD 不是一开始就完美的。

它是从一个简单想法开始：

> 做一个 AI 编程学习站。

然后一步步补清楚：

- 它是内容站，不是 landing page
- 它有 6 个学习阶段
- 它有 5 类工具索引
- 它有首页和文章页
- 它用 Markdown 管理内容
- 它参考 Sukka's Blog 的三栏技术博客风格
- 它需要标题、继续阅读、侧栏文章链接都能跳转

这就是写 PRD 的真实过程。

不是一口气写出完美答案，而是不断把模糊变清楚。

> **AI 不怕你需求复杂，它怕你需求模糊。**

下一篇，我们就可以拿着这份 PRD，继续拆任务：先搭项目基础，再一步步把页面做出来。
