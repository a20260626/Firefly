> 网站看起来已经成型了，但还不能只凭感觉说"完成"。
> 阶段四的收尾，要做一次项目闭环检查：从 PRD、任务、数据、页面、文章、导航、视觉到构建验证，确认每一环都真正连上。

---

上一篇我们做了视觉统一。

首页、文章页、侧栏、正文宽度、卡片和链接样式都逐渐进入同一套视觉语言。

到这里，这个网站已经很像一个可用的内容站了。

但实战项目不能停在一句：

> 看起来差不多了。

真正的收尾应该是：

> **回到 PRD，检查我们最初说要做的东西，是不是都已经落到了页面和代码里。**

这就是项目闭环检查。

它不是为了挑刺。

它是为了确认项目没有只完成一堆零散功能，而是从目标、结构、实现到体验都连成了一个整体。

---

## 什么叫项目闭环

项目闭环可以理解成一条线：

```text
想法 -> PRD -> 任务拆分 -> 代码实现 -> 页面体验 -> 验证 -> 提交
```

这条线里任何一段断了，项目都会有问题。

比如：

- PRD 写得很清楚，但代码没有实现
- 页面看起来有内容，但文章没有进入阶段索引
- 首页能跳转，但刷新文章页打不开
- 样式很好看，但表格和代码块会撑破正文
- 本地能跑，但构建失败
- 功能做完了，但没有提交保存

这些都不算真正闭环。

闭环的意思是：

> 你能从最初需求一路追到最终页面，并且用验证证明它确实工作。

这一步很适合放在阶段四末尾。

因为它会把前面所有实战动作重新串起来。

---

## 第一步：回看 PRD 的核心目标

检查项目时，第一件事不是打开代码。

而是回看 PRD。

本站 PRD 的核心目标大概是：

> 做一个帮助零基础用户学习 AI 编程的内容站，从认知、起步、对话、实战、调试到发布，形成完整学习路径。

这句话里有几个关键词：

- 零基础用户
- AI 编程
- 内容站
- 完整学习路径
- 6 个阶段
- 文章可持续扩展

所以检查时要问：

1. 首页是否像内容站，而不是营销落地页？
2. 用户是否能看到 6 个学习阶段？
3. 用户是否能看到 5 类工具索引？
4. 用户是否能进入文章阅读页？
5. 新增文章是否能进入对应阶段和工具？
6. 页面是否适合长期写文章？

如果这些问题都能回答"是"，说明项目方向没有跑偏。

---

## 第二步：检查学习阶段是否落地

PRD 里最重要的结构之一，是 6 个学习阶段。

它们不是只写在文档里的概念。

它们应该落到三个地方：

| 位置 | 应该看到什么 |
|------|--------------|
| 数据层 | `stages` 里有 6 个阶段 |
| 首页 | Learning Stages 显示 6 个阶段 |
| 文章页 | 右侧仍然能看到 Learning Stages |

这一步要检查的不只是数量。

还要检查顺序和语义。

6 个阶段应该形成一条学习曲线：

1. 认知篇：ai编程能做什么
2. 起步篇：环境从0到1
3. 对话篇：用ai写第一行代码
4. 实战篇：从网页开始
5. 调试篇：翻车与修复
6. 发布篇：让世界看到

如果页面只显示了 6 个卡片，但顺序乱了，就不够。

如果阶段显示了，但文章没有自动归到阶段下面，也不够。

真正落地的标准是：

> 阶段既存在于数据里，也能在页面上指导用户阅读。

---

## 第三步：检查工具索引是否落地

PRD 里的另一条结构，是 5 类工具索引。

它们分别是：

| 工具分类 | 作用 |
|----------|------|
| ai编程助手 | Codex / Claude / GPT |
| ai IDE/编辑器 | Cursor / VS Code |
| 版本控制 | Git / GitHub |
| 前端基础 | HTML / CSS / JavaScript |
| 部署发布 | GitHub Pages / Vercel / Netlify |

检查工具索引时，要看三件事：

1. 数据里是否有 5 类工具
2. 页面上是否显示 5 类工具
3. 每篇文章是否能按 `tools` 进入对应工具分类

工具索引不是普通装饰。

它解决的是按问题查找。

比如用户想找 Git 相关内容，就应该能在"版本控制"里看到对应文章。

用户想找 AI 编程助手相关内容，就应该能在"ai编程助手"里看到对应文章。

如果 Tool Index 只是显示名字，但没有文章链接，那它还没有真正完成。

---

## 第四步：检查文章系统

内容站最核心的是文章。

所以要检查文章系统是否可持续。

本站的文章应该具备这些能力：

| 能力 | 检查方式 |
|------|----------|
| Markdown 写正文 | 文章放在 `src/content/articles` |
| frontmatter 写元数据 | 每篇文章有 title、stage、tools、updatedAt |
| 注册表统一管理 | `src/data/articles.ts` 能导入文章 |
| 自动归类 | 文章能进入阶段和工具索引 |
| 独立打开 | `?article=slug` 能显示对应文章 |
| 支持长文 | 标题、表格、代码块、图片能稳定渲染 |

这里有一个很重要的检查：

> 新增一篇文章时，需要改多少地方？

比较理想的状态是：

1. 新增一个 Markdown 文件
2. 在文章注册表里导入并注册
3. 文章自动进入首页、阶段、工具、最近文章和前后篇

如果新增文章需要手动改很多页面，后续维护就会很累。

---

## 第五步：检查首页体验

首页是用户进入网站的第一站。

它需要回答三个问题：

1. 这里有什么？
2. 我该从哪里开始？
3. 我能点哪里继续阅读？

所以首页检查可以按模块来做：

| 模块 | 检查点 |
|------|--------|
| 学习档案 | 文章、阶段、工具数量是否正确 |
| 文章预览 | 标题、摘要、日期、阶段是否显示 |
| 继续阅读 | 是否能进入文章页 |
| Learning Stages | 是否显示阶段和文章链接 |
| Tool Index | 是否显示工具和文章链接 |
| 最近文章 | 是否能快速进入新文章 |

还要特别检查标题能不能点。

因为用户看到文章标题时，通常会直接点击标题。

如果只有"继续阅读"能点，体验会有一点断。

首页检查不是看它有没有很多内容。

而是看它能不能把用户带进内容。

---

## 第六步：检查文章页体验

文章页是内容站的核心。

它要让用户稳定阅读，也要让用户继续移动。

检查文章页时，可以看这些点：

1. 标题、日期、阶段是否清楚
2. Markdown 正文是否正常渲染
3. 表格是否不会撑破正文
4. 代码块是否不会撑破正文
5. 图片是否限制在正文宽度内
6. 左侧文章目录是否自动生成
7. 点击目录是否能跳到小节
8. 右侧 Learning Stages 是否保留
9. 右侧 Tool Index 是否保留
10. 最近文章是否能跳转
11. 上一篇 / 下一篇是否正确

文章页的目标不是把内容塞进去。

它要让长文阅读变得稳定。

如果一篇普通文章能读，但一篇有表格和代码块的文章就撑破布局，那还没有完成。

所以验收文章页时，一定要找复杂内容测试。

---

## 第七步：检查导航闭环

站内导航要保证用户不会走到死路。

至少要检查这些入口：

| 入口 | 应该发生什么 |
|------|--------------|
| 首页文章标题 | 打开对应文章 |
| 继续阅读 | 打开对应文章 |
| Learning Stages 文章链接 | 打开对应文章 |
| Tool Index 文章链接 | 打开对应文章 |
| 最近文章 | 打开对应文章 |
| 上一篇 / 下一篇 | 打开前后文章 |
| 顶部首页 | 回到首页 |

还要检查几个细节：

- URL 是否同步变成 `?article=xxx`
- 刷新文章页是否仍然显示当前文章
- 点击新文章后页面是否回到顶部
- 第一篇文章是否禁用上一篇
- 最后一篇文章是否禁用下一篇

导航最怕"看起来能点，但状态没同步"。

比如正文换了，但 URL 没变。

或者 URL 变了，但刷新后打不开。

这些都要在闭环检查里发现。

---

## 第八步：检查视觉闭环

视觉统一不是看一眼说"不错"。

它也可以被检查。

比如：

1. 首页和文章页是否使用同样的三栏宽度？
2. 左侧和右侧侧栏是否宽度一致？
3. 中间正文是否稳定在合适宽度？
4. 卡片背景、边框、圆角、阴影是否统一？
5. 链接颜色是否统一？
6. hover 状态是否统一？
7. 侧栏 sticky 是否正常？
8. 移动端是否变成单列？
9. 文字是否会挤出按钮或卡片？
10. 页面是否没有明显重叠和溢出？

视觉闭环的重点是：

> 不同页面看起来属于同一个系统。

如果首页像一个网站，文章页像另一个网站，说明视觉还没有闭环。

---

## 第九步：检查构建

最后一定要跑构建。

不要只看本地开发服务器。

开发服务器能跑，不代表正式构建能过。

检查命令是：

```bash
npm run build
```

如果构建通过，说明 TypeScript 和打包流程没有发现硬错误。

如果构建失败，不要忽略。

把错误复制给 AI，让它判断：

- 是类型错误
- 是导入路径错误
- 是 Markdown 元数据错误
- 是依赖问题
- 还是构建配置问题

本站文章越来越多以后，构建可能会出现 bundle 体积提醒。

这类提醒不是失败。

它是在告诉你：后续如果内容继续增多，可以考虑代码拆分或按需加载文章。

第一版先记录这个风险，不必马上复杂化。

---

## 第十步：检查 Git 状态

做完项目闭环检查后，还要看 Git。

常用命令是：

```bash
git status --short --branch
```

它能告诉你当前有哪些文件被修改、新增或未跟踪。

检查 Git 状态有两个目的：

1. 确认这次改了哪些文件
2. 准备一次清楚的提交

对于这个项目，新增文章通常会涉及：

- 新的 Markdown 文章文件
- `src/data/articles.ts` 注册表更新

如果还有样式、组件或数据结构变化，也要一起看清楚。

不要在不知道改了什么的情况下提交。

好的提交应该像项目过程中的一个存档点。

---

## 给 AI 的闭环检查任务

你可以这样让 AI 帮你检查项目：

```text
请对当前项目做一次阶段四闭环检查。

目标：
确认这个网站是否已经从 PRD、任务拆分、数据结构、首页、文章页、站内导航、视觉统一到构建验证形成闭环。

请检查：
1. 6 个学习阶段是否在数据和页面里都落地
2. 5 类工具索引是否在数据和页面里都落地
3. 首页文章预览、标题跳转、继续阅读是否正常
4. 文章页 Markdown、目录、侧栏、上一篇 / 下一篇是否正常
5. Learning Stages、Tool Index、最近文章是否能跳转
6. 首页和文章页宽度、卡片、链接样式是否统一
7. 表格、代码块、图片和长链接是否不会撑破正文
8. 运行 npm run build 验证
9. 查看 git status，总结本次改动

不要新增功能，只做检查和必要的小修。
```

最后一句很重要：

> 不要新增功能，只做检查和必要的小修。

闭环检查不是继续往里塞功能。

它是把已经做过的东西确认清楚。

## 一个可复用的验收清单

以后你做别的项目，也可以用类似清单。

```text
项目闭环检查清单：

1. PRD 目标是否仍然清楚？
2. 页面是否对应 PRD 的核心模块？
3. 数据结构是否支撑页面展示？
4. 用户主要路径是否能走通？
5. 所有可点击入口是否行为一致？
6. 页面刷新后状态是否正确？
7. 复杂内容是否不会破坏布局？
8. 移动端是否能正常阅读？
9. 构建命令是否通过？
10. Git 是否准备好提交？
```

这套清单不只适用于内容站。

做工具站、作品集、小程序、管理后台，也都可以用。

本质都是同一件事：

> 从需求出发，沿着用户路径走一遍，再用构建和版本控制收尾。

## 常见问题

### 1. 项目闭环检查是不是测试？

它不是完整自动化测试。

但它是一种手动验收。

对早期项目来说，闭环检查很有价值。

等项目变复杂以后，再逐步补自动化测试。

### 2. 发现小问题要不要马上修？

如果是影响核心路径的小问题，应该马上修。

比如文章打不开、链接不能跳、构建失败。

如果是锦上添花的优化，可以先记录，放到后续迭代。

闭环检查的目标不是无限打磨，而是确认当前阶段可交付。

### 3. 什么时候可以提交？

当核心路径走通、构建通过、Git 状态清楚，就可以提交。

提交信息要说明这次完成了什么。

比如：

```text
feat: add stage four project closure article
```

或者更大的阶段提交：

```text
feat: expand stage four practice content
```

提交不是仪式。

它是给项目留下一个可以回到的稳定点。

## 小结

这一篇做的是阶段四的项目闭环检查。

它把前面所有实战动作重新串起来：

1. 回看 PRD 目标
2. 检查 6 个学习阶段
3. 检查 5 类工具索引
4. 检查文章系统
5. 检查首页体验
6. 检查文章页体验
7. 检查站内导航
8. 检查视觉统一
9. 运行构建
10. 查看 Git 状态并准备提交

这一步做完，阶段四就不再只是"我们做了很多东西"。

它会变成一条完整的实战记录：

> 从 PRD 出发，拆成任务，一步步做出首页、文章页、导航和视觉，最后用检查和提交收尾。

这就是 AI 编程里非常重要的一种能力：

> **不只是让 AI 写代码，而是带着 AI 把一个项目做成闭环。**
