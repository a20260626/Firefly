> 项目基础搭好以后，页面还只是一个能运行的空架子。
> 下一步不是急着做视觉效果，而是把 PRD 里的学习路径和工具分类，真正变成页面可以使用的信息架构。

---

上一篇我们完成了项目基础。

项目能启动，目录也搭好了。

现在要进入更关键的一步：

> **把 PRD 里的 6 个阶段和 5 类工具，从文档里的文字，变成网站里的结构。**

这一步听起来不像写页面那么直观。

但它决定了这个网站后面能不能持续增加文章、自动归类、自动生成索引。

如果说 M1 是把空文件夹变成能运行的项目，那么这一篇要做的，就是把这个项目变成一个有秩序的内容站。

---

## 什么是信息架构

信息架构这个词听起来有点大。

放到本站里，其实可以理解成一句话：

> **用户如何理解内容，页面就应该如何组织内容。**

本站的用户是刚开始接触 AI 编程的人。

他们最需要的不是一堆零散文章，而是一条能跟着走的路线。

所以 PRD 里先定了 6 个学习阶段：

| 阶段 | 主题 | 用户在这一阶段要解决的问题 |
|------|------|----------------------------|
| 认知篇 | ai编程能做什么 | 先知道 AI 编程到底能帮自己做什么 |
| 起步篇 | 环境从0到1 | 把编辑器、终端、Git 这些基础环境装起来 |
| 对话篇 | 用ai写第一行代码 | 学会把需求说清楚，让 AI 帮你写代码 |
| 实战篇 | 从网页开始 | 用 AI 辅助做出一个真实网页项目 |
| 调试篇 | 翻车与修复 | 遇到报错时知道怎么定位和修复 |
| 发布篇 | 让世界看到 | 把作品部署出去，并继续迭代 |

这是一条从害怕到动手、从动手到上线的学习曲线。

同时，PRD 里又定了 5 类工具：

| 工具分类 | 示例 | 用户怎么理解它 |
|----------|------|----------------|
| ai编程助手 | Codex / Claude / GPT | 跟 AI 对话，让它生成、解释、修改代码 |
| ai IDE/编辑器 | Cursor / VS Code | 写代码、看文件、管理项目的地方 |
| 版本控制 | Git / GitHub | 保存历史、提交版本、协作和回退 |
| 前端基础 | HTML / CSS / JavaScript | 网页本身的结构、样式和交互 |
| 部署发布 | GitHub Pages / Vercel / Netlify | 让别人通过链接看到你的作品 |

这两套结构不是重复的。

它们分别回答两个问题：

1. **我应该按什么顺序学习？**
2. **我想查某类工具时去哪里找？**

这就是本站的信息架构核心。

---

## 为什么不能直接写在页面里

最简单的做法，是直接在 `App.tsx` 里写死这些内容。

比如：

```tsx
<section>
  <h2>认知篇</h2>
  <p>ai编程能做什么</p>
</section>
```

这样当然能显示。

但问题也很明显：

- 如果阶段名称改了，要去页面代码里找
- 如果新增文章，要手动改很多链接
- 如果首页和文章页都要显示阶段索引，会出现重复代码
- 如果工具索引也要关联文章，手写会越来越乱

所以我们不应该把信息架构写死在页面里。

更好的方式是：

> **先把阶段和工具做成数据，再让页面从数据生成内容。**

这就是从"写页面"变成"做内容系统"的第一步。

---

## 第一步：把 6 个阶段变成数据

阶段数据可以放在 `src/data/taxonomy.ts` 里。

它不需要很复杂，但要足够表达页面需要的信息。

例如：

```ts
export const stages = [
  {
    id: 'cognition',
    order: 1,
    name: '认知篇',
    topic: 'ai编程能做什么',
    description: '建立心智模型，消除恐惧。',
  },
  {
    id: 'setup',
    order: 2,
    name: '起步篇',
    topic: '环境从0到1',
    description: '编辑器、终端、Git 一站搞定。',
  },
];
```

这里每个字段都有作用。

| 字段 | 作用 |
|------|------|
| `id` | 给代码使用的稳定标识 |
| `order` | 控制阶段显示顺序 |
| `name` | 页面上显示的阶段名称 |
| `topic` | 阶段主题 |
| `description` | 解释这一阶段解决什么问题 |

注意，`id` 和 `name` 不应该混用。

`name` 是给用户看的，可以改得更自然。

`id` 是给代码用的，最好稳定一些。

---

## 第二步：把 5 类工具变成数据

工具索引也一样。

它应该是另一组结构化数据。

例如：

```ts
export const tools = [
  {
    id: 'ai-assistant',
    name: 'ai编程助手',
    examples: 'Codex / Claude / GPT',
    description: '对话生成代码，解释错误，辅助重构。',
  },
  {
    id: 'editor',
    name: 'ai IDE/编辑器',
    examples: 'Cursor / VS Code',
    description: '写代码、看文件、运行项目的环境。',
  },
];
```

工具数据和阶段数据的作用不同。

阶段负责"学习顺序"。

工具负责"查找入口"。

比如一篇关于 Git 的文章，可能属于起步篇，也会进入版本控制工具索引。

这就是为什么文章后面要同时标记 `stage` 和 `tools`。

---

## 第三步：让文章带上元数据

只有阶段和工具还不够。

文章本身也要声明自己属于哪里。

所以每篇 Markdown 文章开头都有 frontmatter：

```yaml
---
title: 用 AI 写第一行代码
stage: 对话篇
tools:
  - ai编程助手
  - ai IDE/编辑器
  - 前端基础
updatedAt: 2026-07-03
---
```

这段不是正文。

它是文章的"身份证"。

网站读取它以后，就知道：

1. 文章标题是什么
2. 它属于哪个学习阶段
3. 它关联哪些工具分类
4. 它什么时候更新

这样一来，文章只要写好元数据，就能自动出现在多个位置：

- 首页文章流
- Learning Stages
- Tool Index
- 最近文章
- 上一篇 / 下一篇导航

这就是信息架构真正发挥作用的地方。

---

## 第四步：从数据生成 Learning Stages

PRD 里说首页要有 6 个阶段。

如果直接手写，会变成一组固定 HTML。

但现在有了数据，就可以从 `stages` 生成。

思路大概是：

```tsx
{articlesByStage.map((stage) => (
  <section key={stage.id}>
    <p>阶段 {stage.order}</p>
    <h3>{stage.name}</h3>
    <p>{stage.topic}</p>
    {stage.articles.map((article) => (
      <button key={article.slug}>{article.title}</button>
    ))}
  </section>
))}
```

这里最重要的是 `articlesByStage`。

它不是手写出来的，而是由文章数据计算出来的。

也就是说：

> 只要新增一篇 `stage: 实战篇` 的文章，它就会自动出现在实战篇下面。

这比每次手动加链接稳得多。

---

## 第五步：从数据生成 Tool Index

工具索引也是同样的逻辑。

每篇文章可以关联多个工具。

例如这篇文章同时关联：

- ai编程助手
- ai IDE/编辑器
- 前端基础
- 版本控制

网站可以根据 `tools` 字段，把文章分配到不同工具分类下。

页面生成逻辑大概是：

```tsx
{toolsWithArticles.map((tool) => (
  <section key={tool.id}>
    <h3>{tool.name}</h3>
    <p>{tool.examples}</p>
    {tool.articles.map((article) => (
      <button key={article.slug}>{article.title}</button>
    ))}
  </section>
))}
```

这样做以后，Tool Index 就不再只是装饰模块。

它变成了一个真实可用的文章入口。

用户想看 Git 相关内容，就可以去"版本控制"下面找。

用户想看 AI 对话方法，就可以去"ai编程助手"下面找。

---

## 第六步：让首页和文章页共享同一套结构

一个很容易踩的坑是：

首页写一套 Learning Stages。

文章页又重新写一套 Learning Stages。

这样做短期没问题，但后期很痛苦。

因为你改一处，另一处可能忘记改。

更好的做法是：

> **首页和文章页都使用同一套数据、同一套渲染逻辑。**

本站后来就把这些模块保持一致：

- 首页有学习档案
- 文章页也保留学习档案
- 首页有 Learning Stages
- 文章页也保留 Learning Stages
- 首页有 Tool Index
- 文章页也保留 Tool Index
- 首页和文章页的三栏宽度保持一致

这会让网站更像一个完整系统，而不是几个临时拼起来的页面。

用户从首页点进文章时，不会迷路。

他仍然能看到阶段、工具和最近文章。

---

## 第七步：给 AI 的任务要怎么写

如果你想让 AI 帮你完成这一步，不要只说：

```text
帮我做阶段和工具。
```

这太模糊。

更好的写法是：

```text
请完成 M2：信息架构。

目标：
把 PRD 里的 6 个学习阶段和 5 类工具索引落到代码数据层，并在首页渲染出来。

要求：
1. 在 src/data/taxonomy.ts 中维护 stages 和 tools 数据
2. stages 包含 id、order、name、topic、description
3. tools 包含 id、name、examples、description
4. 文章元数据需要能关联 stage 和 tools
5. 首页从数据生成 Learning Stages 和 Tool Index
6. 不做完整文章阅读页，不做复杂视觉
7. 完成后运行 npm run build 验证
```

这段提示词的重点不是"请你写代码"。

重点是它明确了：

- 数据放在哪里
- 数据长什么样
- 页面如何使用数据
- 这一步不做什么
- 怎么验证

AI 拿到这样的任务，更容易做出可维护的结果。

---

## 第八步：验收这一步

信息架构做完以后，不要只看页面有没有字。

你应该检查这些问题：

1. 首页是否显示 6 个阶段
2. 首页是否显示 5 类工具
3. 每个阶段的顺序是否正确
4. 每篇文章是否进入了正确阶段
5. 每篇文章是否进入了对应工具分类
6. 点击文章链接是否能打开对应文章
7. 新增文章时是否只需要补 Markdown 和注册表
8. `npm run build` 是否通过

这些检查比视觉更重要。

因为信息架构一旦错了，后面写再多文章都会乱。

## 常见问题

### 1. 阶段名称应该用中文还是英文？

给用户看的地方用中文。

给代码用的 `id` 用英文更稳。

例如：

```ts
id: 'practice'
name: '实战篇'
```

这样既方便用户理解，也方便代码维护。

### 2. 一篇文章能不能属于多个阶段？

第一版不建议。

学习阶段代表主线顺序，一篇文章最好只属于一个阶段。

但工具分类可以多个。

例如一篇文章可以属于"实战篇"，同时关联"ai编程助手"、"前端基础"和"版本控制"。

这样结构会更清楚。

### 3. 为什么要先做信息架构，再做视觉？

因为视觉应该服务结构。

如果内容关系还没定，样式会反复改。

先把阶段、工具、文章关系跑通，再统一卡片宽度、侧栏位置、链接样式，会更稳。

---

## 小结

这一篇做的是 M2：信息架构。

它的目标不是让页面更花哨，而是让网站真正有秩序。

你需要记住这条线：

1. PRD 里先定义 6 个学习阶段
2. PRD 里再定义 5 类工具索引
3. 代码里把它们变成结构化数据
4. 每篇文章用 frontmatter 声明所属阶段和工具
5. 页面从数据生成 Learning Stages 和 Tool Index
6. 首页和文章页共享同一套结构

做到这里，网站就不再是一堆孤立页面。

它开始变成一个可以持续扩展的学习系统。

> **信息架构的价值，是让新文章有地方去，让用户知道往哪里走。**
