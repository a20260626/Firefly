> 网站上线不是结束，而是第一次真正进入真实环境。
> 你要做的不是立刻庆祝收工，而是打开线上链接，像一个真正的读者一样检查它。

---

上一篇我们完成了静态网站部署。

项目从本地电脑走到了公网。
你终于拿到了一个别人也能打开的链接。

但发布成功只说明一件事：

> 文件已经被部署平台放到了线上。

它还不能说明：

- 首页体验是否正常
- 文章链接是否都能打开
- 移动端是否好读
- 图片资源是否加载
- 上一篇 / 下一篇是否能跳转
- 读者是否能理解这个网站在讲什么

所以发布篇还需要最后一层能力：

> **上线后检查，并根据反馈持续迭代。**

这篇文章就讲这件事。

---

## 先区分“部署成功”和“体验正常”

部署平台显示成功，不等于网站体验正常。

它只代表：

1. 平台拉到了代码
2. 依赖安装完成
3. 构建命令通过
4. 构建产物被发布
5. 平台生成了访问链接

但用户真正关心的是另一组问题：

1. 打开速度是否能接受
2. 页面是否清楚
3. 链接是否能点
4. 内容是否能读懂
5. 手机上是否舒服
6. 出错时有没有明显断裂

所以上线后的第一件事不是继续写新功能。

而是做一次完整的线上巡检。

---

## 用真实线上链接检查首页

先打开部署平台给你的线上地址。

不要只看本地：

```text
http://127.0.0.1:5177/
```

要看线上：

```text
https://your-site.example.app/
```

首页至少检查这些内容：

1. 学习档案模块是否显示
2. 最新文章是否按顺序出现
3. 文章标题是否能点击
4. `继续阅读` 是否能进入文章页
5. Learning Stages 是否完整显示 6 个阶段
6. Tool Index 是否完整显示 5 类工具
7. 页面左右留白是否舒服
8. 滚动时没有明显卡顿或错位

检查时不要带着“我知道它应该怎么用”的心态。

要假设读者第一次打开这个网站。

他们能不能一眼看懂：

> 这是一个从零开始学习 AI 编程的网站。

如果不能，首页就还需要继续调整。

---

## 检查每一类文章入口

本站的内容被分成 6 个学习阶段。

上线后不能只检查一篇文章。
至少要每个阶段打开一篇：

| 阶段 | 检查重点 |
|------|----------|
| 认知篇 | 初学者是否能理解 AI 编程能做什么 |
| 起步篇 | 环境搭建内容是否清楚 |
| 对话篇 | 提示词和工具使用是否能落地 |
| 实战篇 | 做网站的过程是否连贯 |
| 调试篇 | 报错处理是否可复用 |
| 发布篇 | 上线流程是否讲得明白 |

这一步的目的不是逐字校对。

而是确认信息架构没有断。

也就是：

> 用户能不能从首页进入阶段，再进入文章，再继续阅读下一篇。

---

## 特别检查上一篇和下一篇

本站文章页有上一篇 / 下一篇按钮。

上线后要重点检查它们。

因为这类导航最容易出现边界问题：

- 第一篇不应该有可点击的上一篇
- 最后一篇不应该有可点击的下一篇
- 中间文章应该能正常前后跳转
- 跳转后页面应该回到顶部
- 标题、目录和正文应该同步更新

可以这样抽查：

1. 打开第一篇文章
2. 看上一篇是否禁用
3. 点击下一篇
4. 打开中间某篇文章
5. 连续点击几次下一篇
6. 再点击上一篇返回
7. 打开最后一篇文章
8. 看下一篇是否禁用

如果这条阅读线是顺的，网站会更像一本完整的小书。

---

## 检查文章目录是否仍然好用

文章页左侧有文章目录。

它的价值在于：

> 当文章很长时，读者可以快速知道自己读到哪里，也可以跳到关心的小节。

上线后要检查：

1. 目录是否显示主要标题
2. 点击目录是否能跳到对应段落
3. 滚动页面时目录是否保持在可见区域
4. 标题文字过长时是否换行自然
5. 手机端是否不会挤压正文

文章目录不是装饰。

它是长文阅读体验的一部分。

如果目录太宽、太窄、太靠边，读者会感觉页面不稳定。

---

## 检查移动端，而不是只看桌面端

很多学习网站是在桌面端写出来的。

但读者可能会用手机打开。

至少检查两个宽度：

```text
桌面宽度：约 1440px
手机宽度：约 390px
```

手机端重点看：

1. 标题是否换行自然
2. 按钮是否能点
3. 代码块是否横向溢出
4. 表格是否挤破屏幕
5. 图片是否超出容器
6. 侧边栏模块是否排到正文下方
7. 正文行宽是否适合阅读

如果手机端看起来乱，说明页面还没有真正准备好被公开分享。

因为你无法控制别人用什么设备打开链接。

---

## 检查资源路径和大小写

上线后常见问题之一是：

> 本地正常，线上图片不显示。

原因可能是路径大小写。

Windows 对文件名大小写不敏感。
但很多线上环境更严格。

比如本地可能都能读：

```text
/assets/M1-Build-Success.svg
/assets/m1-build-success.svg
```

线上却可能只有一个能正常加载。

所以资源检查要看：

1. 文件是否真的在 `public/assets`
2. Markdown 里的路径是否和文件名完全一致
3. 大小写是否一致
4. 是否误用了本地绝对路径
5. 是否引用了临时目录里的文件

线上网站不能依赖你电脑里的临时文件。

凡是要展示给读者的资源，都应该放进项目目录。

---

## 收集反馈时不要只问“好不好”

如果你把链接发给朋友，不要只问：

> 你觉得怎么样？

这个问题太宽了。

对方很可能只会说“还不错”。

更好的问法是：

1. 你打开首页后，能看懂这是做什么的吗？
2. 你会先点哪篇文章？
3. 哪个标题让你最想继续看？
4. 哪个地方让你觉得困惑？
5. 手机上有没有看不清或点不到的地方？
6. 如果你是零基础，会不会知道下一步做什么？

反馈越具体，越容易变成改进任务。

不要把反馈当成评价。

把它当成线索。

---

## 把反馈变成任务

收到反馈后，不要马上乱改。

先把反馈归类。

可以分成四类：

| 类型 | 示例 |
|------|------|
| 内容问题 | 某篇文章看不懂，术语太多 |
| 导航问题 | 不知道下一篇在哪里 |
| 视觉问题 | 字太密、留白不舒服 |
| 技术问题 | 链接打不开、图片不显示 |

然后把它们改写成任务：

```text
反馈：手机上文章目录挤占空间。
任务：调整移动端文章页布局，让目录移动到正文上方或隐藏为折叠入口。
```

```text
反馈：不知道应该从哪篇开始。
任务：首页增加更明确的学习阶段入口，并突出第一阶段第一篇。
```

任务越清楚，AI 越能帮你做。

---

## 每次迭代保持小步提交

上线后很容易进入一种状态：

> 看到问题就随手改，改着改着忘了自己动了哪里。

这会让项目很快变乱。

更稳的做法是小步迭代：

1. 选一个明确问题
2. 修改少量文件
3. 本地检查
4. 运行 `npm run build`
5. 提交 Git
6. 推送到 GitHub
7. 等待自动部署
8. 线上复查

每次提交只解决一个主题。

比如：

```bash
git commit -m "fix: improve mobile article layout"
```

或者：

```bash
git commit -m "docs: clarify deployment checklist"
```

这样以后回看历史时，你能知道每一步为什么发生。

---

## 用 AI 做线上巡检助手

AI 很适合帮你把检查流程结构化。

你可以这样问：

```text
我已经把一个 Vite + React 静态内容站部署上线。
请帮我设计一份上线后巡检清单。

网站结构包括：
- 首页文章列表
- 学习档案模块
- Learning Stages 6 个阶段
- Tool Index 5 类工具
- 文章页目录
- 上一篇 / 下一篇按钮

请按桌面端、移动端、内容、导航、资源路径、发布流程六类输出。
每一项都要能实际检查，不要写空泛建议。
```

如果你已经收到反馈，也可以这样问：

```text
下面是用户反馈，请帮我整理成可执行任务。
要求：
1. 按影响程度排序
2. 每个任务说明要改哪里
3. 每个任务给出验收标准
4. 不要一次性建议大重构

反馈如下：
……
```

这时 AI 的作用不是替你决定网站方向。

而是把零散信息变成清晰任务。

---

## 形成自己的迭代节奏

一个网站真正变好，不靠一次大改。

它靠的是稳定节奏。

你可以给自己设一个简单循环：

1. 每次新增一篇文章
2. 每次修一个明显体验问题
3. 每次发布前跑构建
4. 每次发布后打开线上链接检查
5. 每隔几篇文章回看一次首页结构

这个节奏会让项目一直保持可控。

不要等问题堆满再处理。

小步维护比大规模返工轻松得多。

---

## 什么时候算一个阶段完成

发布篇不是拿到链接就完成。

更合理的完成标准是：

1. 网站有稳定线上链接
2. 首页能说明网站主题
3. 主要文章能正常打开
4. 导航路径能串起学习曲线
5. 移动端没有明显阅读障碍
6. 新增文章后能稳定发布
7. 遇到线上问题知道从哪里查

达到这些标准，才说明这个项目从“能跑”进入了“可持续维护”。

这也是学习 AI 编程很重要的一步：

> 你不只是让 AI 帮你生成页面，而是在学会管理一个会变化的项目。

---

## 小结

这篇文章完成的是发布篇第四步：上线后检查和持续迭代。

你需要记住：

1. 部署成功不等于体验正常
2. 上线后要用真实线上链接检查
3. 首页、文章页、目录、导航、资源和移动端都要看
4. 反馈要变成具体任务，而不是停留在感觉
5. 每次迭代要小步修改、小步提交、小步发布
6. AI 可以帮你整理巡检清单和任务拆解
7. 网站真正成熟，靠的是持续维护节奏

到这里，一个从零开始的网站已经走过了完整路径：

> 想清楚要做什么 → 搭项目 → 写页面 → 加文章 → 修问题 → 发布上线 → 持续迭代

这条路径比任何单个工具都重要。

因为工具会变，但这条工作流会一直有用。
