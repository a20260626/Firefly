完成 [卡点、搜索和有效信息判断](/posts/handle-blockers-search-and-judge-information/) 后，这一组文章的操作流程基本闭环了：安装工具、Fork 和克隆仓库、在本地运行、认识文件结构、修改配置、新增文章、提交推送，最后部署到 GitHub Pages。

但如果只记住了几条命令，这次学习很快就会失效。工具会升级，项目会变化，下一次遇到的错误也不会完全相同。真正值得留下来的，是一套可以迁移到其他项目和技能上的解决问题方法。

## 问题是什么

本篇只解决一个问题：完成一次 AI 编程项目后，怎样判断自己到底学到了什么，而不是只确认“网站上线了”。

一次完整复盘，要把结果分成四层：

| 层次 | 复盘问题 | Firefly 示例 |
| --- | --- | --- |
| 结果 | 最后做成了什么 | Firefly 在本地运行并发布到 GitHub Pages |
| 方法 | 中间使用了什么流程 | 提出问题、分解任务、搜索信息、逐步验证 |
| 能力 | 哪些能力可以迁移 | 阅读项目结构、定位错误、判断资料可靠性 |
| 系统 | 下次怎样更快重复 | 清单、文章模板、验证命令和卡点记录 |

如果只能回答“我照着步骤做完了”，说明学习还停留在模仿层。更完整的回答应该是：

> 我能把一个模糊目标拆成最小任务，知道每一步的输入、动作和验收标准；遇到错误时，我能先保留证据，再搜索可靠信息，用最小实验验证假设，最后把解决过程整理成下次可复用的文档。

这就是本次学习的核心成果。

## 适用环境是什么

本篇适用于以下情况：

- 完成一次 Firefly 或其他 GitHub 项目的本地搭建和部署；
- 使用 ChatGPT、Codex 或其他 AI 工具参与编程；
- 学习做网站、写文章、配置工具或处理其他多步骤任务；
- 想把一次性的成功经历整理成自己的工作系统；
- 遇到“会做一次，但下次不会做”的问题。

复盘不要求等待项目完全成功。即使部署失败、文章没有显示或构建中断，也可以复盘，因为失败过程里同样包含环境信息、判断依据和下一步线索。

复盘前准备三类材料：

1. 最初的目标和执行清单；
2. 终端命令、错误信息、修改过的文件和 Git 提交记录；
3. 最终页面、Actions 日志和线上验证结果。

不要只依靠记忆复盘。记忆会保留“最后成功”的印象，却容易忘记中间哪些步骤走错了、哪些答案其实没有用。

## 具体怎么做

### 1. 先复述完整过程

不用马上评价好坏，先用一句话复述这次流程：

```text
我在 Windows 上安装 Node.js、pnpm、Git、GitHub Desktop、VS Code 和 ChatGPT，
Fork 并克隆 Firefly，安装依赖后在本地运行，认识项目配置，新增 Markdown 文章，
用 Git 提交并推送，最后通过 GitHub Actions 发布到 GitHub Pages。
```

然后把过程拆成最小执行单位：

| 阶段 | 最小执行单位 | 验收证据 |
| --- | --- | --- |
| 定义问题 | 写出目标、现状、约束和验收标准 | 一段清晰的问题描述 |
| 准备环境 | 分别检查每个工具版本 | 版本命令有输出 |
| 获取项目 | Fork 后克隆自己的仓库 | 远程地址指向自己的仓库 |
| 本地运行 | 安装依赖并启动开发服务器 | 浏览器打开本地首页 |
| 认识结构 | 找到文章、配置和部署文件 | 能说出文件职责 |
| 修改内容 | 新增或修改一个 Markdown 文件 | 本地文章页能打开 |
| 提交同步 | 检查、提交并推送改动 | GitHub 出现新提交 |
| 发布网站 | 运行并检查部署工作流 | 线上 URL 能访问 |
| 处理卡点 | 记录错误并验证假设 | 得到原因和复现方法 |

如果某一步无法写出验收证据，说明它还不是一个完整的执行单位，需要继续拆分。

### 2. 区分“记住的命令”和“理解的关系”

命令是表面知识，关系才是可迁移的知识。例如：

| 只记命令 | 真正理解的关系 |
| --- | --- |
| 记住 `pnpm dev` | 开发服务器会读取项目配置并提供本地预览 |
| 记住 `pnpm build` | 构建会检查内容、生成静态页面并暴露结构问题 |
| 记住 `git push` | 本地提交只有推送到远程后，GitHub Actions 才能使用它 |
| 记住修改 `ASTRO_BASE` | 项目站点部署在子路径时，资源和链接必须带上正确前缀 |
| 记住某个修复答案 | 先确定现象、证据和假设，再选择风险最小的修改 |

复盘时，针对每一个关键动作回答三个问题：

```text
这个动作读取了什么输入？
它改变了什么状态？
我用什么证据确认它生效？
```

例如，`git push origin master` 的输入是本地已有提交，改变的是远程分支状态，验证证据是 GitHub 提交记录和部署工作流，而不是终端没有报错。

### 3. 找出自己真正获得的五种能力

#### 能力一：把愿望改写成问题

“学 AI 编程”“做一个网站”是方向，不是动作。现在应该能够补充目标、现状、环境、约束、输出和验收标准，把它改写成别人可以理解、AI 可以执行的问题。

#### 能力二：把问题拆成层级

大目标可以拆成阶段，阶段可以拆成任务，任务还可以继续拆成最小执行单位。拆分的标准不是“列表很长”，而是每一项都能独立执行和验证。

#### 能力三：阅读陌生项目

不需要一开始读懂所有源代码，但要能通过 `package.json`、配置目录、路由、内容目录和 workflow 找到入口，建立一张项目地图。

#### 能力四：使用证据排查问题

遇到卡点时，先看完整错误、当前版本、文件差异和日志，再搜索信息。能够区分“现象”“猜测”“原因”和“验证结果”，就不容易被看似合理的答案带偏。

#### 能力五：把一次经历变成系统

把成功步骤、失败原因、验证方法和注意事项写成清单或文章。下次遇到同类任务时，可以复用流程，而不是重新从零开始。

### 4. 复盘一次具体卡点

选择本次过程中最难的一处，不要只写“后来解决了”。使用下面的因果链：

```text
现象：线上文章页打不开。
证据：首页能打开，文章链接缺少 /Firefly/ 前缀。
假设：项目站点部署在子路径，但构建配置没有使用正确的 base。
操作：检查并修正 ASTRO_BASE，再重新构建。
验证：dist 中资源路径正确，线上文章页返回 200。
结论：项目站点的 base 路径必须与 GitHub Pages 仓库名一致。
```

一个好的复盘结论应该能回答“为什么”，并给出下一次的检查动作：

```text
下次部署 GitHub Pages 项目站点前，先确认站点 URL、ASTRO_SITE、ASTRO_BASE、
构建输出路径和线上完整文章 URL 是否一致。
```

### 5. 判断哪些内容还没有学会

可以用“独立完成测试”代替自我感觉。把原流程中的一个环节重新做一遍，但不直接复制原来的答案：

- 重新写一条清晰的问题；
- 重新定位一个配置文件；
- 重新新增一篇最小 Markdown 文章；
- 重新运行检查和构建；
- 重新查看 Git 差异和提交记录；
- 重新解释一次部署失败日志。

如果只能在原文中找到答案，说明是查找能力；如果能根据原理重新完成，说明已经形成理解。

### 6. 把复盘结果写成下一次的工作模板

最终留下的不是长篇感想，而是一张能直接使用的模板：

```markdown
# 项目复盘

## 目标
- 我想完成什么？
- 什么结果算完成？

## 环境
- 操作系统：
- 项目目录：
- Node.js / 包管理器版本：

## 流程
- [ ] 定义问题
- [ ] 准备工具
- [ ] 获取项目
- [ ] 本地运行
- [ ] 修改内容
- [ ] 检查并构建
- [ ] 提交并推送
- [ ] 部署并验证

## 卡点
- 现象：
- 证据：
- 假设：
- 最小验证：
- 结论：

## 可迁移方法
- 我学会了什么？
- 下次可以提前检查什么？
- 哪一步还需要继续练习？
```

当这张模板能被用于另一个项目时，说明这次复盘已经从“Firefly 教程”升级成了“自己的问题解决系统”。

## 先整理一份成果证据清单

复盘不能只依赖“我感觉会了”。先把这次项目留下的客观证据放在一起：

| 证据 | 它能证明什么 | 它不能单独证明什么 |
| --- | --- | --- |
| 本地首页截图 | 开发服务器曾经成功显示页面 | 生产构建和线上部署一定成功 |
| `pnpm check` 输出 | 当前源码通过 Astro 诊断 | 页面视觉和交互完全正确 |
| `pnpm build` 输出 | 项目能够生成生产静态文件 | 这些文件已经发布到 Pages |
| Git commit | 一组修改被保存为版本 | commit 已经推送到远程 |
| GitHub Actions 成功记录 | 某个 commit 完成构建和部署 | 浏览器访问的就是最新内容 |
| 线上文章 URL | 公开页面可以访问 | 所有内部链接和资源都正确 |
| 卡点记录 | 能复述某次排查过程 | 已经能解决所有同类问题 |

建议记录最终版本的关键信息：

~~~text
仓库：a20260626/Firefly
分支：master
最终 commit：填写短哈希
本地检查：pnpm check 成功
生产构建：pnpm build 成功
部署工作流：Deploy to GitHub Pages 成功
线上首页：https://a20260626.github.io/Firefly/
代表文章：填写一篇线上文章地址
~~~

这份清单把“完成”变成可以核对的状态。以后回看时，也能快速知道当时使用了哪个仓库、分支和版本。

## 用能力阶梯判断自己处在哪一层

同一条命令可以处在不同的学习层次：

| 层级 | 表现 | 例子 |
| --- | --- | --- |
| 识别 | 看见时知道它大概做什么 | 知道 `pnpm build` 是构建命令 |
| 模仿 | 按教程可以重复执行 | 复制命令后成功生成 `dist` |
| 解释 | 能说明输入、状态变化和输出 | 知道源码如何变成静态页面 |
| 调整 | 环境变化后知道修改哪里 | 仓库名变化时检查 `base` 路径 |
| 诊断 | 失败时能通过证据定位层级 | 区分内容错误、构建错误和部署错误 |
| 迁移 | 能在另一个项目复用方法 | 在新 Astro 项目找到对应脚本和部署配置 |
| 教授 | 能为别人设计步骤和验证 | 写出不会误导初学者的操作指南 |

第一次完成项目后，很多能力处在“模仿”或“解释”层是正常的。不要为了显得学会而把每项都评为最高，也不要因为还不能独立诊断就否定已有成果。

可以给关键能力标记当前层级：

~~~text
问题定义：调整
环境安装：解释
阅读项目结构：解释
修改 Markdown：调整
Git 提交：解释
GitHub Pages 部署：模仿
构建错误排查：识别
信息判断：解释
~~~

下一轮学习只需要让一两项上升一级，不必同时补齐所有能力。

## 警惕五种常见的学习错觉

### 错觉一：页面打开了，所以我理解了项目

页面打开只能证明某条运行链路成功。真正理解至少还包括：知道启动脚本在哪里、内容从哪里读取、修改后如何验证，以及生产构建与开发服务器有什么不同。

### 错觉二：AI 修好了，所以我也会了

AI 完成修改是项目结果，不自动等于个人能力。至少要能回答：它改了哪个文件、为什么改、验证命令是什么、如果结果不符如何回退。

### 错觉三：命令没有报错，所以任务完成了

命令成功只证明这一层完成。例如 `git push` 成功不代表部署成功，deploy 成功也不代表你访问了正确的文章 URL。每层都需要自己的验收证据。

### 错觉四：看懂答案等于能独立做

阅读时产生的熟悉感很强，但它可能只是识别。关掉教程重新完成、解释错误原因或处理一个小变化，才能测试是否形成了可调用的知识。

### 错觉五：做得越快，学得越好

第一次操作的速度意义不大。更重要的是第二次是否减少了盲目尝试，能否更快找到入口、保留证据并选择最小修改。

## 做一次闭卷自测

闭卷不是完全禁止查资料，而是先不看本系列文章，观察自己能走到哪里。可以新建一份自测记录，依次完成：

1. 找到仓库的包管理器和启动命令；
2. 说明文章文件应该放在哪里；
3. 新建一篇最小文章并填写合法 frontmatter；
4. 启动本地服务器并找到文章 URL；
5. 运行检查和生产构建；
6. 查看本次文件差异；
7. 写出合适的 Conventional Commit 信息；
8. 解释 push 后为什么会触发部署；
9. 找到 Actions 中与当前 commit 对应的运行；
10. 在线上验证标题、正文、链接和资源。

每一步使用下面的评分：

| 分数 | 标准 |
| --- | --- |
| 0 | 不知道从哪里开始 |
| 1 | 看原教程后可以完成 |
| 2 | 只查看项目文件或官方文档可以完成 |
| 3 | 可以独立完成并解释原理 |
| 4 | 出现变化或错误时仍能调整并验证 |

总分不是为了比较别人，而是找到下一轮练习的重点。某一步得 0 或 1，就把它拆成一个 15 到 30 分钟的小练习；不要重学整个项目。

## 做一个故意变化的迁移挑战

完全重复同一套步骤容易依赖记忆。更好的练习是保留目标，但主动改变一个条件：

- 把文章文件名换成新的英文 slug；
- 新增分类或标签，而不是复制原文章；
- 修改一个站点配置并确认影响范围；
- 在新分支上完成修改，再解释为什么不会自动发布；
- Fork 另一个 Astro 项目，寻找它对应的启动和构建脚本；
- 使用不同仓库名，推导新的 GitHub Pages 项目路径。

一次只改变一个条件，并记录：

~~~text
不变目标：新增文章并成功构建。
变化条件：文章属于新的分类。
原有方法中仍然有效的部分：文件位置、frontmatter、检查命令。
必须重新确认的部分：分类字段格式和页面展示方式。
验证证据：本地分类页、文章页和生产构建。
~~~

能够指出“哪些方法仍有效、哪些细节必须重新查”，才是真正的迁移能力。迁移不是把 Firefly 的路径照搬到所有项目。

## 复盘 AI 在项目中的角色

把本次 AI 的参与拆成几类，可以判断自己在哪些地方过度依赖、哪些地方使用得当：

| AI 角色 | 合理用法 | 需要自己负责的部分 |
| --- | --- | --- |
| 解释者 | 把错误和配置翻译成易懂语言 | 对照真实文件和日志核实 |
| 规划者 | 把目标拆成步骤和验收条件 | 确认步骤符合当前项目 |
| 搜索助手 | 提炼关键词和候选来源 | 判断来源、版本和风险 |
| 编码助手 | 生成小范围修改 | 审查 diff、测试和保留用户改动 |
| 调试助手 | 提出候选假设和验证命令 | 提供完整证据，一次验证一个假设 |
| 文档助手 | 整理过程、模板和清单 | 删除不准确或无法验证的结论 |

AI 不拥有项目的最终责任。文件是否该改、命令是否安全、线上结果是否正确，都必须由项目中的证据确认。

复盘时可以回答：

- 哪一步 AI 明显节省了时间？
- 哪一步 AI 因缺少上下文给出了不适用答案？
- 我提供了哪些信息后，答案质量明显提升？
- 哪个修改我能解释，哪个修改仍然像黑箱？
- 下一次应该让 AI 先检查、先计划，还是直接实现？

## 建立自己的最小工作流

本系列最终可以压缩成一条日常工作流：

~~~text
定义目标和验收
  ↓
检查项目与当前状态
  ↓
拆出一个最小改动
  ↓
本地运行并观察
  ↓
执行 check 和 build
  ↓
审查 git diff
  ↓
commit 和 push
  ↓
检查 Actions 与线上页面
  ↓
记录卡点和可复用结论
~~~

每个阶段都有明确出口：

| 阶段 | 最小出口条件 |
| --- | --- |
| 定义 | 能写出目标、约束和验收标准 |
| 修改 | diff 只包含预期文件和内容 |
| 本地验证 | 目标页面可见，控制台无相关错误 |
| 静态检查 | 检查和构建达到仓库要求 |
| 版本控制 | commit 范围清晰，工作区状态可解释 |
| 部署 | 最新 commit 对应的 build 和 deploy 成功 |
| 线上验证 | 直接 URL、资源和最新内容都正确 |
| 沉淀 | 留下原因、操作、验证和关键词 |

这条流程不要求每次都写长文。小改动可以使用简短清单，高风险或首次操作则保留更完整记录。

## 制定一个 30 天巩固计划

不要在一天内反复阅读所有文章。把练习分散到真实的小任务里：

### 第 1 周：巩固本地工作流

- 独立启动 Firefly；
- 修改一段已有文章；
- 新增一篇最小草稿；
- 运行 `pnpm check` 和 `pnpm build`；
- 用自己的话解释开发与构建的区别。

### 第 2 周：巩固 Git 和发布

- 使用 `git status` 和 `git diff` 审查修改；
- 创建一个范围明确的 commit；
- 推送并找到对应 Actions 运行；
- 验证线上 commit 与内容；
- 记录一次缓存或路径问题的排查过程。

### 第 3 周：训练故障诊断

- 人为制造一个容易恢复的 frontmatter 错误；
- 只根据错误日志定位文件和字段；
- 写出事实、假设、实验、结果；
- 修复后运行相同验证；
- 整理搜索关键词和安全注意事项。

### 第 4 周：完成迁移练习

- 选择另一个小型 Astro 或静态站点项目；
- 找到它的脚本、内容、配置和部署入口；
- 比较它与 Firefly 的差异；
- 完成一个最小内容修改；
- 写一页迁移复盘，而不是复制 Firefly 教程。

每周只需要完成一个小闭环。比起每天阅读大量资料，稳定地“修改—验证—解释—记录”更容易形成长期能力。

## 选择下一阶段学习方向

完成 Firefly 入门后，不必立刻学习所有前端技术。根据自己的目标选择一条主线：

| 目标 | 下一步重点 | 推荐小项目 |
| --- | --- | --- |
| 持续写博客 | Markdown、内容结构、图片、SEO | 完善三篇高质量文章 |
| 修改界面 | HTML、CSS、Astro 组件 | 重做一个文章卡片或页脚 |
| 添加交互 | JavaScript、TypeScript、Svelte | 添加可验证的小型交互组件 |
| 自动化发布 | Git、Actions、构建与缓存 | 增加一个安全的检查工作流 |
| 深入排错 | 浏览器工具、日志、最小复现 | 分析并记录一个真实 bug |
| 学习开源协作 | 分支、issue、PR、代码审查 | 向文档提交一个小 PR |

选择标准不是“哪个最热门”，而是哪个方向能服务于下一个具体目标。把目标控制在一到两周可以交付的范围，再重复本系列的方法。

## 写一份给未来自己的毕业说明

最后用一页文字回答：

~~~markdown
# Firefly 阶段复盘

## 我完成了什么
- 项目地址：
- 线上地址：
- 最终版本：

## 我现在能独立做什么
- （填写）

## 我仍需查资料才能做什么
- （填写）

## 我尚未理解什么
- （填写）

## 最有价值的一次卡点
- 现象：
- 原因：
- 验证：
- 下次提前检查：

## 下一项具体任务
- 目标：
- 截止条件：
- 验收证据：
~~~

诚实地区分“能独立完成”“需要查资料”和“尚未理解”，比写一句“全部掌握”更有价值。几周后重新阅读这份说明，就能看到哪些能力真正增长了。

## 如何验证完成

完成本篇后，应该能够独立回答下面的问题：

- [ ] 我能用一句话说清楚这次项目的目标和完成标准；
- [ ] 我能列出从目标到线上结果的完整阶段；
- [ ] 我能把一个阶段拆成可以单独验证的最小执行单位；
- [ ] 我能解释 `pnpm dev`、`pnpm build`、`git commit` 和 `git push` 的状态变化；
- [ ] 我能说出 Firefly 中文章、配置和部署文件的大致职责；
- [ ] 我能区分命令、原理、证据和结论；
- [ ] 我能复述一个卡点的现象、假设、操作和验证结果；
- [ ] 我能在不复制原答案的情况下重新完成一个小任务；
- [ ] 我能把解决过程整理成清单或模板；
- [ ] 我能把这套方法迁移到另一个 GitHub 项目。

如果以上项目大部分都能做到，那么你真正学到的就不只是“如何发布 Firefly”，而是如何使用 AI 和工具完成一个陌生的、多步骤的真实任务。

## 失败时先检查什么

### 复盘变成了流水账

- 不要只记录“先做了什么，再做了什么”；
- 为每一步补上输入、动作、输出和验收证据；
- 把“我为什么这样做”写出来；
- 选择一个失败点，补充当时的错误信息和判断过程。

### 觉得自己什么都学会了，但离开文章就不会

- 进行一次不看原文的独立完成测试；
- 只允许自己查看项目文件和官方文档；
- 先写出自己的假设，再与原来的做法比较；
- 把不会的地方标记出来，重新拆成一个更小的练习。

### 只记住了 Firefly，换项目就不会

- 把项目名称替换成另一个 Astro、Vite 或静态网站项目；
- 重新寻找入口文件、配置文件、内容目录和部署 workflow；
- 只保留方法，不照搬具体路径和版本；
- 比较两个项目的相同点和不同点。

### 复盘写得很长，但下次仍然找不到

- 把长文最后压缩成一页检查清单；
- 为每个常见卡点保留搜索关键词和验证命令；
- 给文章加上清晰的标题、阶段和链接；
- 每次再次使用后，删除过时步骤，补充新的证据。

### 把“成功上线”误认为“学习完成”

- 重新解释每个关键动作改变了什么状态；
- 检查能否在没有原答案的情况下处理一个小变化；
- 说明如果换电脑、换分支或换仓库名，哪些步骤需要调整；
- 把不会解释的命令暂时列为下一轮学习任务。

这次 Firefly 练习的终点，不是收藏一组命令，也不是让 AI 替自己完成所有操作。真正的收获是：面对一个模糊目标时，知道如何定义它、拆开它、搜索它、验证它，并把过程重构成下一次仍然有效的系统。这样的能力，才是 AI 编程时代最值得持续训练的能力。
