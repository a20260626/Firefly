完成 [把 Firefly 在本地运行起来](/posts/run-firefly-locally/) 后，下一步不是立刻修改很多文件，而是先建立一张“项目地图”。知道文件负责什么，再做范围明确的配置修改，遇到问题时才能快速定位。

## 为什么先认识结构，再开始修改

在一个已经可以运行的项目里，同一个界面结果可能由多层文件共同决定。例如文章标题来自 Markdown，文章页面结构来自布局，颜色来自样式，侧栏开关来自配置，最终路由又由 Astro 页面文件生成。

如果只根据页面外观猜文件位置，常见结果是：

- 为了改文字，误改了负责布局的组件。
- 在多个相似文件中重复修改同一个值。
- 把应该放进配置的数据硬编码到组件中。
- 图片放错目录，本地可见但正式构建失败。
- 配置对象增加字段，却没有同步类型定义。
- 页面出现问题后，不知道哪一处修改造成影响。

项目地图的目的不是背下所有文件，而是学会沿着职责寻找正确入口：

```text
我要改变什么
→ 它属于内容、配置、路由、布局、组件、样式还是资源
→ 找到对应目录
→ 查看统一出口和类型约束
→ 只修改一个最小值
→ 在页面和构建中验证
```

## 修改前的安全规则

第一次浏览项目结构时，先遵守六条规则：

1. **先读后改。** 打开文件先看导入、类型和相邻配置，不立即删除看不懂的代码。
2. **一次一个目标。** 修改站点副标题时，不顺便更换头像、导航和壁纸。
3. **修改前看 Git 状态。** 先区分之前已有变更与本次准备修改的文件。
4. **不移动核心目录。** 不因为目录多就自行合并、重命名或重新分类。
5. **不绕过类型。** TypeScript 报错时修正数据结构，不使用随意的类型断言掩盖问题。
6. **每次都验收。** 页面变化、`git diff`、`pnpm check` 和构建结果共同构成证据。

开始前执行：

```powershell
git status --short --branch
git diff --stat
```

如果工作区已有修改，先记录它们属于什么任务。不要把用户已有变更当作“脏文件”随意撤销。

## 问题是什么

本篇只解决一个问题：**如何在不破坏项目的前提下，找到 Firefly 的核心文件并修改一个站点配置。**

完成标准是：

- 能说出文章、页面、组件、布局、配置和静态资源分别放在哪里；
- 能从 `src/config/index.ts` 找到配置的统一出口；
- 能修改站点标题或副标题，并在本地页面看到变化；
- 修改后 `pnpm check` 和 `pnpm build` 都能通过。

本篇暂时不新增功能，不重写布局，也不同时修改多个配置模块。一次只验证一个变化，便于判断结果是否正确。

## 适用环境是什么

本文适用于已经完成以下准备的 Windows 开发环境：

- 已经 Fork 并克隆 Firefly 仓库；
- 已经在项目根目录成功执行过 `pnpm install`；
- Node.js 版本不低于 22，pnpm 可正常使用；
- 使用 VS Code 打开了包含 `package.json` 和 `pnpm-lock.yaml` 的项目根目录；
- 能够使用 `pnpm dev` 启动本地开发服务器。

本文以当前 Firefly 仓库为准。仓库升级后，个别文件名或目录可能变化，遇到差异时先查看根目录的 `README.md`、`src/config/README.md` 和 `package.json`。

## 从页面结果反推文件位置

当你不知道从哪里开始时，可以先给目标分类：

| 想修改的结果 | 首先查看 | 通常还会关联 |
| --- | --- | --- |
| 文章标题、正文、标签 | `src/content/posts/` | 内容集合定义、文章布局 |
| 站点名称和副标题 | `src/config/siteConfig.ts` | `src/types/`、组件读取处 |
| 头像和个人信息 | `src/config/profileConfig.ts` | 图片资源、侧栏组件 |
| 导航菜单 | `src/config/navBarConfig.ts` | 路由或外部链接 |
| 页面 URL | `src/pages/` | 布局、页面组件 |
| 多个页面共用的界面 | `src/components/` | 配置、样式、工具函数 |
| 页面整体骨架 | `src/layouts/` | 组件、插槽、全局样式 |
| 字体、颜色、间距 | `src/styles/` | 组件局部样式、主题配置 |
| 图片、图标和下载文件 | `src/assets/` 或 `public/` | 使用方式和构建路径 |

分类只是确定第一个入口。真正修改前还要搜索配置在哪里被读取，确认它是否影响预期页面。

## 具体怎么做

### 1. 先确认自己打开的是项目根目录

在 VS Code 的资源管理器中确认当前目录包含这些文件：

```text
Firefly/
├── package.json       # 项目脚本、依赖和包管理器信息
├── pnpm-lock.yaml     # 锁定依赖版本
├── astro.config.mjs   # Astro 构建配置
├── public/            # 原样复制到网站根目录的静态文件
├── scripts/           # 图标、图片、字体等构建脚本
└── src/               # 网站源代码和文章内容
```

如果看不到 `package.json`，说明当前目录可能是仓库的上一级或某个子目录。先在终端执行：

```bash
pwd
```

Windows PowerShell 也可以执行：

```powershell
Get-Location
```

进一步验证：

```powershell
git rev-parse --show-toplevel
Get-ChildItem package.json, pnpm-lock.yaml, astro.config.mjs
```

Git 命令输出的根目录应与 VS Code 打开的工作区一致。如果 VS Code 标题、集成终端目录和 Git 根目录分别指向不同的 Firefly 副本，先统一工作目录，再开始修改。

### 2. 先认识 `src` 目录

Firefly 的主要开发内容集中在 `src`：

```text
src/
├── assets/       # 需要由 Astro 优化处理的图片等资源
├── components/   # 可复用的 Astro、Svelte 和其他界面组件
├── config/       # 站点功能和视觉效果配置
├── content/      # Markdown 文章和内容集合
├── layouts/      # 页面整体布局
├── pages/        # 页面路由，文件名对应网址路径
├── plugins/      # Markdown、代码块等内容处理插件
├── styles/       # 全局样式和主题样式
├── types/        # TypeScript 类型定义
└── utils/        # 通用工具函数
```

可以先记住四个最常用的位置：

| 目标 | 位置 |
| --- | --- |
| 新增文章 | `src/content/posts/` |
| 修改站点配置 | `src/config/` |
| 修改页面路由 | `src/pages/` |
| 修改公共界面 | `src/components/` 或 `src/layouts/` |

### 不需要一开始深入的目录

`src/plugins/`、`src/utils/` 和 `scripts/` 很重要，但通常不是修改站点名称或新增文章的第一入口。只有在目标明确涉及 Markdown 处理、共享逻辑或构建自动化时才深入。

看到自动生成文件时也要谨慎。某些常量、图标或低质量图片数据由构建脚本生成，直接手工修改可能在下一次构建时被覆盖。先搜索文件顶部注释、生成脚本和 `package.json` 命令，再决定修改源文件还是生成结果。

### 3. 认识文章和路由的关系

文章放在 `src/content/posts/`，例如：

```text
src/content/posts/
├── install-required-environment.md
├── fork-firefly-and-clone.md
├── run-firefly-locally.md
└── setup-project-foundation.md
```

Markdown 文件顶部的 frontmatter 决定标题、日期、分类和标签；文章正文负责页面内容。文章详情页的路由由文章文件名生成，例如：

```text
src/content/posts/run-firefly-locally.md
→ http://127.0.0.1:24321/posts/run-firefly-locally/
```

因此，文章内容通常不需要手动修改 `src/pages`。只有在要增加一种新的页面类型时，才需要进一步研究页面路由文件。

### 路由文件和内容文件的区别

`src/pages/` 决定“网站有哪些页面以及 URL 怎样匹配”，`src/content/posts/` 决定“文章页面里有哪些内容”。新增普通文章时，现有动态路由会读取文章集合，因此只需新增 Markdown。

例如：

```text
新增一篇文章 → 修改 src/content/posts/
新增一个独立工具页 → 研究 src/pages/
改变所有文章的整体结构 → 研究文章 layout
改变文章卡片展示 → 研究相关 component
```

不要为了新增文章复制一个新的页面路由，否则容易产生重复页面和维护负担。

### 4. 认识 `src/config` 配置目录

Firefly 把配置按功能拆成多个文件，并通过 `src/config/index.ts` 统一导出。常用配置如下：

| 文件 | 主要用途 |
| --- | --- |
| `siteConfig.ts` | 站点标题、副标题、网址、主题色、分页和文章页开关 |
| `profileConfig.ts` | 头像、昵称、个人签名和社交链接 |
| `navBarConfig.ts` | 顶部导航菜单和外部链接 |
| `sidebarConfig.ts` | 左右侧栏位置及侧栏组件 |
| `backgroundWallpaper.ts` | 壁纸、横幅、渐变和水波纹动画 |
| `effectsConfig.ts` | 樱花等动画特效 |
| `commentConfig.ts` | 评论系统 |
| `footerConfig.ts` | 页脚内容 |

不确定某个配置在哪里时，先打开 `src/config/README.md`，再搜索配置名称。不要为了修改标题去改布局组件，也不要把多个功能配置混到同一个文件里。

### 配置出口为什么重要

`src/config/index.ts` 是配置的统一出口。其他文件通常通过 `@/config` 导入配置，而不是分别记住每个配置文件的相对路径。

统一出口的价值包括：

- 调用方只依赖一个稳定入口。
- 配置模块可以在内部重新整理，而不必修改所有使用位置。
- 搜索导出名称可以快速找到定义与引用。
- 避免复杂的 `../../config/...` 相对路径。

修改现有字段通常不需要改 `index.ts`；新增一个配置模块或新增导出时，才需要确认统一出口是否同步更新。

### 配置与类型定义要保持一致

配置对象经常使用类似 `SiteConfig` 的类型：

```ts
export const siteConfig: SiteConfig = {
```

这个类型规定允许哪些字段、字段是什么数据类型，以及哪些字段必填。如果只修改现有字符串值，通常不必改类型；如果新增字段或改变数据结构，则要在 `src/types/` 找到对应定义，并检查所有使用位置。

类型报错不是障碍，而是在提醒“当前数据与项目约定不一致”。不要通过 `as any` 或删除类型标注来让错误暂时消失。

### 5. 修改一个最小配置：站点副标题

打开 `src/config/siteConfig.ts`，找到：

```ts
export const siteConfig: SiteConfig = {

	title: "Steve",

	subtitle: "Demo site",
```

上面的代码展示了配置所在位置。将 `subtitle` 修改为自己的站点副标题，例如：

```ts
	subtitle: "AI 编程学习记录",
```

保持项目原有的 Tab 缩进和双引号格式即可。

如果还要修改站点名称，只修改同一个对象中的 `title`：

```ts
	title: "我的 Firefly 学习站",
```

一次先改一个字段。保存文件后，观察本地开发服务器和浏览器是否自动刷新。

### 6. 搜索配置的定义和使用位置

修改前可以在 VS Code 中使用全局搜索，搜索 `siteConfig` 或具体字段名 `subtitle`。也可以在终端使用：

```powershell title="搜索配置引用"
rg -n "siteConfig|subtitle" src
```

搜索结果通常会包含配置定义、类型定义、统一导出位置，以及读取配置的组件或布局。先看定义，再看一个实际使用位置，就能判断修改会影响哪些页面。

搜索时优先使用明确的变量名或字段名，不要用“标题”“颜色”这类过于宽泛的词遍历整个项目。如果电脑没有 `rg`，使用 VS Code 全局搜索即可。搜索是为了定位，不要在结果列表中批量替换所有同名文本。

### 7. 修改视觉效果时，先看对应配置

例如要默认关闭水波纹动画，打开 `src/config/backgroundWallpaper.ts`，确认 `common.waves` 使用：

```ts
waves: {
	enable: {
		desktop: false,
		mobile: false,
	},
	switchable: false,
},
```

这里的 `desktop` 和 `mobile` 控制默认状态，`switchable` 控制用户是否能在控制面板中重新切换。只想默认关闭但保留用户切换能力时，可以保留 `switchable: true`；希望访问每个页面都不出现水波纹动画时，使用 `switchable: false`。

布尔配置的含义要结合字段层级理解。`desktop`、`mobile` 和 `switchable` 控制的是不同维度，不能因为都使用 `true/false` 就认为可以互换。修改前查看相邻注释、类型定义和 README，确认每个字段的职责。

### 8. 区分 `src/assets` 和 `public`

两者都能存放资源，但处理方式不同：

| 目录 | 处理方式 | 适合内容 |
| --- | --- | --- |
| `src/assets/` | 由 Astro 导入、处理和优化 | 组件中使用的源图片、需要构建优化的资源 |
| `public/` | 构建时原样复制 | favicon、固定路径文件、无需处理的静态资源 |

不要只因为“图片能打开”就随意选择目录。放入 `public/` 的文件通常通过站点根路径引用；放入 `src/assets/` 的文件通常通过模块导入。部署在子路径时，错误的绝对路径可能导致本地正常、线上丢失。

文章配图应优先参考项目中已有文章的路径写法。复制一种已验证的组织方式，比自行发明新的资源目录更稳妥。

### 9. 用开发服务器检查修改

如果开发服务器没有运行，在项目根目录执行：

```bash
pnpm dev
```

打开终端显示的 `Local` 地址。不要假设端口一定是 `24321`，因为端口可能已经被其他进程占用。刷新首页，确认浏览器标题或页面中使用站点副标题的位置已经更新。

配置修改要在真正使用该配置的页面验收。修改导航就检查桌面端和移动端菜单；修改文章开关就打开文章详情页；修改侧栏还要检查不同宽度。只看首页不一定覆盖所有影响范围。

如果页面保存后出现错误，先查看终端中的第一条报错。不要继续编辑更多配置，也不要依靠反复刷新等待它自行恢复。

### 10. 修改前后都查看 Git 差异

在终端执行：

```bash
git diff -- src/config/siteConfig.ts
git status --short
```

确认差异只有自己计划修改的配置字段。若出现大量无关文件变化，先暂停，不要直接提交。

一套完整的最小修改循环是：

```text
记录当前 Git 状态 → 找到职责正确的文件
→ 阅读配置、类型和使用位置 → 只修改一个字段
→ 保存并观察开发服务器 → 在对应页面验收
→ 查看 git diff → 执行 check 和 build
```

如果结果不正确，可以根据 `git diff` 手动恢复刚才那一个字段。不要使用会清除整个工作区的命令，因为其中可能包含其他尚未提交的文章修改。

## 如何验证完成

按顺序完成下面的检查：

- [ ] 当前目录能看到 `package.json` 和 `pnpm-lock.yaml`；
- [ ] 能指出文章目录 `src/content/posts/`；
- [ ] 能指出配置目录 `src/config/`；
- [ ] 能指出页面目录 `src/pages/`；
- [ ] 能在 `src/config/index.ts` 找到配置导出；
- [ ] 已修改 `siteConfig.ts` 中的一个字段；
- [ ] 已在本地浏览器打开终端显示的 `Local` 地址；
- [ ] 浏览器中能看到配置修改后的效果；
- [ ] `git diff` 中没有无关修改；
- [ ] `pnpm check` 执行成功；
- [ ] `pnpm build` 执行成功；
- [ ] 关闭开发服务器后，修改没有被误认为已经提交。

验证命令：

```bash
pnpm check
pnpm build
git diff --check
```

三条命令验证不同方面：

- `pnpm check` 检查 Astro、内容和类型诊断。
- `pnpm build` 验证完整正式构建流程。
- `git diff --check` 检查空白字符等补丁格式问题。

命令通过后仍要阅读实际页面。类型正确不代表文案合适，构建成功也不代表导航、图片和移动端布局符合预期。

### 保存项目地图笔记

```text
项目根目录：
配置统一出口：src/config/index.ts
站点配置：src/config/siteConfig.ts
配置说明：src/config/README.md
配置类型：src/types/ 中对应文件
文章目录：src/content/posts/
页面路由：src/pages/
公共组件：src/components/
整体布局：src/layouts/
源资源：src/assets/
原样静态资源：public/
本次修改字段：
本地验收页面：
检查结果：
```

目录名称可以查找，不必死记。真正需要记住的是“先判断职责，再从统一入口和类型找到正确文件”。

完成后，项目结构就不再是“看不懂的一堆文件”，而是一张可以按职责查找的地图。下一步可以继续学习如何 [新增一篇 Markdown 文章](/posts/write-first-code-with-ai/)，或者先通过 `git status --short` 检查当前改动。

## 失败时先检查什么

### 修改后页面没有变化

先确认修改的是 `src/config/siteConfig.ts`，并且保存了文件。再检查浏览器是否打开了当前项目的开发服务器地址，而不是 GitHub Pages 或其他本地项目。

继续按顺序检查：

1. VS Code 标签页是否仍显示未保存状态。
2. 终端是否在保存后触发重新编译。
3. 当前终端与编辑器是否打开同一个 Firefly 目录。
4. 搜索结果是否表明页面读取的是另一个配置字段。
5. 浏览器本地存储是否覆盖了某些可切换配置。

不要通过在多个文件中写入同一个新值来“碰运气”。先找到当前页面的实际数据来源。

### `pnpm check` 报 frontmatter 错误

检查 Markdown 文件顶部的 `---` 是否成对存在，日期是否使用合法格式，标签是否是数组。配置文件则检查引号、逗号和花括号是否完整。

如果这次只计划修改配置，却出现某篇 Markdown 的 frontmatter 错误，先看 `git status` 判断该文章是不是工作区中已有的另一项修改。不要为了通过检查而随意重写不属于当前目标的内容。

### `pnpm build` 报 TypeScript 或配置错误

查看第一条错误的文件路径和行号，不要只看最后一行。回到最近一次修改的位置，逐项恢复或修正。可以把完整错误、修改前后的差异和 Node.js/pnpm 版本一起交给 AI 分析。

常见原因包括：

- 字符串缺少引号或逗号；
- 布尔值写成了字符串，例如 `"false"`；
- 新字段不在对应类型中；
- 对象层级与类型定义不一致；
- 导入名称或导出名称拼写不同；
- 资源路径只在开发环境下偶然可用。

一次只修复第一条错误。后续错误可能只是由第一处语法或类型问题连带产生。

### 水波纹仍然出现

检查 `backgroundWallpaper.ts` 中桌面端和移动端是否都为 `false`，再检查 `switchable` 是否仍为 `true`。浏览器还可能保存了控制面板的 `localStorage` 状态，清除站点本地存储后重新打开页面。

### 不知道该改哪个文件

先描述目标，再搜索配置名称。例如“我要修改顶部菜单”对应 `navBarConfig.ts`，“我要修改侧栏组件”对应 `sidebarConfig.ts`，“我要修改文章内容”对应 `src/content/posts/`。不要根据文件名猜测后一次改很多文件。

可以向 AI 提供目标页面、想改变的结果、当前项目结构、相关搜索结果和不允许改变的内容，然后要求它先定位文件，不立即修改。

### 修改后出现大量无关差异

先执行：

```powershell
git status --short
git diff --stat
```

判断是格式化工具改写了整个文件、生成脚本更新了常量，还是工作区原本就有其他任务。不要直接执行 `git add .`，也不要用破坏性命令清空全部修改。只保留并提交与目标相关的文件。

### 配置修改在刷新后又恢复

先确认源文件中的值是否真的被改回。如果文件没有变化，但页面状态恢复，可能是浏览器本地存储、可切换设置或运行时状态覆盖了默认配置。默认值只决定首次状态，不一定覆盖用户已经保存的选择。

先在隐私窗口或清除该站点的本地存储后测试，确认是默认配置问题还是已保存状态问题。清除前注意不要误删其他网站数据。

### 图片本地正常但构建或线上缺失

检查图片位于 `src/assets/` 还是 `public/`，并确认引用方式符合该目录的处理规则。再执行 `pnpm build`，查看构建日志是否报告找不到文件。

路径大小写也要一致。Windows 文件系统有时不会暴露大小写问题，但部署环境可能严格区分 `Image.png` 和 `image.png`。

## 向 AI 反馈项目结构或配置问题的模板

```text
我的目标是：
希望改变的页面是：
我认为相关的目录或文件是：
我搜索的关键词是：
搜索结果是：
我修改的字段是：
git diff 是：
pnpm check / build 的第一条错误是：
浏览器实际结果是：
预期结果是：
不希望改变的内容是：

请先判断这个目标属于内容、配置、路由、布局、组件、样式还是资源，
再指出最可能的单个入口文件和一个最小验证步骤。
```

不要把整个仓库一次性贴给 AI。先提供项目地图、相关文件片段、搜索结果和实际错误，信息会更集中，也更容易得到可验证的下一步。
