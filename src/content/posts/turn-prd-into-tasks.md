> PRD 写完以后，不要立刻让 AI 一口气写完整网站。
> 真正稳妥的做法，是把 PRD 拆成一组可以逐步完成、逐步验证、逐步提交的小任务。

---

上一篇我们复盘了本站的 PRD 是怎么写出来的。

现在问题来了：

> **PRD 写好了，下一步到底做什么？**

很多人会在这里犯一个常见错误：直接把整份 PRD 丢给 AI，然后说：

```text
请根据这个 PRD 帮我完整实现网站。
```

AI 当然会努力做。

但这通常不是最好的方式。

因为一个完整网站里同时包含太多东西：

- 项目脚手架
- 页面布局
- 数据结构
- Markdown 文章系统
- 文章阅读页
- 目录生成
- 首页文章卡片
- 侧栏索引
- 响应式样式
- 交互跳转

如果一次全做，任何一个地方出错，你都很难判断问题到底出在哪里。

所以阶段四的第三步，是学会把 PRD 拆成任务。

---

## 为什么要拆任务

拆任务不是为了显得专业，而是为了降低失控概率。

你可以把 AI 想象成一个执行力很强的搭档。

如果你一次给它一个大到模糊的目标，它会自己补很多细节。

这些细节可能对，也可能不对。

但如果你把目标拆成一小步一小步，它每次只需要处理一个明确问题，结果就会稳定很多。

比如下面这两种说法：

### 不推荐

```text
帮我做完整个 AI 编程学习网站。
```

这个需求太大。

AI 需要同时决定技术栈、布局、数据结构、视觉风格、页面跳转、文章渲染方式。

### 推荐

```text
根据 PRD，先完成 M1：项目基础。
要求：
1. 使用 Vite + React + TypeScript
2. 建立基础目录结构
3. 添加阶段和工具的静态数据
4. 首页先能显示学习档案、阶段数量和工具数量
5. 完成后能 npm run build
```

这个需求就清楚得多。

AI 知道第一步只做项目基础，不会突然开始做复杂文章页。

---

## 拆任务的核心原则

拆任务时，可以记住一句话：

> **每个任务都应该有明确产物、明确边界、明确验证方式。**

这三个词很重要。

| 原则 | 含义 |
|------|------|
| 明确产物 | 做完以后应该多出什么东西 |
| 明确边界 | 这一步不做什么 |
| 明确验证 | 怎么知道它真的做好了 |

举个例子。

如果任务叫：

> 做首页。

这就太模糊。

首页包含什么？文章卡片做不做？右侧索引做不做？点击跳转做不做？

更好的写法是：

> M3：首页体验。首页需要展示文章预览卡片、6 个学习阶段、5 类工具索引和最近文章；点击文章标题或继续阅读进入文章页；不做搜索、不做评论、不做后台。

这样 AI 才知道边界。

---

## 本站的任务是怎么拆的

本站最终可以拆成几个里程碑。

这些里程碑不是一开始就完美设计出来的，而是在实现过程中逐渐清晰的。

但回头看，它们大概是这样的：

| 里程碑 | 目标 | 验证方式 |
|--------|------|----------|
| M1 | 项目基础 | 项目能启动，基础数据能显示，构建通过 |
| M2 | 信息架构 | 6 个阶段和 5 类工具索引能从数据层渲染 |
| M3 | 首页体验 | 首页能展示文章预览、阶段列表、工具索引 |
| M4 | 文章阅读 | 能打开单篇文章，渲染 Markdown，显示目录 |
| M5 | 侧栏索引 | Learning Stages、Tool Index、最近文章能跳转 |
| M6 | 视觉统一 | 首页和文章页三栏宽度、卡片风格、链接样式一致 |

这组任务有一个特点：它们不是按文件拆的，而是按用户能感知到的功能拆的。

用户不会关心你改了哪个组件。

用户只关心：

- 我能不能看到文章？
- 我能不能点击进去？
- 我能不能知道下一篇读什么？
- 页面看起来是不是统一？

所以任务也应该围绕这些体验来拆。

---

## M1：项目基础

M1 的目标不是把网站做漂亮。

M1 只解决一个问题：

> **这个项目能不能跑起来，并且有一套稳定的数据基础？**

它应该包含：

1. 初始化 Vite + React + TypeScript
2. 建立基本文件结构
3. 创建阶段和工具数据
4. 首页显示学习档案
5. 能通过 `npm run build`

它不应该包含：

- 复杂文章阅读页
- Markdown 渲染
- 文章目录
- 参考站视觉细节
- 部署上线

这就是边界。

如果 M1 做太多，后面会很难维护。

> **第一步不是把房子装修好，而是先把地基和承重结构搭起来。**

---

## M2：信息架构

M2 开始把 PRD 里的学习结构落到数据层。

本站有两条核心结构：

1. 6 个学习阶段
2. 5 类工具索引

所以 M2 的任务是让这些结构不再只是写在 PRD 里，而是变成代码里的数据。

比如阶段数据要包含：

- id
- 顺序
- 名称
- 主题
- 描述

工具数据要包含：

- id
- 名称
- 示例工具
- 描述

这样页面就不需要写死文案。

后续如果要改“对话篇”的主题，只改数据就可以。

这一步的验证方式很简单：

> 页面能根据数据渲染出 6 个阶段和 5 类工具。

---

## M3：首页体验

M3 是用户第一次真正看到网站形状的阶段。

它要解决的是：

> 用户打开首页时，能不能理解这个网站有什么内容、应该从哪里开始读？

所以首页需要有：

- 学习档案
- 文章预览卡片
- 6 个阶段
- Learning Stages
- Tool Index
- 最近文章

这里最重要的不是把样式做到最终版，而是把首页的信息组织跑通。

比如文章卡片要能展示：

- 标题
- 摘要
- 日期
- 所属阶段
- 继续阅读

做到这里，网站已经开始像一个内容站了。

---

## M4：文章阅读

M4 是本站最关键的一步。

因为一个内容站最终还是要让用户阅读文章。

这一阶段要做几件事：

1. 用 query 参数打开文章页
2. 从 Markdown 文件读取正文
3. 渲染 Markdown 内容
4. 根据标题生成文章目录
5. 点击目录能跳到对应小节
6. 文章页保留左右侧栏

这里有一个重要决策：

> 首页不直接展开完整文章，点击后进入独立文章页。

这让首页更像博客首页，也让阅读页更专注。

M4 的验证方式是：

```text
打开 /?article=write-first-code-with-ai
应该能看到文章正文、左侧学习档案和文章目录、右侧索引模块。
```

这就是明确验证。

---

## M5：侧栏索引和跳转

当文章多起来以后，光能阅读还不够。

用户还需要在文章之间移动。

所以 M5 主要优化跳转路径：

- 首页文章标题可以点击
- 继续阅读可以点击
- Learning Stages 下的文章可以点击
- Tool Index 下的文章可以点击
- 最近文章可以点击
- 文章页也保留这些入口

这个阶段看起来像“小修小补”，但体验提升很明显。

参考成熟博客时你会发现：

> 任何看起来像标题的文章文字，大概率都应该能点。

所以我们也同步了这个行为。

---

## M6：视觉统一

最后才进入视觉统一。

为什么不是一开始就做？

因为如果结构没定，视觉会反复推翻。

等首页、文章页、侧栏、文章系统都跑通以后，再统一视觉就更稳。

本站最后参考 Sukka's Blog，把宽度和气质统一成：

- 浅灰背景
- 白色卡片
- 细边框
- 轻阴影
- 蓝色链接
- 左右侧栏约 320px
- 中间正文约 820px
- 三栏间距统一

这一步的目标不是让页面变“炫”，而是让它更像一个长期可维护的技术博客。

---

## 如何把任务交给 AI

拆好任务以后，不要一次性全发。

一次只发一个任务。

比如：

```text
根据 PRD，完成 M2：信息架构。

要求：
1. 建立 6 个学习阶段的数据结构
2. 建立 5 类工具索引的数据结构
3. 首页从数据层渲染这些内容
4. 不做文章阅读页
5. 完成后运行 npm run build 验证
```

注意这里有一句：

> 不做文章阅读页。

这句话很重要。

它告诉 AI 这一步的边界。

AI 很容易热心过头，一次做很多东西。

你要学会给它划线。

---

## 每个任务完成后都要验证

AI 写完代码以后，不要只看它说“完成了”。

你要让它验证。

本站每一步都会做几类检查：

1. `npm run build` 是否通过
2. 首页是否能正常打开
3. 文章页 URL 是否返回 200
4. 新增文章是否进入 Learning Stages
5. 新增文章是否进入 Tool Index
6. Git 工作区有哪些变化

验证不是为了形式。

验证是为了防止你在错误基础上继续往前走。

> **小步提交，小步验证，是 AI 编程里最稳的节奏。**

---

## 一个可复用的拆任务模板

你以后拿到任何 PRD，都可以这样拆。

```text
请根据下面的 PRD，把项目拆成 5 到 8 个里程碑。

要求：
1. 每个里程碑只解决一个主要问题
2. 每个里程碑都写清楚目标
3. 每个里程碑都写清楚不做什么
4. 每个里程碑都写清楚验收方式
5. 按从基础到复杂的顺序排列

[粘贴 PRD]
```

然后你再让 AI 执行第一步：

```text
先做 M1。
只做 M1 范围内的事情。
完成后告诉我改了哪些文件，以及怎么验证。
```

这套流程非常适合新手。

因为你不需要一次理解整个项目。

你只需要理解下一步。

---

## 小结

PRD 写完以后，不要急着让 AI 一口气做完整网站。

更好的路线是：

1. 把 PRD 拆成 5 到 8 个里程碑
2. 每个里程碑只解决一个核心问题
3. 明确这一步做什么、不做什么
4. 每步完成后都验证
5. 验证通过后再进入下一步

本站就是这样一步一步做出来的。

先 M1 项目基础。

再 M2 信息架构。

再 M3 首页体验。

再 M4 文章阅读。

再 M5 侧栏跳转。

最后 M6 视觉统一。

这不是最炫的方式，但它很稳。

而对新手来说，稳定比炫技重要得多。

> **AI 可以跑得很快，你要负责让它跑在正确的路线上。**
