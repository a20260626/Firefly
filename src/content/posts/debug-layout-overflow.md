> 有些问题不会让页面白屏，也不会让构建失败。
> 页面能打开，Console 也没有红色错误，但正文被撑宽、侧栏错位、表格溢出，这就是样式类调试要解决的问题。

---

上一篇我们练习了页面白屏但构建通过时，如何用浏览器 Console 找运行时错误。

那类问题通常有红色报错。

但还有一类问题更安静：

> **没有报错，但页面看起来坏了。**

比如：

- 文章正文突然变得特别宽
- 表格把卡片撑破
- 代码块横向溢出
- 长链接顶出容器
- 图片超过正文宽度
- 左右侧栏被挤到奇怪的位置

这类问题通常不是 JavaScript 报错。

而是 CSS、内容宽度和布局约束的问题。

所以这一篇要练习的是样式类调试。

---

## 样式问题和运行时错误不一样

运行时错误通常会在 Console 里给出红色提示。

比如：

```text
TypeError: Cannot read properties of undefined
```

但样式问题不一定有报错。

浏览器会照常渲染页面。

只是渲染结果不是你想要的。

所以调试样式时，不能只等错误信息。

你要学会观察页面：

| 现象 | 可能方向 |
|------|----------|
| 页面横向滚动 | 某个元素太宽 |
| 正文卡片被撑开 | 表格、代码块、长链接或图片溢出 |
| 侧栏被挤掉 | 三栏总宽度超过容器 |
| 按钮文字挤出 | 按钮宽度或换行规则不合理 |
| 移动端布局乱 | 响应式断点没有处理好 |

样式问题的关键词是：

> **找出是谁把容器撑坏了。**

---

## 第一步：先判断是全站问题还是单篇文章问题

遇到布局坏了，不要马上改 CSS。

先判断范围。

你可以打开几个页面：

1. 首页
2. 当前文章页
3. 前一篇文章页
4. 有表格的文章页
5. 有代码块的文章页

然后观察：

| 结果 | 说明 |
|------|------|
| 所有页面都坏 | 可能是全局布局或宽度变量问题 |
| 只有文章页坏 | 可能是文章正文样式问题 |
| 只有某一篇文章坏 | 可能是这篇文章内容太宽 |
| 只有移动端坏 | 可能是响应式断点问题 |

这个判断很重要。

如果只有一篇文章坏，就不要一上来重写全站布局。

先看那篇文章里有没有表格、长代码、长链接或图片。

---

## 第二步：用开发者工具找溢出元素

页面横向滚动或正文被撑宽时，可以用浏览器开发者工具检查。

打开 Elements 面板，逐层看：

- `body`
- `.site-shell`
- `.layout-grid`
- `.article-card`
- `.article-body`
- `table`
- `pre`
- `code`
- `img`

通常把页面撑宽的是这些元素：

| 元素 | 常见问题 |
|------|----------|
| `table` | 单元格内容太长，不能换行 |
| `pre` | 代码块默认不换行 |
| `code` | 行内代码里有很长字符串 |
| `a` | 长链接没有断行 |
| `img` | 图片没有 `max-width: 100%` |
| grid 子项 | 没有 `min-width: 0`，导致不能收缩 |

如果你不知道怎么看，可以把现象告诉 AI：

```text
文章页没有报错，但页面出现横向滚动。
我怀疑是正文内容撑宽了。
请告诉我应该优先检查哪些 CSS 选择器和元素。
```

这比直接说"页面丑了帮我改"更有效。

---

## 第三步：先处理容器收缩

在 CSS Grid 或 Flex 布局里，一个很常见的问题是：

> 子元素默认不愿意缩小。

即使外层设置了宽度，里面的内容也可能把它撑开。

这种情况下，常用修复是给关键容器加：

```css
min-width: 0;
```

比如文章卡片：

```css
.article-card {
  min-width: 0;
  overflow: hidden;
}
```

这句话的意思是：

> 允许这个卡片在网格里正常收缩，不要被内部内容强行撑宽。

很多宽度问题看起来像表格或代码块导致的。

但根源可能是父容器没有允许收缩。

所以排查时要同时看父容器和内部元素。

---

## 第四步：处理表格溢出

Markdown 文章里最容易撑宽正文的是表格。

尤其是表格里有长词、长路径或长 URL。

如果表格没有处理，可能会把整个正文卡片撑开。

常见修复思路：

```css
.article-body table {
  width: 100%;
  max-width: 100%;
  border-collapse: collapse;
}

.article-body th,
.article-body td {
  white-space: normal;
  overflow-wrap: anywhere;
  vertical-align: top;
}
```

这里几个点很重要：

- `max-width: 100%` 限制表格不超过正文
- `white-space: normal` 允许内容换行
- `overflow-wrap: anywhere` 允许长字符串断开
- `vertical-align: top` 让多行内容更易读

表格不是不能宽。

但它不能破坏整个页面。

---

## 第五步：处理代码块溢出

代码块也很常见。

Markdown 里的代码块通常会被渲染成：

```html
<pre><code>...</code></pre>
```

默认情况下，`pre` 会保留空格和换行，不主动折行。

这对代码可读性有好处，但也容易横向溢出。

可以根据项目风格选择两种方式。

第一种：允许代码块内部横向滚动。

```css
.article-body pre {
  max-width: 100%;
  overflow-x: auto;
}
```

第二种：允许长代码换行。

```css
.article-body pre code {
  white-space: pre-wrap;
  word-break: break-word;
}
```

本站这类学习文章更重视不撑破布局，所以可以适当允许换行。

如果是专业代码文档，也可以保留横向滚动。

关键是：

> 代码块可以自己滚动，但不能把整个页面撑宽。

---

## 第六步：处理长链接和长单词

长链接是另一个隐藏杀手。

比如：

```text
https://example.com/a/very/very/very/long/path/that/does/not/break
```

如果浏览器找不到合适断点，它可能会把整行撑开。

可以在正文容器上加：

```css
.article-body {
  overflow-wrap: anywhere;
}
```

这会让长链接、长路径、长变量名有机会断行。

它对内容站很有用。

因为文章里经常会出现：

- URL
- 文件路径
- npm 包名
- Git commit
- 长命令
- 长变量名

这些都可能让布局出问题。

---

## 第七步：处理图片宽度

图片也要限制。

尤其是截图。

如果图片原图很宽，直接显示可能会超过正文区域。

常见做法是：

```css
.article-body img {
  display: block;
  max-width: 100%;
  height: auto;
}
```

这样图片会按正文宽度缩放。

如果图片外面有 `figure`，也要确保：

```css
.article-image {
  max-width: 100%;
}
```

图片可以大，但不能突破正文容器。

这是内容站的基本防线。

---

## 第八步：检查三栏总宽度

如果不是单篇文章撑宽，而是整个页面布局不对，就要看三栏总宽度。

本站三栏大概由这些变量控制：

```css
--layout-sidebar-width: 320px;
--layout-content-width: 820px;
--layout-gap: 32px;
```

总宽度大致是：

```text
左侧栏 + 中间栏 + 右侧栏 + 两个间距
```

也就是：

```text
320 + 820 + 320 + 32 * 2 = 1524px
```

如果浏览器窗口比这个窄，布局就需要响应式处理。

比如在中等屏幕下改成两栏或单栏。

在移动端改成单列。

如果没有响应式断点，页面就可能横向滚动。

所以布局问题不只看内容，也要看视口宽度。

---

## 第九步：用二分法确认是哪段内容撑坏

如果你怀疑某篇文章内容导致溢出，但看不出来是哪一段，可以用二分法。

方法是：

1. 临时注释掉文章后一半内容
2. 看布局是否恢复
3. 如果恢复，问题在后半段
4. 如果没恢复，问题在前半段
5. 继续缩小范围

这听起来笨，但很有效。

尤其是长文里有很多表格、代码块、链接时。

当然，真正改文件前要小心，不要把内容弄丢。

可以先复制出问题段落，或者让 AI 帮你只定位，不做大规模删除。

---

## 给 AI 的布局调试模板

遇到布局坏了，可以这样问 AI：

```text
页面没有报错，npm run build 也通过，但文章页布局被撑宽了。

现象：
[描述页面横向滚动 / 正文卡片变宽 / 表格溢出 / 代码块溢出]

当前 URL：
[粘贴 URL]

最近改动：
[说明刚才改了哪篇文章、哪个 CSS 或哪个组件]

请你：
1. 先判断这是全局布局问题，还是单篇文章内容问题
2. 列出最可能撑宽页面的元素
3. 优先检查 article-card、article-body、table、pre、code、img、a
4. 给出最小 CSS 修改方案
5. 修改后告诉我如何验证
```

这个模板会让 AI 先判断范围。

范围判断对样式问题特别重要。

---

## 验证布局是否修好

修完后，不要只看一个屏幕。

至少检查：

1. 首页是否正常
2. 当前文章页是否正常
3. 有表格的文章页是否正常
4. 有代码块的文章页是否正常
5. 有图片的文章页是否正常
6. 浏览器窗口变窄时是否正常
7. 页面是否没有横向滚动条
8. `npm run build` 是否通过

如果只看当前大屏，很容易漏掉移动端。

如果只看普通段落文章，很容易漏掉表格和代码块。

样式修复要用复杂内容验证。

---

## 常见问题

### 1. 代码块应该换行还是横向滚动？

看网站类型。

学习笔记站可以适当换行，避免破坏布局。

如果是专业 API 文档，横向滚动也可以接受。

关键是不要让整个页面横向滚动。

### 2. 为什么加了 max-width 还会撑开？

可能是父容器没有 `min-width: 0`。

在 Grid 或 Flex 布局里，这个问题很常见。

所以要同时检查父容器和子元素。

### 3. 没有报错的问题也要让 AI 修吗？

可以，但要描述清楚现象。

比如：

```text
文章页出现横向滚动，怀疑是表格或代码块撑宽。
```

不要只说"页面不好看"。

越具体，AI 越容易定位。

---

## 小结

这一篇练习的是样式类调试。

你需要记住这条排查线：

1. 先判断是全站问题、文章页问题，还是单篇文章问题
2. 用开发者工具找可能溢出的元素
3. 父容器先检查 `min-width: 0`
4. 表格要限制宽度并允许换行
5. 代码块要自己处理滚动或换行
6. 长链接要允许断行
7. 图片要 `max-width: 100%`
8. 三栏布局要考虑总宽度和响应式
9. 修完后用复杂内容和窄屏验证

样式问题没有红色报错，但它一样有线索。

> **页面被撑坏时，先找最宽的那个东西。**
