> 信息架构已经有了，接下来要让用户真正看见它。
> 首页不是把所有东西随便堆出来，而是把文章、阶段、工具和最近更新组织成一条清楚的阅读入口。

---

上一篇我们把 PRD 里的 6 个阶段和 5 类工具变成了数据结构。

这一步很关键，但它更多发生在代码里。

用户打开网站时，真正看到的是首页。

所以这一篇要进入阶段四的下一步：

> **把信息架构变成首页体验。**

首页要回答三个问题：

1. 这个网站有什么内容？
2. 我应该从哪篇文章开始？
3. 如果我想按阶段或工具查找，入口在哪里？

如果这三个问题回答清楚，首页就不是一个装饰页面，而是一个真正可用的内容导航。

---

## 首页不是 landing page

很多新手做网站时，第一反应是做一个大大的 hero：

- 一行很大的标题
- 一句宣传语
- 一个"立即开始"按钮
- 几张漂亮卡片

这对产品官网有用。

但本站不是产品官网。

本站是一个学习内容站。

内容站的首页更像一个"入口大厅"：

- 用户可以看到最新文章
- 用户可以知道学习顺序
- 用户可以从工具分类进入
- 用户可以继续阅读下一篇

所以首页的重点不是喊口号，而是帮助用户进入内容。

这也是为什么我们没有把首页做成纯粹的欢迎页。

我们要让用户一打开网站，就能看到文章和路径。

---

## 首页需要哪些模块

根据 PRD，首页最终需要几个核心模块：

| 模块 | 作用 |
|------|------|
| 学习档案 | 告诉用户本站当前有多少文章、阶段和工具 |
| 文章预览 | 展示主要文章，提供进入阅读页的入口 |
| Learning Stages | 按 6 个阶段展示学习路线 |
| Tool Index | 按 5 类工具组织文章 |
| 最近文章 | 让用户快速看到最新内容 |

这些模块不是为了填满页面。

它们各自负责一个阅读场景。

学习档案负责建立整体感。

文章预览负责把内容展示出来。

Learning Stages 负责学习路线。

Tool Index 负责工具查找。

最近文章负责更新感。

首页要做的事，就是把这些模块放到合适的位置。

---

## 中间区域：文章预览是主角

首页最重要的区域应该留给文章。

因为用户来这里，最终是为了阅读。

所以中间区域适合放文章预览卡片。

每张文章卡片至少要包含：

| 信息 | 为什么需要 |
|------|------------|
| 标题 | 用户判断要不要点进去的第一依据 |
| 摘要 | 让用户快速知道这篇文章讲什么 |
| 日期 | 让用户知道内容更新节奏 |
| 阶段 | 让用户知道文章在学习路径里的位置 |
| 继续阅读 | 给出明确的进入文章页动作 |

标题本身也应该能点击。

因为在成熟博客里，文章标题通常就是文章链接。

如果只有"继续阅读"能点，用户会多绕一步。

所以首页文章卡片应该支持两个入口：

1. 点击标题进入文章页
2. 点击继续阅读进入文章页

这是一件小事，但会让页面顺手很多。

---

## 摘要不要写成说明书

文章预览里的摘要很重要。

它不是完整正文，也不是功能说明。

它的任务是让用户用几秒钟判断：

> 这篇是不是我现在想读的？

比如一篇文章叫《用 AI 写第一行代码》。

摘要可以写：

> 从最小需求开始，学会把一句模糊想法改成 AI 能执行的清楚指令。

这比"本文介绍如何使用 AI 编程工具"更有用。

好的摘要应该具备三个特点：

1. 说清楚问题
2. 暗示阅读收益
3. 不展开所有细节

如果摘要太长，首页会变得像文章正文。

如果摘要太短，用户又不知道文章值不值得点。

所以首页摘要要克制。

---

## 左侧：学习档案给用户一个总览

首页左侧适合放学习档案。

它不需要复杂，但要稳定。

比如：

| 指标 | 内容 |
|------|------|
| 文章 | 当前文章数量 |
| 阶段 | 6 |
| 工具 | 5 |

这个模块的意义不是炫耀数量。

它告诉用户：

> 这是一个有结构、会持续增长的学习站。

当文章越来越多时，学习档案会成为站点状态的一个小窗口。

而且它也能验证数据是否正常：

- 新增文章后，文章数量应该变化
- 阶段数量应该保持 6
- 工具数量应该保持 5

这就是一个既给用户看，也给开发者验收的模块。

---

## 右侧：Learning Stages 是学习路线

Learning Stages 不只是一个分类列表。

它代表用户的学习顺序。

所以右侧模块应该按照阶段顺序显示：

1. 认知篇
2. 起步篇
3. 对话篇
4. 实战篇
5. 调试篇
6. 发布篇

每个阶段下面可以显示对应文章链接。

这样用户就能知道：

- 现在有哪些阶段
- 每个阶段有哪些文章
- 下一步应该读什么

这比单纯显示"分类"更友好。

因为新手最怕的不是内容少，而是不知道先看哪一个。

Learning Stages 的价值，就是把阅读顺序亮出来。

---

## Tool Index 是另一条查找路径

如果 Learning Stages 解决的是"按顺序学"，那么 Tool Index 解决的就是"按问题查"。

用户可能不是每次都从第一篇开始读。

他可能会直接想找：

- Git 怎么用
- Codex 怎么提需求
- VS Code 是什么
- 部署到哪里
- HTML / CSS / JavaScript 分别负责什么

这时工具索引就有用了。

它按用户视角把文章分到 5 类：

| 工具 | 用户想解决的问题 |
|------|----------------|
| ai编程助手 | 怎么让 AI 帮我写代码 |
| ai IDE/编辑器 | 在哪里写代码、看文件、运行项目 |
| 版本控制 | 怎么保存、提交、回退 |
| 前端基础 | 网页本身由什么组成 |
| 部署发布 | 怎么让别人看到作品 |

一篇文章可以进入多个工具分类。

比如这篇首页体验文章，既和前端基础有关，也和 AI 编程助手有关。

这让工具索引比普通分类更灵活。

---

## 最近文章让网站有更新感

内容站需要让用户知道它还活着。

最近文章模块就负责这件事。

它可以按 `updatedAt` 或文章顺序展示最新几篇。

这个模块不需要很大。

但它能提供一个很自然的入口：

> 我不想按阶段看，我就想看最近更新了什么。

对于持续写作的网站来说，最近文章很有价值。

它也会鼓励作者保持小步更新。

因为每新增一篇文章，首页就会发生一点变化。

---

## 首页布局为什么适合三栏

本站参考的是清爽技术博客风格。

所以首页采用了三栏结构：

| 位置 | 模块 |
|------|------|
| 左侧 | 学习档案 |
| 中间 | 文章预览和阶段正文区域 |
| 右侧 | Learning Stages、Tool Index、最近文章 |

三栏的好处是：

1. 中间专注阅读和浏览文章
2. 左侧提供站点总览
3. 右侧提供导航和跳转

这和很多技术博客的阅读习惯一致。

但三栏也要注意宽度。

中间正文太窄，文章卡片会拥挤。

侧栏太宽，主内容会被压缩。

所以布局应该保持克制：

- 左侧固定为一个适合卡片的宽度
- 中间给文章预览足够空间
- 右侧保持导航模块可读
- 三栏之间留出稳定间距

视觉不是为了好看而好看。

它要服务阅读路径。

---

## 首页和文章页要保持一致

首页做完后，还要考虑文章页。

如果首页是三栏，文章页突然变成完全不同的布局，用户会觉得断开。

所以本站后来保持了几个一致：

- 首页有学习档案，文章页也有
- 首页有 Learning Stages，文章页也有
- 首页有 Tool Index，文章页也有
- 首页和文章页使用相同的侧栏宽度
- 首页和文章页的链接样式保持一致

这样用户从首页进入文章时，会感觉自己仍然在同一个系统里。

这就是内容站的连续性。

## 如何让 AI 帮你做首页体验

你可以把任务这样交给 AI：

```text
请完成 M3：首页体验。

目标：
基于现有阶段、工具和文章数据，搭建内容站首页。

要求：
1. 首页保留学习档案模块
2. 中间区域展示文章预览卡片
3. 文章标题和继续阅读都能跳转到文章页
4. 添加 Learning Stages，按 6 个阶段显示文章链接
5. 添加 Tool Index，按 5 类工具显示文章链接
6. 添加最近文章模块
7. 首页和文章页后续要能复用同一套数据
8. 不做完整 Markdown 阅读器，不做复杂搜索和评论
9. 完成后运行 npm run build 验证
```

这段提示词里最重要的是边界。

它明确说：

> 不做完整 Markdown 阅读器，不做复杂搜索和评论。

因为首页体验这一步只解决"入口"问题。

文章阅读页可以放到下一步。

这样项目会更稳。

## 验收首页体验

首页完成后，可以按下面这些问题检查：

1. 首页能否显示文章预览？
2. 每篇文章是否有标题、摘要、日期和阶段？
3. 点击文章标题是否能进入文章页？
4. 点击继续阅读是否能进入文章页？
5. Learning Stages 是否显示 6 个阶段？
6. Tool Index 是否显示 5 类工具？
7. 最近文章是否显示最新内容？
8. 新增文章后，首页是否能自动更新？
9. `npm run build` 是否通过？

这些检查可以帮助你判断：

首页是不是只是"看起来有东西"，还是已经真的和数据系统连起来了。

## 常见问题

### 1. 首页要不要放完整文章？

第一版不建议。

首页应该是入口，不是阅读页。

如果首页直接展开完整文章，会变得很长，也会削弱文章页的作用。

更好的方式是显示摘要和继续阅读。

### 2. 首页文章要显示多少篇？

第一版可以全部显示。

等文章数量多了，再考虑分页、筛选或只显示最近几篇。

不要一开始就做复杂列表。

先把核心阅读路径跑通更重要。

### 3. 阶段列表和工具索引会不会重复？

会有一些重复，但这是有意义的。

同一篇文章可以同时出现在阶段和工具里。

阶段是学习顺序。

工具是查找入口。

它们服务的是不同场景。

## 小结

这一篇做的是首页体验。

它把上一篇的信息架构真正展示给用户。

你需要记住这条线：

1. 首页不是 landing page，而是内容入口
2. 中间区域以文章预览为主
3. 标题和继续阅读都应该能跳转
4. Learning Stages 负责学习顺序
5. Tool Index 负责工具查找
6. 最近文章负责更新感
7. 首页和文章页要共享同一套结构和宽度

做到这里，网站已经不只是能运行，也不只是有数据。

它开始像一个真正的内容站。

> **首页的任务，是让用户知道这里有什么，并且知道下一步该点哪里。**
