> 项目做起来以后，报错一定会出现。
> 调试篇的第一件事，不是背错误大全，而是学会把现场整理清楚，让 AI 帮你一起定位问题。

---

上一篇我们完成了阶段四的项目闭环检查。

从 PRD 到任务，从首页到文章页，从导航到视觉，再到构建和 Git，整个实战项目已经跑通了一轮。

但真实开发不会永远顺利。

只要继续写代码，就一定会遇到：

- 页面白屏
- 构建失败
- 样式错位
- 点击没反应
- Markdown 渲染异常
- 图片路径不对
- Git 状态看不懂

这些不是失败。

这些是开发的一部分。

所以从这一篇开始，我们进入阶段五：

> **调试篇：翻车与修复。**

调试篇要解决的不是"如何永远不出错"。

而是：

> **出错以后，如何不慌，并且一步一步把问题找出来。**

---

## 新手最容易犯的调试错误

新手遇到报错时，最常见的反应是直接把一句话丢给 AI：

```text
我的页面坏了，帮我修。
```

这句话太模糊。

AI 不知道：

- 是哪里坏了？
- 什么时候坏的？
- 改了哪个文件？
- 浏览器有没有报错？
- 终端有没有报错？
- 是本地开发坏了，还是构建坏了？
- 是只有一个页面坏了，还是全站都坏了？

所以它只能猜。

猜对了当然好。

猜错了，你就会跟着它绕圈。

调试的第一课是：

> **不要只描述感受，要提供现场。**

比如不要只说"白屏了"。

而要说：

```text
我运行 npm run dev 后，打开 http://127.0.0.1:5177 页面白屏。
浏览器 Console 里有这个错误：
[粘贴完整错误]

我刚才修改了 src/data/articles.ts 和 src/content/articles/xxx.md。
请帮我判断可能原因。
```

这就是一个能让 AI 工作的调试请求。

---

## Debug 的核心不是修，是定位

很多人以为调试就是"修 bug"。

其实调试更重要的是定位。

你要先弄清楚：

| 问题 | 要判断什么 |
|------|------------|
| 哪里出错 | 浏览器、终端、构建、样式、数据、路由？ |
| 什么时候出错 | 改完哪个文件之后开始的？ |
| 出错范围 | 全站坏了，还是某一篇文章坏了？ |
| 是否可复现 | 每次都会出现，还是偶尔出现？ |
| 有没有错误信息 | Console、终端、构建日志有没有提示？ |

只有定位清楚，修复才会稳定。

否则很容易变成：

> 改一点，刷新一下，没好，再问 AI，再改一点。

这种方式会让项目越来越乱。

更好的方式是先缩小范围。

先确定问题在哪一层，再动手改。

---

## 第一步：分清错误发生在哪里

做网页项目时，错误大概会出现在几个地方。

| 位置 | 常见表现 |
|------|----------|
| 终端 | `npm run dev` 或 `npm run build` 报错 |
| 浏览器 Console | 页面白屏、组件报错、运行时报错 |
| 页面视觉 | 页面能打开，但布局错位、文字溢出 |
| 数据层 | 某篇文章不显示、阶段或工具归类错误 |
| 跳转逻辑 | 点击没反应、URL 变了但内容没变 |
| Git | 不知道哪些文件改了，或者提交前状态混乱 |

调试时先问自己：

> 这个问题是在终端里发生，还是在浏览器里发生？

如果终端已经报错，先看终端。

如果终端没报错，但页面白屏，打开浏览器开发者工具看 Console。

如果页面能打开但样式错，重点看 CSS 和元素宽度。

不要一上来就让 AI 改所有文件。

先把问题分层。

---

## 第二步：复制完整错误，而不是截半句

AI 调试很依赖错误信息。

所以复制错误时，不要只复制最后一行。

尽量复制完整上下文。

比如构建失败时，终端可能会显示：

```text
src/data/articles.ts:7:42 - error TS2307: Cannot find module '../content/articles/example.md?raw'
```

这行信息里已经包含很多线索：

- 出错文件是 `src/data/articles.ts`
- 出错位置是第 7 行
- 错误类型是找不到模块
- 可能是文章文件名或路径写错了

如果你只告诉 AI：

```text
构建失败了。
```

它就要猜。

如果你把完整错误给它，它就能直接分析。

---

## 第三步：告诉 AI 你刚才改了什么

报错通常和最近改动有关。

所以调试时要把最近动作告诉 AI。

比如：

```text
刚才我新增了一篇 Markdown 文章：
src/content/articles/debug-with-ai-first-time.md

同时修改了：
src/data/articles.ts

然后 npm run build 报错。
```

这比单纯贴错误更有用。

因为 AI 可以把错误和改动联系起来。

很多问题其实都很简单：

- import 名字写错
- slug 和文件名不一致
- frontmatter 少了 `tools`
- stage 名称不在 taxonomy 里
- Markdown 表格格式写坏

这些问题只要有"错误信息 + 最近改动"，就很好定位。

---

## 第四步：一次只改一个可疑点

调试时最怕一口气改很多地方。

因为你会不知道到底是哪一处修好了问题。

更稳的做法是：

1. 先判断最可能的原因
2. 只改这个原因相关的文件
3. 重新运行验证
4. 如果没好，再继续缩小范围

比如新增文章后构建失败。

先不要改样式、组件、路由。

先检查：

- Markdown 文件是否存在
- import 路径是否正确
- slug 是否注册
- frontmatter 是否完整
- stage / tools 是否和 taxonomy 匹配

这就是"按最小范围排查"。

AI 很适合帮你列出这个排查顺序。

你可以问：

```text
请不要直接大改代码。
先根据这个错误，列出最可能的 3 个原因，并告诉我应该先检查哪个文件。
```

这句话很有用。

它会让 AI 先做判断，而不是立刻乱改。

---

## 第五步：用构建命令做验证

修完以后，一定要验证。

不要只看 AI 说"应该好了"。

对这个项目来说，最基本的验证是：

```bash
npm run build
```

如果构建通过，至少说明：

- TypeScript 没有硬错误
- Markdown 导入能被打包
- 文章注册表没有明显类型问题
- Vite 能生成生产版本

如果构建失败，就继续看错误。

如果构建通过，但页面仍然不对，就去浏览器里检查具体 URL。

比如：

```text
http://127.0.0.1:5177/?article=debug-with-ai-first-time
```

这样可以确认文章页是否真的能打开。

---

## 第六步：让 AI 解释错误，而不是只给补丁

新手调试时，不要只让 AI 修。

更好的方式是让它解释。

比如：

```text
请先解释这个错误是什么意思。
然后告诉我它最可能和哪个文件有关。
最后再给出最小修改方案。
```

这会让你学到东西。

长期来看，你会逐渐看懂常见错误：

- `Cannot find module` 通常是路径或文件名问题
- `has invalid frontmatter` 通常是文章元数据问题
- `is not assignable to type` 通常是 TypeScript 类型不匹配
- 页面白屏但构建通过，可能是运行时错误
- 样式错位但没有报错，通常要查 CSS 和布局

调试篇的目标不是让你背所有错误。

而是让你形成判断路径。

---

## 一个好用的 AI Debug 模板

以后遇到问题，可以直接套这个模板：

```text
我在这个项目里遇到了一个问题，请帮我 debug。

现象：
[页面白屏 / 构建失败 / 点击没反应 / 样式错位]

我刚才做了什么：
[说明最近修改的文件和操作]

错误信息：
[粘贴终端或浏览器 Console 的完整错误]

相关文件：
[列出你认为相关的文件路径]

请你：
1. 先解释错误是什么意思
2. 判断最可能的原因
3. 给出排查顺序
4. 只在必要时给出最小修改方案
5. 修改后告诉我如何验证
```

这比一句"帮我修 bug"强太多。

因为它给了 AI 足够上下文，也限制了它不要乱改。

## 本站最常见的几类问题

结合这个网站，后续最容易遇到这些问题。

### 1. 新文章打不开

常见原因：

- Markdown 文件没放到 `src/content/articles`
- `articles.ts` 没导入
- slug 写错
- URL 里的 `article` 参数不对

优先检查文章注册表。

### 2. 构建提示 frontmatter 错误

常见原因：

- 缺少 `title`
- 缺少 `stage`
- 缺少 `tools`
- `tools` 没写成列表
- `stage` 名称和 `taxonomy.ts` 不一致

优先检查 Markdown 顶部的元数据。

### 3. 文章进入了错误分类

常见原因：

- `stage` 写错
- `tools` 少写或写错
- 分类名称和 taxonomy 不完全一致

优先检查 frontmatter 里的 `stage` 和 `tools`。

### 4. 页面宽度被撑开

常见原因：

- 表格太宽
- 代码块太长
- 长链接没有换行
- 图片没有限制最大宽度

优先检查正文 CSS。

### 5. 点击链接后内容没变

常见原因：

- 没有调用统一的打开文章函数
- URL 更新了但状态没更新
- slug 不存在
- 点击后没有滚动到顶部，误以为没变

优先检查跳转逻辑。

## 常见问题

### 1. 把完整错误贴给 AI 会不会太长？

一般不会。

错误上下文越完整，AI 越容易判断。

如果日志特别长，可以先贴最相关的一段：

- 报错文件
- 报错行号
- 错误类型
- 最后几行堆栈

### 2. AI 让我改很多文件怎么办？

可以让它收敛。

比如说：

```text
请先不要大改。
只给我最小修改方案，并说明为什么只需要改这些文件。
```

调试时，小改动通常更安全。

### 3. 没有报错，但页面就是不对怎么办？

这种通常是样式或逻辑问题。

你需要描述可见现象：

- 哪个页面
- 哪个模块
- 预期是什么
- 实际是什么
- 什么时候开始不对

如果有截图，也可以给 AI 看。

---

## 小结

这一篇是调试篇的开篇。

你需要记住的不是某个具体错误，而是调试顺序：

1. 不要只说"坏了"，要描述现象
2. 先分清错误发生在终端、浏览器、页面、数据还是 Git
3. 复制完整错误信息
4. 告诉 AI 最近改了什么
5. 一次只改一个可疑点
6. 修完后运行 `npm run build`
7. 需要时打开具体文章 URL 验证
8. 让 AI 解释错误，而不是只给补丁

调试不是和错误对抗。

调试是把混乱的现场整理成 AI 和你都能理解的信息。

> **报错不可怕，可怕的是不知道从哪里开始看。**
