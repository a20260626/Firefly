> “学 AI 编程”是一个方向，不是一个可以直接执行的问题。

很多人第一次使用 AI 编程时，会直接输入：“教我编程”“帮我做一个网站”或“这个项目为什么不能运行”。这些话表达了愿望，却没有提供足够的信息。AI 只能猜测你的目标、环境和限制，得到的回答自然可能很长，却不一定能解决眼前的问题。

真正有效的第一步不是马上写代码，而是把愿望改写成一个**有范围、有上下文、有完成标准的问题**。这篇文章只解决这一件事。

## 问题是什么

一个清晰的问题，至少要让自己和 AI 知道下面六件事：

| 元素 | 要回答的问题 | Firefly 示例 |
| --- | --- | --- |
| 目标 | 最终想得到什么结果？ | 在 Windows 电脑上运行 Firefly 博客 |
| 现状 | 现在已经做到哪里？ | 已克隆仓库，还没有安装依赖 |
| 环境 | 在什么系统、版本和目录中操作？ | Windows、Node.js 22、Firefly 根目录 |
| 约束 | 必须遵守哪些条件？ | 使用 pnpm，不更换项目技术栈 |
| 输出 | 这一次希望得到什么？ | 下一条需要执行的命令 |
| 验收 | 怎样证明这一步完成了？ | 浏览器能够打开本地首页 |

只有主题，没有这些元素时，它仍然是一个模糊目标。

例如：

```text
模糊目标：我要学 AI 编程。
```

可以先改写成：

```text
清晰问题：
我想使用 AI 辅助，在 Windows 电脑上把 Firefly 博客运行起来，
然后新增一篇 Markdown 文章，并部署到 GitHub Pages。
我目前已经有 GitHub 账号，但还没有准备开发环境。
请先告诉我第一个最小执行步骤，并说明如何验证它已经完成。
```

前一句只能得到宽泛的学习建议；后一句可以直接产生下一步行动。

清晰不等于写得很长。判断问题是否清晰，只需要看它能不能回答三个核心问题：

1. **现在在哪里？**
2. **下一步要到哪里？**
3. **怎样知道已经到达？**

## 适用环境是什么

这套方法不依赖某一种编程语言，也不只适用于 Firefly。下面这些情况都可以使用：

- 第一次向 ChatGPT 或其他 AI 编程助手提问。
- 安装 Node.js、pnpm、Git 等开发工具。
- 项目无法启动、构建失败或页面显示异常。
- 想修改配置，但不知道应该改哪个文件。
- 想新增功能，但需求还只是一个想法。
- 搜索了很多教程，却不知道哪一篇适合当前项目。
- 学习做饭、写作、设计等非编程技能，需要把大目标拆成行动。

尤其当你出现下面三种感觉时，应当先停下来重新定义问题：

- 不知道第一步做什么。
- AI 一次给出十几个步骤，不知道从哪里开始。
- 操作失败后，只能说“不能用”“没反应”或“报错了”。

本系列后续会以 Windows、Firefly、Node.js 和 GitHub Pages 为具体环境，但“先定义问题”的方法可以迁移到其他系统和项目。

## 具体怎么做

### 第一步：写下原始问题

先保留最自然的说法，不要急着修饰。

```text
我想学 AI 编程。
```

这句话的作用是确定方向。接下来要把方向变成任务。

### 第二步：填写问题六要素

复制下面的模板，把空白处补完整：

```text
我的目标是：
我的当前进度是：
我的操作环境是：
必须遵守的约束是：
我这次希望得到的输出是：
判断完成的标准是：
```

如果某一项暂时不知道，可以直接写“不知道”，并让 AI 帮助确认。不要用猜测冒充事实。

以运行 Firefly 为例：

```text
我的目标是：在本地运行 Firefly 博客。
我的当前进度是：已经用 GitHub Desktop 克隆仓库。
我的操作环境是：Windows 电脑，当前位于 Firefly 项目根目录。
必须遵守的约束是：使用项目指定的 pnpm，Node.js 版本不低于 22。
我这次希望得到的输出是：下一条需要执行的命令。
判断完成的标准是：终端显示本地地址，浏览器能打开博客首页。
```

### 第三步：给问题分类

分类不是为了增加术语，而是为了缩小搜索和检查范围。

| 类型 | 常见信号 | 首先关注 |
| --- | --- | --- |
| 目标问题 | “我想做……” | 结果与验收标准 |
| 环境问题 | 命令不存在、版本不符 | 系统、版本、安装路径 |
| 操作问题 | 不知道点哪里、改哪个文件 | 当前页面、目录和具体对象 |
| 运行问题 | 启动失败、终端报错 | 执行命令与完整报错 |
| 显示问题 | 页面空白、样式异常 | 页面地址、控制台和相关配置 |
| 部署问题 | 本地正常、线上异常 | 分支、工作流、构建和网站路径 |

例如，“`pnpm` 不是可识别的命令”首先是环境问题，不要一开始就修改 Firefly 的页面代码。

### 第四步：把问题拆成最小执行单位

一个最小执行单位应该只包含一个动作，并且做完后能立刻观察结果。推荐写成：

> **动作 + 对象 + 可观察结果**

下面是几组对比：

| 仍然太大 | 最小执行单位 |
| --- | --- |
| 安装开发环境 | 打开 PowerShell，执行 `node --version`，记录输出结果 |
| 学会 Git | 在 Firefly 根目录执行 `git status`，确认当前分支和文件状态 |
| 修改博客 | 打开 `src/config/`，找到站点配置文件，不进行修改 |
| 发布文章 | 在本地首页找到新文章标题，并打开文章详情页 |

如果一个步骤中连续出现多个“然后”，通常还可以继续拆分。

### 第五步：一次只让 AI 推进一步

第一次提问时，不必要求 AI 一次完成全部流程。可以使用下面的提示词：

```text
我的目标是：
我的当前进度是：
我的系统和软件版本是：
我已经完成的步骤是：
我现在看到的结果是：
我预期看到的结果是：

请先判断这属于哪一类问题，
然后只给我一个最小执行步骤，
并告诉我完成后应该看到什么。
```

执行后，把真实结果反馈给 AI，再决定下一步：

```text
AI 给出一个动作
→ 自己执行
→ 观察真实结果
→ 把结果反馈给 AI
→ 决定下一个动作
```

这样可以避免前面一步已经失败，后面却继续执行八个无效步骤。

### 第六步：保存问题卡片

每次开始新任务时，新建一张问题卡片：

```markdown
## 问题名称

- 目标：
- 现状：
- 环境与版本：
- 约束：
- 期望输出：
- 验收标准：
- 当前最小执行单位：
- 执行结果：
- 下一步：
```

问题解决后不要立即删除。它可以继续重构成文章、检查清单或自己的故障手册。

## 如何验证完成

完成本篇练习，不需要安装软件，也不需要写代码。只要完成下面这张检查表：

- [ ] 我写下了一个当前真正想解决的原始问题。
- [ ] 我明确写出了目标，而不是只写“学习”或“了解”。
- [ ] 我记录了当前进度，没有让 AI 猜测我做到哪里。
- [ ] 我写出了系统、软件版本或操作目录。
- [ ] 我写出了必须遵守的约束。
- [ ] 我说明了这一次希望 AI 输出什么。
- [ ] 我定义了一个能够观察或检查的完成标准。
- [ ] 我判断了问题属于目标、环境、操作、运行、显示或部署中的哪一类。
- [ ] 我把下一步改写成了“动作 + 对象 + 可观察结果”。
- [ ] 这个下一步可以在较短时间内独立完成。

最后，把整理后的问题交给一个不了解背景的人阅读。如果对方不需要追问“你用什么系统”“你做到哪一步”“什么才算完成”，问题就已经足够清晰。

你也可以直接把问题交给 AI，并要求它只回答下面三项：

```text
1. 复述我的目标。
2. 指出仍然缺少的信息。
3. 给出一个最小执行步骤和验收方法。
```

AI 如果能准确复述目标，并给出可以立即执行的单一步骤，就说明问题定义基本合格。

## 失败时先检查什么

如果仍然得不到有效回答，不要马上换一个 AI 或复制更多教程。按顺序检查：

1. **目标是不是动作？** “学习 Firefly”仍然很模糊，“在本地打开 Firefly 首页”才可以验证。
2. **有没有写当前进度？** AI 不知道你尚未安装工具，可能直接让你执行项目命令。
3. **环境是否具体？** 至少说明操作系统、软件版本、当前目录或页面地址。
4. **有没有提供原始信息？** 报错要完整复制，不要只说“失败了”。
5. **期望结果是否明确？** 同一个操作可能有多种正确结果，必须说明你想看到什么。
6. **步骤是不是仍然太大？** 如果十分钟内不知道从哪里动手，就继续拆分。
7. **一次是否混入多个问题？** 安装、运行、改配置和部署应当分别处理。
8. **是否把猜测写成了事实？** 不确定的版本、路径和状态应先检查，再继续操作。

最常见的失败不是“不会编程”，而是同时提出了太多问题。例如：

```text
帮我安装环境、运行 Firefly、修改网站、写文章并部署。
```

把它缩小成：

```text
我使用 Windows，现在还不知道是否安装了 Node.js。
请只告诉我如何检查 Node.js 版本，并说明什么结果代表可以继续。
```

问题一旦清晰，下一步通常也会变得清晰。
