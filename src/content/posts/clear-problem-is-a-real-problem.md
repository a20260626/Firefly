> “学 AI 编程”是一个方向，不是一个可以直接执行的问题。

很多人第一次使用 AI 编程时，会直接输入：“教我编程”“帮我做一个网站”或“这个项目为什么不能运行”。这些话表达了愿望，却没有提供足够的信息。AI 只能猜测你的目标、环境和限制，得到的回答自然可能很长，却不一定能解决眼前的问题。

真正有效的第一步不是马上写代码，而是把愿望改写成一个**有范围、有上下文、有完成标准的问题**。这篇文章只解决这一件事。

## 为什么问题越模糊，答案越容易失效

AI 不知道你的电脑上发生了什么。它只能依据你提供的文字推测环境，再从许多可能的方案中选择一个回答。问题中缺少的信息越多，AI 需要做的假设就越多。

例如，只说“Firefly 运行不了”，至少存在下面这些可能：

- Node.js 没有安装，或者版本不符合要求。
- `pnpm` 没有安装，终端无法识别命令。
- 终端没有位于 Firefly 项目根目录。
- 依赖尚未安装，或者安装过程曾经失败。
- 开发服务器已经启动，只是打开了错误的端口。
- 页面可以加载，但某项配置或资源出现错误。

如果没有更多信息，AI 可能一次给出十几种排查方法。回答看起来很全面，真正适合当前情况的可能只有一条。更麻烦的是，只要第一条假设错了，后面的操作就可能全部失去意义。

一个有效问题的作用，就是主动减少这些分支：

```text
模糊问题
→ AI 猜测环境和目标
→ 给出许多可能方案
→ 用户不知道该选哪一个

清晰问题
→ 排除不符合现状的分支
→ 锁定一个最小动作
→ 执行并观察结果
```

因此，描述问题不是在“浪费提问时间”，而是在减少试错成本。通常多花两分钟整理上下文，会比盲目执行一长串命令更快。

## 问题是什么

一个清晰的问题，至少要让自己和 AI 知道下面六件事：

| 元素 | 要回答的问题 | Firefly 示例 |
| --- | --- | --- |
| 目标 | 最终想得到什么结果？ | 在 Windows 电脑上运行 Firefly 博客 |
| 现状 | 现在已经做到哪里？ | 已克隆仓库，还没有安装依赖 |
| 环境 | 在什么系统、版本和目录中操作？ | Windows、Node.js 22、Firefly 根目录 |
| 约束 | 必须遵守哪些条件？ | 使用 pnpm，不更换项目技术栈 |
| 输出 | 这一次希望得到什么？ | 下一条需要执行的命令 |
| 验收 | 怎样证明这一步完成了？ | 浏览器能够打开本地首页 |

只有主题，没有这些元素时，它仍然是一个模糊目标。

例如：

```text
模糊目标：我要学 AI 编程。
```

可以先改写成：

```text
清晰问题：
我想使用 AI 辅助，在 Windows 电脑上把 Firefly 博客运行起来，
然后新增一篇 Markdown 文章，并部署到 GitHub Pages。
我目前已经有 GitHub 账号，但还没有准备开发环境。
请先告诉我第一个最小执行步骤，并说明如何验证它已经完成。
```

前一句只能得到宽泛的学习建议；后一句可以直接产生下一步行动。

清晰不等于写得很长。判断问题是否清晰，只需要看它能不能回答三个核心问题：

1. **现在在哪里？**
2. **下一步要到哪里？**
3. **怎样知道已经到达？**

### 必要信息和补充信息

提问时也不需要把所有资料都塞进去。可以把信息分成两层：

| 层级 | 内容 | 是否必须 |
| --- | --- | --- |
| 必要信息 | 目标、当前进度、环境、实际结果、期望结果 | 第一次提问就提供 |
| 补充信息 | 完整日志、配置片段、截图、最近改动、已尝试方案 | 与当前问题有关时再提供 |

例如，询问“下一步该安装什么”时，不必贴出整个项目源码；询问构建失败时，则应提供执行的命令和完整报错。信息不是越多越好，而是要与当前判断有关。

可以用一个简单标准筛选材料：

> 删除这条信息后，AI 是否可能对环境、原因或下一步产生不同判断？

如果答案是“会”，就保留；如果答案是“不会”，暂时不必提供。

## 适用环境是什么

这套方法不依赖某一种编程语言，也不只适用于 Firefly。下面这些情况都可以使用：

- 第一次向 ChatGPT 或其他 AI 编程助手提问。
- 安装 Node.js、pnpm、Git 等开发工具。
- 项目无法启动、构建失败或页面显示异常。
- 想修改配置，但不知道应该改哪个文件。
- 想新增功能，但需求还只是一个想法。
- 搜索了很多教程，却不知道哪一篇适合当前项目。
- 学习做饭、写作、设计等非编程技能，需要把大目标拆成行动。

尤其当你出现下面三种感觉时，应当先停下来重新定义问题：

- 不知道第一步做什么。
- AI 一次给出十几个步骤，不知道从哪里开始。
- 操作失败后，只能说“不能用”“没反应”或“报错了”。

本系列后续会以 Windows、Firefly、Node.js 和 GitHub Pages 为具体环境，但“先定义问题”的方法可以迁移到其他系统和项目。

## 具体怎么做

### 第一步：写下原始问题

先保留最自然的说法，不要急着修饰。

```text
我想学 AI 编程。
```

这句话的作用是确定方向。接下来要把方向变成任务。

### 第二步：填写问题六要素

复制下面的模板，把空白处补完整：

```text
我的目标是：
我的当前进度是：
我的操作环境是：
必须遵守的约束是：
我这次希望得到的输出是：
判断完成的标准是：
```

如果某一项暂时不知道，可以直接写“不知道”，并让 AI 帮助确认。不要用猜测冒充事实。

以运行 Firefly 为例：

```text
我的目标是：在本地运行 Firefly 博客。
我的当前进度是：已经用 GitHub Desktop 克隆仓库。
我的操作环境是：Windows 电脑，当前位于 Firefly 项目根目录。
必须遵守的约束是：使用项目指定的 pnpm，Node.js 版本不低于 22。
我这次希望得到的输出是：下一条需要执行的命令。
判断完成的标准是：终端显示本地地址，浏览器能打开博客首页。
```

### 第三步：给问题分类

分类不是为了增加术语，而是为了缩小搜索和检查范围。

| 类型 | 常见信号 | 首先关注 |
| --- | --- | --- |
| 目标问题 | “我想做……” | 结果与验收标准 |
| 环境问题 | 命令不存在、版本不符 | 系统、版本、安装路径 |
| 操作问题 | 不知道点哪里、改哪个文件 | 当前页面、目录和具体对象 |
| 运行问题 | 启动失败、终端报错 | 执行命令与完整报错 |
| 显示问题 | 页面空白、样式异常 | 页面地址、控制台和相关配置 |
| 部署问题 | 本地正常、线上异常 | 分支、工作流、构建和网站路径 |

例如，“`pnpm` 不是可识别的命令”首先是环境问题，不要一开始就修改 Firefly 的页面代码。

同一个大任务也可能依次经过多种问题类型。例如“把文章发布上线”可以经历：

```text
目标问题：我要发布什么内容？
→ 操作问题：文章应该放在哪个目录？
→ 运行问题：本地构建为什么失败？
→ 部署问题：为什么本地正常但线上打不开？
```

当前只处理最先阻止你继续前进的那一类。不要因为最终目标是部署，就在本地环境尚未运行时研究 GitHub Actions。

### 第四步：把问题拆成最小执行单位

一个最小执行单位应该只包含一个动作，并且做完后能立刻观察结果。推荐写成：

> **动作 + 对象 + 可观察结果**

下面是几组对比：

| 仍然太大 | 最小执行单位 |
| --- | --- |
| 安装开发环境 | 打开 PowerShell，执行 `node --version`，记录输出结果 |
| 学会 Git | 在 Firefly 根目录执行 `git status`，确认当前分支和文件状态 |
| 修改博客 | 打开 `src/config/`，找到站点配置文件，不进行修改 |
| 发布文章 | 在本地首页找到新文章标题，并打开文章详情页 |

如果一个步骤中连续出现多个“然后”，通常还可以继续拆分。

### 第五步：一次只让 AI 推进一步

第一次提问时，不必要求 AI 一次完成全部流程。可以使用下面的提示词：

```text
我的目标是：
我的当前进度是：
我的系统和软件版本是：
我已经完成的步骤是：
我现在看到的结果是：
我预期看到的结果是：

请先判断这属于哪一类问题，
然后只给我一个最小执行步骤，
并告诉我完成后应该看到什么。
```

执行后，把真实结果反馈给 AI，再决定下一步：

```text
AI 给出一个动作
→ 自己执行
→ 观察真实结果
→ 把结果反馈给 AI
→ 决定下一个动作
```

这样可以避免前面一步已经失败，后面却继续执行八个无效步骤。

### 第六步：保存问题卡片

每次开始新任务时，新建一张问题卡片：

```markdown
## 问题名称

- 目标：
- 现状：
- 环境与版本：
- 约束：
- 期望输出：
- 验收标准：
- 当前最小执行单位：
- 执行结果：
- 下一步：
```

问题解决后不要立即删除。它可以继续重构成文章、检查清单或自己的故障手册。

### 第七步：根据结果逐轮收窄问题

问题定义不是一次写完就永远不变。每执行一个动作，都会获得新的事实，问题也应该随之更新。

假设第一次检查得到：

```text
执行：node --version
结果：v22.17.0
```

这时“是否安装 Node.js”已经不再是问题。下一张问题卡片可以更新为：

```text
已确认：Node.js 版本为 v22.17.0，符合要求。
当前目标：确认 pnpm 是否可用。
下一步：执行 pnpm --version。
验收标准：终端输出版本号，而不是“无法识别命令”。
```

如果结果与预期不一致，也不要马上跳到新的教程。先原样记录：

```text
我执行了：pnpm --version
实际结果：终端提示 pnpm 不是可识别的命令。
预期结果：输出 pnpm 版本号。
请判断缺少的是安装、环境变量还是终端重启，并只给出一个检查步骤。
```

这种“执行—观察—更新”的循环，会不断把宽泛问题压缩成可以验证的具体问题。

## 三个完整改写示例

### 示例一：不知道如何开始

**原始问题：**

```text
教我用 AI 做博客。
```

这个问题没有说明博客类型、当前基础和本次输出。可以改成：

```text
我想在 Windows 上使用现有的 Firefly 项目搭建个人博客。
我有 GitHub 账号，但还没有检查开发工具。
我希望先确认电脑是否具备运行项目的条件，不要一次给出完整部署流程。
请给我第一个检查动作、需要执行的位置，以及成功时应看到的结果。
```

### 示例二：命令执行失败

**原始问题：**

```text
pnpm 不能用，怎么办？
```

“不能用”可能表示命令不存在、安装失败或项目构建失败。可以改成：

```text
系统：Windows 11
终端：PowerShell
当前目录：Firefly 项目根目录
Node.js 版本：v22.17.0
执行命令：pnpm install
实际结果：pnpm 不是可识别的命令
预期结果：开始安装项目依赖
已尝试：关闭并重新打开 PowerShell，结果不变

请先判断问题发生在 pnpm 安装、环境变量还是项目本身，
然后只给出一个最小检查步骤和判断标准。
```

### 示例三：页面修改没有显示

**原始问题：**

```text
我改了文章，但是页面没变化。
```

可以改成：

```text
我正在本地运行 Firefly，开发服务器没有报错。
我修改了 src/content/posts/ 中的一篇 Markdown 文章并保存。
浏览器中的文章详情页仍显示旧内容，手动刷新后也没有变化。
我的目标是确认修改的是不是当前页面对应的源文件。

请先告诉我如何核对“页面地址—文章 slug—源文件名”的对应关系，
不要先建议清理缓存或重新安装依赖。
```

这个版本明确限制了排查范围，可以避免一开始就做成本较高、且可能无关的操作。

## 怎样向 AI 提供报错和截图

遇到故障时，输入材料的质量会直接影响判断结果。

### 提供报错时

- 写出报错前执行的完整命令。
- 从第一条错误开始复制，不要只截取最后一句。
- 保留文件路径、行号、错误代码和依赖名称。
- 说明报错是每次出现，还是偶尔出现。
- 如果日志很长，先提供错误附近的完整段落，再询问是否需要更多内容。

不要只写：

```text
报了一堆红字。
```

应该写：

```text
我在 Firefly 根目录执行 pnpm build。
构建在 Astro 阶段停止，下面是从第一条 error 开始的完整输出：
（粘贴日志）
```

### 提供截图时

截图适合表达页面布局、按钮位置和视觉异常，但不适合替代可以复制的终端文字。截图中应包含足够上下文，例如浏览器地址、当前窗口标题或完整错误区域，同时注意遮挡账号、令牌、邮箱和本地隐私路径。

如果既有截图又有文本，优先把关键报错复制成文字，再用截图补充界面位置。

## 如何验证完成

完成本篇练习，不需要安装软件，也不需要写代码。只要完成下面这张检查表：

- [ ] 我写下了一个当前真正想解决的原始问题。
- [ ] 我明确写出了目标，而不是只写“学习”或“了解”。
- [ ] 我记录了当前进度，没有让 AI 猜测我做到哪里。
- [ ] 我写出了系统、软件版本或操作目录。
- [ ] 我写出了必须遵守的约束。
- [ ] 我说明了这一次希望 AI 输出什么。
- [ ] 我定义了一个能够观察或检查的完成标准。
- [ ] 我判断了问题属于目标、环境、操作、运行、显示或部署中的哪一类。
- [ ] 我把下一步改写成了“动作 + 对象 + 可观察结果”。
- [ ] 这个下一步可以在较短时间内独立完成。

最后，把整理后的问题交给一个不了解背景的人阅读。如果对方不需要追问“你用什么系统”“你做到哪一步”“什么才算完成”，问题就已经足够清晰。

你也可以直接把问题交给 AI，并要求它只回答下面三项：

```text
1. 复述我的目标。
2. 指出仍然缺少的信息。
3. 给出一个最小执行步骤和验收方法。
```

AI 如果能准确复述目标，并给出可以立即执行的单一步骤，就说明问题定义基本合格。

## 失败时先检查什么

如果仍然得不到有效回答，不要马上换一个 AI 或复制更多教程。按顺序检查：

1. **目标是不是动作？** “学习 Firefly”仍然很模糊，“在本地打开 Firefly 首页”才可以验证。
2. **有没有写当前进度？** AI 不知道你尚未安装工具，可能直接让你执行项目命令。
3. **环境是否具体？** 至少说明操作系统、软件版本、当前目录或页面地址。
4. **有没有提供原始信息？** 报错要完整复制，不要只说“失败了”。
5. **期望结果是否明确？** 同一个操作可能有多种正确结果，必须说明你想看到什么。
6. **步骤是不是仍然太大？** 如果十分钟内不知道从哪里动手，就继续拆分。
7. **一次是否混入多个问题？** 安装、运行、改配置和部署应当分别处理。
8. **是否把猜测写成了事实？** 不确定的版本、路径和状态应先检查，再继续操作。

最常见的失败不是“不会编程”，而是同时提出了太多问题。例如：

```text
帮我安装环境、运行 Firefly、修改网站、写文章并部署。
```

把它缩小成：

```text
我使用 Windows，现在还不知道是否安装了 Node.js。
请只告诉我如何检查 Node.js 版本，并说明什么结果代表可以继续。
```

问题一旦清晰，下一步通常也会变得清晰。

## 从“会提问”到“会解决问题”

清晰提问不是为了写出某种固定格式，更不是为了学习“神奇提示词”。模板只是一副脚手架，真正要建立的是下面四个习惯：

1. **区分事实和猜测。** “终端输出 v22.17.0”是事实，“应该不是版本问题”是猜测。
2. **区分现象和原因。** “页面空白”是现象，“组件坏了”只是尚未验证的原因。
3. **一次验证一个变量。** 同时升级依赖、修改配置和清理缓存，会让你无法判断究竟什么起了作用。
4. **为每一步定义反馈。** 命令输出、页面变化、文件生成和检查通过，都是下一步判断的依据。

长期使用问题卡片后，你会发现 AI 的作用不再只是“给答案”，而是与你共同完成一个稳定闭环：

```text
定义问题 → 提出假设 → 执行最小动作 → 观察事实
    ↑                                  ↓
    └──────── 更新问题 ← 记录结果 ────────┘
```

当这个闭环能够持续运转时，即使 AI 第一次判断不完全正确，你也能用新的事实及时纠正方向。这才是 AI 编程中最值得迁移到其他领域的能力。
