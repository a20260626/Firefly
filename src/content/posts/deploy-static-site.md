> 这一篇，我们把部署平台、构建命令、输出目录和上线后检查串成一条完整流程。

---

上一篇我们做的是远程仓库准备。

也就是把本地项目整理好，提交到 Git，并推送到 GitHub。

现在项目已经不只存在于你的电脑里。
部署平台可以读取它了。

于是发布篇进入下一步：

> **选择一个部署平台，把这个静态网站真正发布出去。**

这里的重点不是记住某个平台每一个按钮在哪里。

平台界面会变。
但部署的核心问题很稳定：

1. 代码从哪里来
2. 用什么命令构建
3. 构建产物在哪里
4. 生成的网站能不能正常访问
5. 后续更新怎么触发重新发布

把这五件事想清楚，换平台也不会慌。

---

## 先判断本站属于什么类型

部署前先确认项目类型。

本站是一个：

> **Vite + React + TypeScript 的静态内容站。**

它的特点是：

- 不需要自己写后端服务
- 不需要数据库
- 文章内容在构建时被打包进前端代码
- 浏览器打开后由 JavaScript 渲染页面
- 适合部署到静态站托管平台

所以它适合这些平台：

| 平台 | 适合原因 |
|------|----------|
| GitHub Pages | 和 GitHub 仓库关系紧，适合免费展示项目 |
| Vercel | 前端项目部署体验顺，自动构建方便 |
| Netlify | 静态站托管成熟，配置也比较直观 |

第一次发布，不用追求“最佳平台”。

更重要的是先跑通一次完整流程。

---

## 部署平台需要你告诉它什么

大多数部署平台都会问几个类似问题。

即使界面不同，底层含义也差不多。

### 1. 代码仓库

平台需要知道代码在哪里。

通常选择：

```text
GitHub repository
```

也就是上一篇准备好的远程仓库。

### 2. 项目框架

平台可能会自动识别项目是 Vite。

如果需要手动选择，可以选择：

```text
Vite
```

或者选择一个通用的静态前端项目配置。

### 3. 构建命令

本站的构建命令是：

```bash
npm run build
```

这个命令会把 `src`、Markdown、CSS、图片等内容打包成可发布文件。

### 4. 输出目录

Vite 默认输出目录是：

```text
dist
```

所以部署平台要发布的目录通常就是 `dist`。

这四个配置可以整理成一张表：

| 配置项 | 本站填写 |
|------|----------|
| Repository | 你的 GitHub 仓库 |
| Framework | Vite |
| Build Command | `npm run build` |
| Output Directory | `dist` |

这张表就是第一次部署时最重要的“翻译表”。

---

## 不要把部署理解成上传文件夹

新手很容易以为部署是：

> 把整个项目文件夹传上去。

但对于这种现代前端项目，更常见的流程是：

1. 部署平台拉取 GitHub 仓库
2. 平台安装依赖
3. 平台运行 `npm run build`
4. 平台得到 `dist`
5. 平台把 `dist` 发布成网站

也就是说：

> 你提交的是源码，平台发布的是构建产物。

这就是为什么前面反复强调本地构建。

如果本地 `npm run build` 都无法通过，部署平台大概率也会失败。

---

## GitHub Pages、Vercel、Netlify 该怎么选

如果只是学习和展示，三者都可以。

可以这样粗略选择：

| 你的情况 | 可以优先试 |
|----------|------------|
| 想和 GitHub 项目页放在一起 | GitHub Pages |
| 想要最顺手的前端部署体验 | Vercel |
| 想做静态站并保留后续表单、重定向等扩展空间 | Netlify |

如果你还没有偏好，我会建议：

> 第一版先选一个你最容易登录和配置的平台。

因为这时目标不是研究平台生态。

目标是完成第一次上线：

1. 看到构建成功
2. 拿到线上链接
3. 打开首页
4. 打开文章页
5. 确认资源和导航都正常

跑通之后，再比较平台差异会更有意义。

## 第一次部署前的本地检查

在打开部署平台之前，先在本地做一次检查。

```bash
npm run build
```

然后确认：

1. 构建命令通过
2. `dist` 文件夹生成
3. 首页能打开
4. 至少一篇文章页能打开
5. 图片资源显示正常
6. 上一篇 / 下一篇按钮能跳转
7. Learning Stages 和 Tool Index 能跳转

还可以检查 Git 状态：

```bash
git status --short
```

如果还有未提交的重要改动，先提交。

部署平台读取的是 GitHub 上的代码。
如果本地改了但没有推送，平台不会知道。

---

## 第一次部署的大致流程

不同平台界面会变，但流程通常可以抽象成这样：

1. 登录部署平台
2. 选择从 GitHub 导入项目
3. 授权平台读取对应仓库
4. 选择本站仓库
5. 确认框架为 Vite
6. 填写构建命令 `npm run build`
7. 填写输出目录 `dist`
8. 点击部署
9. 等待平台安装依赖和构建
10. 获取线上访问链接

如果一切顺利，你会拿到类似这样的链接：

```text
https://your-site.example.app
```

这就是别人可以打开的网站地址。

从这一刻开始，项目就不只是本地练习了。

---

## 上线后不要只看首页

部署成功后，很多人会打开首页看一眼，然后就结束。

这还不够。

本站至少要检查这些页面和模块：

1. 首页学习档案是否显示
2. 首页文章列表是否显示
3. 首页文章标题是否能点击
4. `继续阅读` 是否能打开文章页
5. 文章页目录是否正常
6. 最近文章是否正常
7. 上一篇 / 下一篇是否正常
8. Learning Stages 是否能跳转到各文章
9. Tool Index 是否能跳转到各文章
10. 页面在手机宽度下是否没有明显溢出

为什么要查这么多？

因为部署成功只说明“文件发布出去了”。

它不等于“网站体验完全正常”。

---

## 特别检查文章链接

本站文章页使用 query 参数：

```text
/?article=prepare-github-repository
```

这种方式对静态部署比较友好。

因为所有访问都先进入 `index.html`，再由前端根据 `article` 参数决定显示哪篇文章。

上线后要测试几个真实链接：

```text
/?article=ai-programming-capabilities
/?article=setup-project-foundation
/?article=deploy-static-site
```

如果这些链接都能打开，说明文章路由基本正常。

如果首页能打开，但文章页打不开，要优先检查：

- URL 是否写错
- slug 是否已在 `articles.ts` 注册
- Markdown frontmatter 是否有效
- 部署平台是否正确发布了最新代码

## 如果部署失败，先看构建日志

部署失败时，不要直接猜。

部署平台通常会给构建日志。

你要找的是第一条真正导致失败的报错。

常见原因包括：

| 问题 | 可能原因 |
|------|----------|
| 依赖安装失败 | `package-lock.json` 或 Node 版本问题 |
| TypeScript 报错 | 本地没跑构建，线上构建更严格 |
| 找不到文件 | 导入路径写错，或文件名大小写不一致 |
| 找不到 Markdown | 文章文件没提交或 slug 没注册 |
| 图片不显示 | 资源路径或部署 base path 有问题 |

这个时候可以把构建日志复制给 AI，让它帮你读。

但要记得给上下文：

```text
这是部署平台的构建日志。
项目是 Vite + React + TypeScript 静态内容站。
本地构建命令是 npm run build，输出目录是 dist。
请帮我找第一条真正导致失败的错误，并告诉我应该先检查哪个文件。
```

不要只说“部署失败了怎么办”。

给日志，AI 才能帮你定位。

---

## 更新网站时会发生什么

第一次上线后，后续更新会简单很多。

一般流程是：

1. 本地修改文章或代码
2. 本地运行 `npm run build`
3. 提交 Git
4. 推送到 GitHub
5. 部署平台自动重新构建
6. 线上网站更新

也就是说，发布之后你会进入一个循环：

> 修改 → 构建 → 提交 → 推送 → 自动部署 → 检查线上效果

这就是持续迭代。

当你理解这个循环，网站就不再是“一次性作品”。

它会变成一个可以长期更新的项目。

## 让 AI 帮你生成部署检查清单

你可以这样问 AI：

```text
我要部署一个 Vite + React + TypeScript 静态内容站。
代码已经推送到 GitHub。

请帮我生成部署检查清单，包含：
1. 部署平台需要填写的配置
2. 本地部署前要执行的命令
3. 上线后要检查的 URL
4. 如果构建失败，应该优先看哪些日志
5. 如果文章页打不开，应该检查哪些文件

请用新手能看懂的方式解释，不要直接假设我熟悉部署平台。
```

这类提示词的重点是：

- 说明项目技术栈
- 说明代码已经在 GitHub
- 说明你要的是检查清单
- 要求 AI 不要跳过基础解释

AI 最适合帮你做“流程外化”。

把脑子里的模糊担心，变成一张可以逐项勾选的清单。

这篇文章完成的是发布篇第三步：选择部署平台并上线静态网站。

你需要记住：

1. 本站是 Vite + React + TypeScript 静态内容站
2. 部署平台通常从 GitHub 仓库读取代码
3. 构建命令是 `npm run build`
4. 输出目录是 `dist`

第一次部署的意义很大。

它让项目从“我电脑里的练习”变成“别人也能打开的作品”。
