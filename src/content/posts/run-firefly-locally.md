完成 [Fork 和克隆 Firefly](/posts/fork-firefly-and-clone/) 后，代码已经在自己的电脑上了，但“代码存在”不等于“项目可以运行”。本篇把目标缩小为一条可以重复执行的闭环：进入正确目录、安装依赖、启动开发服务器、打开本地页面，并确认修改后页面能够更新。

## 先理解本地运行发生了什么

执行 `pnpm dev` 并不是“打开一个网页文件”。Firefly 是 Astro 项目，本地运行会经过下面的过程：

```text
读取 package.json 中的脚本
→ 加载 node_modules 中的依赖
→ Astro 读取配置、组件和文章
→ 启动本地 HTTP 服务
→ 浏览器通过端口访问页面
→ 文件保存后重新编译并更新页面
```

其中任何一层失败，都会产生不同现象：

| 层级 | 常见现象 |
| --- | --- |
| 工具环境 | `node` 或 `pnpm` 无法识别、版本不符合要求 |
| 项目目录 | 找不到 `package.json`、找不到脚本 |
| 依赖安装 | 下载失败、权限错误、依赖目录不完整 |
| 内容与配置 | Astro 报语法、类型或内容格式错误 |
| 开发服务器 | 端口占用、进程退出、地址无法访问 |
| 浏览器 | 打开了错误地址、缓存旧页面、控制台报错 |

先判断故障发生在哪一层，再处理对应证据，比反复重装整个项目更有效。

### `install`、`dev`、`build` 和 `preview` 的区别

Firefly 常见命令服务于不同阶段：

| 命令 | 作用 | 什么时候使用 |
| --- | --- | --- |
| `pnpm install` | 安装项目声明的依赖 | 首次克隆、依赖清单变化后 |
| `pnpm dev` | 启动支持热更新的开发服务器 | 日常修改和即时预览 |
| `pnpm build` | 生成用于正式部署的静态产物 | 提交或部署前检查 |
| `pnpm preview` | 本地预览正式构建产物 | 验证构建后的效果 |

本篇主要完成前两项。开发页面能够打开，不代表正式构建一定成功；后续提交前仍需单独执行完整检查。

## 问题是什么

“把 Firefly 跑起来”还不够具体。可以把它改写成下面这个可执行问题：

> 我已经把自己的 Firefly 仓库克隆到 Windows 本地。现在请在项目根目录安装依赖，启动开发服务器，打开终端输出的本地地址，并验证首页能够显示、Markdown 修改能够触发更新。

这一步的输入、过程和输出如下：

| 要素 | 本篇内容 |
| --- | --- |
| 输入 | 已经克隆到本地的 Firefly 项目目录 |
| 环境 | Node.js 22 或更高版本、pnpm 9 或更高版本 |
| 执行 | `pnpm install` 和 `pnpm dev` |
| 输出 | 一个可以在浏览器访问的本地 Firefly 网站 |
| 验收标准 | 首页能打开，终端没有启动错误，修改内容后页面能更新 |

本地运行只服务于当前电脑。它和后面的 GitHub Pages 部署是两件事：本地运行用于编辑和预览，部署用于让其他人通过互联网访问。

## 适用环境是什么

本文适用于下面的环境：

- Windows 10 或 Windows 11。
- 已经完成 GitHub Fork，并把自己的 Firefly 仓库克隆到本地。
- 已经安装 Node.js LTS，版本不低于 22。
- 已经安装 pnpm，版本不低于 9。
- 已经安装 VS Code 或其他可以编辑 Markdown 的编辑器。
- 当前项目使用 Astro，开发命令由项目根目录的 `package.json` 提供。

开始前，建议先完成上一篇：[Fork 和克隆 Firefly](/posts/fork-firefly-and-clone/)。如果项目还没有出现在电脑上，不要直接执行 `pnpm install`，先把仓库克隆完成。

## 启动前检查清单

在执行安装命令之前，先确认：

- [ ] 打开的是自己的 Firefly 本地目录。
- [ ] `git status` 能在当前目录正常执行。
- [ ] 当前分支与准备使用的分支一致。
- [ ] `package.json` 和 `pnpm-lock.yaml` 都存在。
- [ ] Node.js 和 pnpm 版本符合要求。
- [ ] 网络能够访问依赖下载服务。
- [ ] 磁盘有足够空间存放 `node_modules` 和构建缓存。
- [ ] 没有另一个安装或开发进程正在操作同一项目。

首次运行时建议保留一个学习记录，写下本地目录、工具版本、执行时间和最终访问地址。以后出现环境差异时，可以直接比较。

## 具体怎么做

### 1. 打开 Firefly 项目根目录

在文件管理器中打开克隆后的 `Firefly` 文件夹，在文件夹空白处右键打开 PowerShell。也可以直接在 PowerShell 中进入目录：

```powershell
cd E:\Documents\GitHub\Firefly
```

路径要替换成自己的实际目录。进入后先确认当前目录确实是项目根目录：

```powershell
Get-Location
Get-ChildItem package.json, pnpm-lock.yaml
```

如果能够看到 `package.json` 和 `pnpm-lock.yaml`，说明终端位置正确。后面的命令都应该在这里执行。

可以再用 Git 确认项目根目录：

```powershell title="确认 Git 项目根目录"
git rev-parse --show-toplevel
git status --short --branch
```

第一条应输出当前 Firefly 根目录，第二条应显示当前分支和文件状态。如果终端提示“不是 Git 仓库”，说明当前位置不是克隆后的项目，先修正目录，不要继续安装依赖。

如果使用 VS Code，推荐通过 **File → Open Folder** 打开 Firefly 根目录，再从 **Terminal → New Terminal** 创建终端。不要只打开单个 Markdown 文件，否则集成终端可能位于其他目录。

### 2. 检查基础工具版本

先不要急着安装依赖，确认 Node.js、pnpm 和 Git 可以被终端识别：

```powershell
node --version
pnpm --version
git --version
```

版本至少应该满足：

```text
Node.js >= 22
pnpm >= 9
```

如果命令输出版本号，说明工具已经进入当前终端的 PATH。版本过低时，先完成 [安装 AI 编程工具](/posts/install-required-environment/)，再继续本篇。

可以把结果记录为：

```text
项目目录：
当前分支：
Node.js 版本：
pnpm 版本：
Git 版本：
检查时间：
```

版本应使用实际输出，不要根据记忆填写。“之前安装过”不能证明当前终端调用的是正确版本。

### 3. 安装项目依赖

在项目根目录执行：

```powershell
pnpm install
```

这个命令会根据 `package.json` 和 `pnpm-lock.yaml` 下载项目依赖，并生成或更新本地依赖目录。首次安装可能需要一些时间，期间不要关闭终端。

安装时可以观察几个阶段：解析依赖、下载缺少的包、把依赖链接到项目、运行必要脚本。只要终端仍有进度变化，就应等待命令结束。不要在安装尚未完成时同时执行 `pnpm dev`。

`pnpm-lock.yaml` 锁定了经过验证的依赖版本组合。正常首次安装不应随意删除它，也不应把命令替换成 `npm install`。如果安装后锁文件发生变化，先查看 `git diff -- pnpm-lock.yaml`，确认是否由包管理器版本或依赖更新造成。

安装结束后，可以检查项目目录中是否出现 `node_modules`：

```powershell
Test-Path node_modules
```

输出 `True` 只代表依赖目录存在，最终是否安装成功仍然要以 `pnpm dev` 能否启动为准。

还应检查安装命令的结束状态：如果最后出现明确的错误，即使 `node_modules` 已经生成，也不能视为安装成功。依赖目录可能只完成了一部分。

### 4. 启动 Astro 开发服务器

继续在同一个项目根目录执行：

```powershell
pnpm dev
```

终端会显示类似下面的地址：

```text
Local   http://localhost:4321/
```

以终端实际显示的地址为准。如果 `4321` 端口已经被其他程序占用，Astro 可能会自动使用另一个端口，例如 `4322`。不要猜端口，直接复制终端输出的 `Local` 地址。

### 理解 Local 和 Network 地址

终端有时会显示不止一个地址：

- **Local** 通常只用于当前电脑访问，例如 `http://localhost:4321/`。
- **Network** 用于同一局域网内的其他设备访问，通常需要额外启动参数和防火墙允许。

本篇只要求 Local 地址正常。不要为了用手机访问而立即关闭防火墙或开放端口；局域网测试应作为独立任务，先理解网络范围和安全影响。

`localhost`、`127.0.0.1` 都代表当前电脑，不是公开网址。把本地地址发给其他人，他们无法从自己的网络打开。

### 5. 在浏览器中打开本地网站

把终端显示的地址粘贴到浏览器中。看到 Firefly 首页后，先检查：

- 页面不是空白页。
- 浏览器地址栏使用的是本地地址，而不是 GitHub Pages 地址。
- 终端没有持续出现红色错误。
- 首页标题、文章列表和导航能够正常显示。

开发服务器运行期间，终端窗口需要保持打开。按 `Ctrl+C` 会停止本地服务。

浏览器请求页面时，终端可能输出新的访问或构建信息，这是正常现象。真正需要关注的是带有 `error`、构建失败、无法解析文件等明确提示，以及开发服务器是否意外回到命令提示符。

第一次打开后可以测试三个页面：首页、任意文章详情页、一个归档或标签页面。这样能确认不只是单个路由偶然可用。

### 6. 用一处小修改验证热更新

不要一开始就修改大量文件。可以用 VS Code 打开一篇自己的 Markdown 文章，在正文末尾临时增加一句文字并保存，然后回到浏览器：

```markdown
> 本地开发服务器已经可以正常更新页面。
```

如果浏览器自动刷新后能看到这句话，说明“编辑文件 → 开发服务器重新构建 → 浏览器显示结果”的最小闭环已经完成。验证结束后，可以删除这句临时文字，避免把测试内容带入正式文章。

修改前先执行 `git status`，修改并保存后再执行：

```powershell
git diff --stat
git diff
```

确认只有预期文章产生变更。删除临时文字后再次检查，确保测试内容已经完全撤销，不要把无关验证文字混入后续提交。

热更新通常很快，但内容数量较多时可能需要短暂重新构建。保存后先看终端是否编译成功，再看浏览器。如果终端已有语法错误，反复刷新浏览器不会解决问题。

### 7. 停止和重新启动服务

完成验证后，在运行 `pnpm dev` 的终端按：

```text
Ctrl+C
```

需要继续工作时，再回到项目根目录执行：

```powershell
pnpm dev
```

能够重复启动，说明本地运行步骤已经可以被下次直接复用。

### 开发服务器的生命周期

可以把一次本地开发会话理解为：

```text
进入项目目录 → pnpm dev → 保持终端运行
→ 编辑和预览 → Ctrl+C → 确认服务停止
```

服务停止后再访问原地址，浏览器应提示无法连接。这是正常结果，说明端口已经没有开发服务器监听。重新执行 `pnpm dev` 后页面才会恢复。

如果关闭终端后页面仍能打开，可能还有另一个开发进程正在运行，或者浏览器只显示了缓存页面。刷新并观察终端与页面响应，再确认实际进程状态。

## 如何验证完成

依次完成下面的检查：

- [ ] 当前终端目录包含 `package.json` 和 `pnpm-lock.yaml`。
- [ ] `node --version` 显示 Node.js 22 或更高版本。
- [ ] `pnpm --version` 显示 pnpm 9 或更高版本。
- [ ] `git --version` 能正常输出版本号。
- [ ] `pnpm install` 执行完成且没有未处理的错误。
- [ ] 项目目录中存在 `node_modules`。
- [ ] `pnpm dev` 能启动 Astro 开发服务器。
- [ ] 浏览器能打开终端输出的 `Local` 地址。
- [ ] Firefly 首页能够显示，不是空白页。
- [ ] 浏览器地址是本地地址，而不是线上地址。
- [ ] 修改并保存 Markdown 后，浏览器能够看到更新结果。
- [ ] 按 `Ctrl+C` 后服务能够停止，再次执行 `pnpm dev` 能重新启动。

当这些项目都完成时，就可以认为 Firefly 已经在本地运行起来了。下一步可以继续：[认识 Firefly 文件结构并修改配置](/posts/setup-project-foundation/)。

### 保存一份运行记录

```text
项目根目录：
当前分支：
Node.js 版本：
pnpm 版本：
pnpm install：成功 / 失败
开发服务器地址：
首页：正常 / 异常
文章页：正常 / 异常
热更新：正常 / 异常
停止并重启：正常 / 异常
最近一次验证时间：
```

地址要记录终端实际输出，不要默认写成 `4321`。如果下次端口改变，只要页面仍能通过新地址访问，就不代表项目配置出现问题。

## 失败时先检查什么

### `pnpm` 不是内部或外部命令

先执行：

```powershell
node --version
npm --version
```

如果 Node.js 也无法识别，说明 Node.js 没有正确安装或终端还没有刷新 PATH。重新打开 PowerShell，再按照 [安装 AI 编程工具](/posts/install-required-environment/) 检查安装步骤。

如果外部 PowerShell 能识别 pnpm，而 VS Code 集成终端不能，完全退出并重新打开 VS Code，让它读取新的 PATH。不要因此修改 Firefly 源码。

### `pnpm install` 失败

先确认三件事：

1. 当前目录确实包含 `package.json`。
2. Node.js 版本满足项目要求。
3. 终端能够访问 npm 镜像或其他依赖下载地址。

不要只复制“安装失败”四个字给 AI。应当把完整错误、执行的命令、Node.js 版本和当前目录一起提供，让 AI 判断是目录、版本、权限还是网络问题。

可以先根据错误关键词缩小范围：

| 现象或关键词 | 优先检查 |
| --- | --- |
| 找不到 `package.json` | 当前目录 |
| `Unsupported engine` | Node.js 或 pnpm 版本 |
| `ETIMEDOUT`、连接重置 | 网络、代理、依赖源 |
| `EACCES`、`EPERM` | 权限、文件占用、安全软件 |
| 校验或锁文件相关错误 | 锁文件、包管理器版本、缓存完整性 |
| 安装脚本报错 | 最早失败的脚本和它前面的日志 |

不要一看到安装错误就删除 `node_modules`、锁文件和所有缓存。删除会丢失现场信息，也可能把原本明确的问题变成另一个问题。先保存日志并确认错误类别。

### 安装命令长时间没有结束

先观察终端是否仍有下载进度、CPU 或网络活动，以及是否在等待输入。不要连续按 `Ctrl+C` 或重复启动多个安装进程。

如果确认长时间完全没有变化，可以按一次 `Ctrl+C` 停止，并记录停止前最后一段输出。下一次排查需要知道它停在解析、下载、链接还是安装脚本阶段。

### `pnpm dev` 找不到命令或启动失败

先重新执行：

```powershell
Get-Location
Get-ChildItem package.json
pnpm install
pnpm dev
```

如果错误提示端口被占用，使用终端新输出的地址；如果错误来自某个配置文件或 Markdown 文件，保留完整报错，再只检查报错指向的文件。

还要区分两种“找不到命令”：

- 系统找不到 `pnpm`：属于工具安装或 PATH 问题。
- pnpm 找不到 `dev` 脚本：可能位于错误目录，或当前 `package.json` 不是 Firefly 的项目清单。

可以执行：

```powershell
Get-Content package.json | Select-String '"dev"'
```

确认当前项目确实声明了开发脚本。不要为了消除报错，在不理解项目配置时手动添加一个猜测的 `dev` 命令。

### 端口被占用

Astro 通常会选择新的可用端口。最简单的处理是使用终端显示的新 Local 地址。如果必须确认某个端口由哪个进程使用，可以在 PowerShell 中执行：

```powershell
Get-NetTCPConnection -LocalPort 4321 -ErrorAction SilentlyContinue
```

先识别进程，再决定是否停止。不要为了释放端口批量结束所有 Node.js 进程，因为其中可能包含其他正在工作的项目。

### 终端启动后立即退出

正常的开发服务器会持续运行。如果命令很快回到 `PS ...>` 提示符，说明进程已经结束。向上查找第一条错误，而不是只复制最后的退出代码。

重新启动前记录：执行命令、当前目录、Node.js 和 pnpm 版本、第一条错误、最近修改文件。只有修正错误后才重复执行，避免用多次重试覆盖重要日志。

### 浏览器打开后是空白页

先打开浏览器开发者工具的 Console，查看是否有红色错误。然后确认终端是否出现构建错误，以及浏览器打开的是否是当前 `pnpm dev` 输出的地址。把错误信息、最近修改的文件和复现步骤一起交给 AI，不要只描述“页面白了”。

按下面的顺序收集证据：

1. 地址栏中的完整本地 URL。
2. 运行开发服务器的终端是否仍然活跃。
3. 终端最近的错误输出。
4. 浏览器 Console 的第一条红色错误。
5. 最近一次成功显示后修改了哪些文件。

如果首页空白但其他页面正常，问题可能只影响首页路由或相关组件；如果所有页面都无法连接，则优先检查服务器和地址。

### 修改文件后浏览器没有更新

先确认文件确实保存在当前项目目录，浏览器打开的地址确实来自当前开发服务器。可以手动刷新一次；如果仍然没有变化，再查看终端是否出现编译错误。修改能够生效后，再进入下一篇的配置和内容工作。

还可以检查：

- VS Code 标签页是否显示未保存圆点。
- 修改的是不是当前 URL 对应的文章文件。
- 电脑上是否有多个 Firefly 副本，编辑器和终端是否打开了不同目录。
- 文件保存后终端是否出现重新编译信息。
- 页面是否被浏览器缓存，强制刷新后是否变化。

### 浏览器提示无法访问此网站

先看运行 `pnpm dev` 的终端。如果已经按过 `Ctrl+C`、关闭终端或进程因错误退出，本地地址自然无法访问。重新启动服务，并使用新输出的地址。

如果服务仍在运行，确认协议、主机和端口都正确：`http://localhost:4321/` 与 `https://localhost:4321/` 不是同一个地址。开发服务器通常使用 `http`。

### 修改后出现 Markdown 或内容错误

先撤回最近一次最小修改，确认页面是否恢复。常见原因包括 frontmatter 引号未闭合、日期格式错误、代码围栏缺少结尾、链接或组件语法不完整。

不要在报错状态下继续修改更多文件。一次只修复终端最先指出的文件和行，保存后观察错误是否消失。

## 下一次怎样快速启动

首次完整验证后，日常启动可以缩短为：

```powershell
cd 你的\Firefly\项目路径
git status --short --branch
pnpm dev
```

不需要每次都重新执行 `pnpm install`。只有首次克隆、切换到包含依赖变化的提交，或者 `package.json`、`pnpm-lock.yaml` 明确发生变化时，才需要重新安装依赖。

开始编辑前先看 Git 状态，结束后按 `Ctrl+C` 停止服务，再查看 `git diff`，就能形成稳定的日常开发习惯。
