完成 [认识 Firefly 文件结构并修改配置](/posts/setup-project-foundation/) 后，已经知道文章放在 `src/content/posts/`。本篇把“写一篇文章”拆成一条可以重复执行的流程：确定文件名、生成 Markdown 文件、填写元数据、写正文、在本地查看，并确认构建没有问题。

## 写文章也是一个可验证的开发任务

在 Firefly 中，文章不是粘贴到后台编辑器里的数据库记录，而是仓库中的源文件。它会经过内容解析、类型检查、页面生成、搜索索引和正式构建。因此一篇文章同时包含三种质量：

| 质量 | 关注内容 |
| --- | --- |
| 内容质量 | 主题是否清楚、结构是否连贯、示例是否准确 |
| 数据质量 | frontmatter 字段、日期、标签和分类是否符合规则 |
| 工程质量 | 路径、图片、链接、构建和 Git 差异是否正确 |

只看编辑器里“写完了”还不够。真正完成要形成闭环：

```text
定义主题 → 规划结构 → 创建文件 → 填写元数据
→ 写最小正文 → 本地预览 → 补充内容
→ 检查链接和资源 → check/build → 查看 Git 差异
```

### AI 在写作流程中适合做什么

AI 可以帮助：

- 把模糊主题改写成明确问题；
- 根据目标读者生成文章提纲；
- 检查段落是否重复、跳跃或缺少前提；
- 把真实操作记录整理成清晰步骤；
- 根据报错定位 Markdown 或 frontmatter 问题；
- 按检查清单审阅标题、摘要、链接和示例。

AI 不应该替你虚构实际测试结果、版本信息、命令输出和个人经历。涉及项目路径、配置字段和命令时，应以当前仓库与实际执行结果为准。

## 问题是什么

“我要写一篇文章”还不够具体。本篇要解决的问题是：**如何在 Firefly 中新增一篇可访问、可构建、可继续修改的 Markdown 文章。**

完成后应该得到：

- 一个放在 `src/content/posts/` 下的 `.md` 文件；
- 一组符合 Firefly 内容规则的 frontmatter；
- 一篇能够在本地文章列表和文章详情页中显示的正文；
- 一条可以检查、构建和提交的 Git 改动。

本篇不处理文章配图设计、不修改页面布局，也不同时新增多个文件。先让一篇最小文章完整跑通，再逐步增加图片、代码和图表。

## 适用环境是什么

本文适用于已经完成以下准备的环境：

- 已经 Fork、克隆并进入 Firefly 项目根目录；
- 已经执行过 `pnpm install`；
- Node.js 版本不低于 22，pnpm 可以正常运行；
- 已经用 VS Code 打开项目；
- 本地开发服务器可以通过 `pnpm dev` 启动。

文章内容由 Astro Content Collection 读取，集合配置位于 `src/content.config.ts`。当前文章文件使用 `.md` 或 `.mdx` 扩展名，放错目录、frontmatter 缺字段或类型不正确，都可能导致检查失败。

## 创建文件前先规划文章

先回答六个问题：

```text
主题：这篇文章只解决什么问题？
读者：谁会阅读，已经具备什么前提？
输入：读者开始前需要有什么？
过程：需要执行哪些阶段？
输出：完成后能得到什么？
验收：读者怎样证明自己做对了？
```

例如：

```text
主题：在 Firefly 中新增第一篇 Markdown 文章。
读者：已经能在本地运行 Firefly 的初学者。
输入：项目目录、VS Code、开发服务器。
过程：确定 slug、创建文件、填写 frontmatter、写正文、预览。
输出：一个可以通过 URL 打开的文章页面。
验收：首页可见、详情页可访问、check 和 build 通过。
```

然后用一个简单提纲限制范围：

```text
为什么要做
需要什么环境
具体怎么做
怎样验证完成
失败时先检查什么
```

先有边界再让 AI 扩写，可以减少重复和与主题无关的内容。

## 具体怎么做

### 1. 先确定文章主题和 slug

先把文章主题写成一个明确的问题，例如：

> 如何在 Firefly 中新增一篇 Markdown 文章，并确认它能在本地页面访问？

再为文章确定文件名。文件名会参与生成文章网址，建议只使用小写英文字母、数字和短横线：

```text
my-first-firefly-article.md
```

这个文件最终对应：

```text
http://127.0.0.1:24321/posts/my-first-firefly-article/
```

文件名尽量稳定。文章发布后再改 slug，会影响旧链接和搜索结果。

### slug 的命名规则

推荐遵循：

- 使用小写英文单词；
- 单词之间使用短横线 `-`；
- 不使用空格、下划线和特殊符号；
- 不把发布日期写进 slug，除非网站有明确约定；
- 不使用 `new-post`、`test2` 这类无法长期理解的名字；
- 名称足够描述主题，但不要复制整句标题。

标题可以以后调整，slug 应尽量保持稳定。例如标题从“我的第一篇 Firefly 文章”改成“在 Firefly 中新增 Markdown 文章”，仍可保留 `my-first-firefly-article.md`。

### 2. 使用项目脚本生成文章文件

在项目根目录打开终端，执行：

```bash
pnpm new-post -- my-first-firefly-article
```

这个命令会在 `src/content/posts/` 下生成 `my-first-firefly-article.md`，并写入一份基础 frontmatter。脚本也支持直接传入 `.md` 扩展名：

```bash
pnpm new-post -- my-first-firefly-article.md
```

如果文件已经存在，脚本会停止并提示错误。此时不要覆盖旧文章，先换一个 slug，或者确认自己是否应该编辑已有文件。

生成后立即执行：

```powershell
git status --short
Get-Content src/content/posts/my-first-firefly-article.md
```

确认只新增了预期文件，并阅读脚本生成的默认字段。不要假设模板永远与其他教程相同，应以当前仓库生成结果和 `src/content.config.ts` 为准。

### 3. 填写 frontmatter

打开新文件，将顶部内容整理成下面的形式：

```yaml
---
title: "我的第一篇 Firefly 文章"
published: 2026-07-12T19:00:00+08:00
updated: 2026-07-12T19:00:00+08:00
pinned: false
description: "记录我第一次在 Firefly 中新增 Markdown 文章的过程。"
tags: [AI编程, Firefly, Markdown]
category: 博客指南
draft: false
---
```

这些字段的作用是：

| 字段 | 作用 |
| --- | --- |
| `title` | 文章标题，必须是字符串 |
| `published` | 发布时间，决定文章日期和排序 |
| `updated` | 最近修改时间，可选但建议填写 |
| `pinned` | 是否置顶，通常填写 `false` |
| `description` | 首页和搜索结果中使用的摘要 |
| `tags` | 标签数组，多个标签用逗号分隔 |
| `category` | 文章分类 |
| `draft` | 是否草稿，确认发布时填写 `false` |

日期使用完整日期时间时，带上时区，例如 `+08:00`。标签必须是数组，布尔值必须写成 `true` 或 `false`，不要把它们写成带引号的普通文本。

### frontmatter 字段怎样填写得更准确

#### `title`

标题应直接说明文章交付的结果。避免只有“学习记录”“一些思考”这类缺少主题的信息。标题可以包含中文、空格和标点，因为它与文件 slug 是两个不同概念。

#### `published` 与 `updated`

`published` 表示首次发布时间，`updated` 表示最近一次有意义的内容更新。只修复一个错别字时不一定需要更新日期；大幅补充步骤、修正结论或更新版本信息时应更新。

日期应是真实、合法且带有明确时区的值。未来时间可能让文章暂时不出现在列表中。

#### `description`

摘要应该回答“这篇文章帮助读者完成什么”。建议使用一到两句话，包含主题、主要动作和最终结果，不要完全复制标题，也不要堆砌标签。

#### `tags` 与 `category`

标签描述文章涉及的多个主题，分类表示文章最主要的归属。优先复用站点已经存在的写法，避免同时创建“AI编程”“AI 编程”“ai编程”等只差空格或大小写的重复标签。

#### `draft` 与 `pinned`

写作过程中可以使用 `draft: true`，避免未完成文章进入公开列表；准备发布时再改为 `false`。`pinned` 会影响文章排序，应只用于真正需要长期置顶的内容，不要把它当作“让新文章出现”的修复手段。

### 4. 写一份最小可用正文

先不要追求很长。可以从一个标题、一个说明段落和一组检查项开始：

```markdown
# 我的第一篇 Firefly 文章

这篇文章用于验证 Markdown 内容能够被 Firefly 读取并显示。

## 我完成了什么

- 创建了 Markdown 文件
- 填写了 frontmatter
- 在本地打开了文章页面
```

文章文件中的第一个 `#` 通常会被作为正文标题，文章页面本身已经有文章主标题。为了避免标题重复，也可以直接从段落或 `##` 小节开始。写作时优先保证内容清楚，再考虑图片、代码块、表格和 Mermaid 图表。

### 用层级组织正文

Markdown 标题应该形成连续层级：

```markdown
## 一级章节

### 章节中的步骤

#### 某一步的补充说明
```

不要为了字体大小从 `##` 直接跳到 `####`。标题是文章结构，不只是视觉样式，也会影响目录和可访问性。

### 常用 Markdown 元素

列表适合并列项：

```markdown
- 第一项
- 第二项
```

有序列表适合必须按顺序执行的步骤：

```markdown
1. 打开项目目录。
2. 创建文章文件。
3. 启动开发服务器。
```

引用适合强调结论或引用提示：

```markdown
> 一次只修改一个变量，并为结果定义验收标准。
```

行内代码用于命令、字段和路径，例如 `` `pnpm check` ``。多行代码应使用代码围栏，并标注语言：

````markdown
```powershell
pnpm check
```
````

表格适合字段对照，不适合放很长的步骤或大段文字。移动端宽度有限，列数应尽量少。

### 5. 需要插入图片时再添加资源

图片可以放在文章相邻的图片目录中，例如：

```text
src/content/posts/
├── images/
│   └── my-first-firefly-article.png
└── my-first-firefly-article.md
```

在 Markdown 中引用：

```markdown
![文章验证截图](./images/my-first-firefly-article.png)
```

图片文件名同样建议使用小写英文和短横线。先让无图片版本通过检查，再一次添加一张图片并重新验证，出错时更容易定位。

### 图片不仅要能显示

一张文章图片还应满足：

- 与相邻段落有明确关系，不是纯装饰堆砌；
- 使用描述内容的替代文本，而不是“图片1”；
- 文件名稳定，不包含临时截图编号和个人路径；
- 文件大小适合网页，不上传未经处理的超大原图；
- 不包含密码、Token、邮箱、私人仓库或本地隐私信息；
- 在桌面和手机宽度下都不会横向溢出；
- 确认拥有使用和发布该图片的权利。

替代文本应描述读者从图中获得的关键信息：

```markdown
![Firefly 本地文章页面显示新增标题和正文](./images/my-first-firefly-article.png)
```

如果图片只是重复正文、无法帮助理解，可以不添加。配图数量应服务于阅读节奏，不需要每个小节都放一张。

### 8. 添加内部链接和外部链接

链接文本应说明目标内容：

```markdown
[安装 AI 编程工具](/posts/install-required-environment/)
```

避免只写“点击这里”。内部文章链接应使用站点真实路径，添加后实际点击验证。外部链接要检查协议、域名和内容是否仍然适用，不要把搜索结果页或带有私人查询参数的地址直接写入文章。

修改 slug 后要同步检查其他文章是否仍链接到旧地址。可以搜索旧 slug：

```powershell
rg -n "my-first-firefly-article" src/content/posts
```

### 9. 让 AI 按轮次协助写作

不要一次要求 AI“帮我写一篇完整文章并保证全部正确”。可以分四轮：

#### 第一轮：确认范围

```text
我的读者是已经能在本地运行 Firefly 的初学者。
文章只解决“新增第一篇 Markdown 文章”。
请复述范围，指出仍缺少的信息，不要开始写正文。
```

#### 第二轮：生成提纲

```text
请按“问题、环境、步骤、验收、排错”生成提纲。
每个小节只解决一个问题，避免展开部署和主题开发。
```

#### 第三轮：根据真实记录扩写

向 AI 提供实际文件路径、执行命令、页面结果和错误。要求它只根据这些事实整理内容，不虚构截图、版本和测试结果。

#### 第四轮：审阅

```text
请检查这篇文章是否存在：步骤跳跃、重复段落、未解释术语、
无法验证的结论、错误路径、缺少验收标准或泄露隐私的信息。
只列出问题和建议，不直接重写全文。
```

这种分轮方式让你保留内容控制权，也更容易识别 AI 的错误假设。

### 10. 在本地页面查看文章

启动开发服务器：

```bash
pnpm dev
```

打开终端显示的 `Local` 地址，再拼接文章 slug：

```text
http://127.0.0.1:24321/posts/my-first-firefly-article/
```

不要只检查详情页。首页、归档页和标签页也应该能找到这篇文章。如果文章没有出现，先确认 `draft: false`、文件位置、文件名和日期是否正确。

阅读详情页时检查：

- 页面主标题是否只出现一次；
- 标题层级是否形成清晰目录；
- 代码块是否完整，没有把后续正文吞进去；
- 列表、表格和引用是否正常渲染；
- 中文标点、英文单词和行内代码之间是否容易阅读；
- 图片替代文本、标题和说明是否准确；
- 内部与外部链接能否到达预期页面；
- 手机宽度下表格和代码块是否可阅读。

不要只在编辑器中检查 Markdown 源码。最终读者看到的是渲染后的页面。

### 11. 用 Git 检查改动范围

文章显示正常后，执行：

```bash
git status --short
git diff -- src/content/posts/my-first-firefly-article.md
git diff --check
```

确认 Git 只显示计划中的 Markdown 文件和明确添加的图片，不要把构建产物、临时截图或编辑器缓存一并加入文章。

对于新文件，普通 `git diff` 可能不会显示尚未跟踪的正文内容，因此还要结合 `git status --short` 和编辑器检查文件。准备提交时应显式选择文章与配图，不要默认加入整个工作区。

### 12. 从草稿变成可发布文章

可以使用下面的发布顺序：

```text
draft: true 完成写作
→ 本地直接访问文章路径检查
→ 检查日期、摘要、分类和标签
→ 点击所有链接并检查图片
→ 改为 draft: false
→ 检查首页、归档、标签和详情页
→ pnpm check
→ pnpm build
→ git diff --check
```

发布前再读一次文章开头和结尾：开头应明确读者能得到什么，结尾应告诉读者怎样验收或继续下一步，而不是突然结束。

## 如何验证完成

按下面的最小清单逐项确认：

- [ ] 文章文件位于 `src/content/posts/`；
- [ ] 文件扩展名是 `.md` 或 `.mdx`；
- [ ] slug 只使用稳定、清晰的英文文件名；
- [ ] frontmatter 的开始和结束标记都是 `---`；
- [ ] `title`、`published`、`description`、`tags`、`category` 和 `draft` 已填写；
- [ ] `tags` 是数组，`draft` 是布尔值；
- [ ] 文章正文能够表达一个明确主题；
- [ ] `draft: false` 的文章能在首页或归档页找到；
- [ ] 文章详情页能够通过 slug 打开；
- [ ] 页面没有显示 Markdown 原始标记或空白内容；
- [ ] `pnpm check` 执行成功；
- [ ] `pnpm build` 执行成功；
- [ ] `git diff --check` 执行成功；
- [ ] `git diff` 中没有无关文件。

推荐按这个顺序执行验证：

```bash
pnpm check
pnpm build
git diff --check
```

只要文章页面、本地构建和 Git 差异都符合预期，这篇文章就已经完成了“可访问、可构建、可继续修改”的最小闭环。

### 内容发布前检查表

- [ ] 标题准确描述结果，没有夸大。
- [ ] 摘要能独立说明文章价值。
- [ ] 术语首次出现时已经解释。
- [ ] 命令、路径和字段来自当前项目。
- [ ] 示例与正文结论一致。
- [ ] 没有虚构执行结果或成功截图。
- [ ] 没有个人信息、凭据和临时测试内容。
- [ ] 引用资料已经正确标明来源。
- [ ] AI 生成内容已经由自己审阅和验证。

## 失败时先检查什么

### `pnpm new-post` 提示文件已存在

说明目标 slug 已经被使用。先执行：

```bash
Get-ChildItem src/content/posts -File
```

确认是要编辑旧文章，还是换一个新的文件名。不要为了绕过提示直接删除旧文件。

### `pnpm check` 报 frontmatter 错误

先检查 `---` 是否成对，`published` 是否是合法日期，`tags` 是否使用数组格式，`draft` 是否使用布尔值。再检查 YAML 中是否有未闭合的引号。

还要核对错误路径。有时真正出错的是工作区中另一篇正在编辑的文章。先修复第一条错误，不要同时重写多个 frontmatter。

### 文章详情页是 404

检查文件是否真的位于 `src/content/posts/`，浏览器地址中的 slug 是否与文件名完全一致，并确认开发服务器已经重新加载了新文件。文件名中的大小写和短横线都可能影响路径。

如果文件刚创建，可以查看终端是否在内容同步时报告错误。frontmatter 无效时，文章可能没有成功进入内容集合，因此问题不一定来自路由本身。

### 首页没有文章，但详情页可以打开

检查 `draft` 是否为 `false`，再检查发布时间是否晚于当前时间。草稿文章可能不会出现在公开文章列表中，未来时间也可能影响排序和展示。

然后查看分类、筛选条件和分页。文章可能已经进入集合，但不在当前首页第一页。使用归档页或站点搜索查找标题，不要重复创建同一文章。

### 页面显示但图片加载失败

确认图片路径是相对于 Markdown 文件的 `./images/...`，文件名大小写完全一致，图片确实保存在仓库中。不要直接引用电脑上的 `E:/` 路径。

再检查 `git status --short` 是否显示图片文件。编辑器能读取电脑上的图片，不代表它已经位于仓库目录并会进入构建产物。

### Markdown 从某个代码块开始显示异常

检查代码围栏是否成对。开始使用三个反引号，结束也必须使用三个反引号，并单独占一行。代码块内部如果还要展示 Markdown 围栏，可以让外层使用四个反引号。

终端或构建日志中的第一个解析错误通常更接近真正原因，不要只根据页面最下方的异常位置判断。

### 链接点击后进入 404

内部链接要检查目标文章文件名与实际生成路径，注意开头和结尾斜杠。外部链接则在浏览器中直接验证目标页面。

如果链接指向尚未发布的草稿文章，应明确决定是否保留，避免线上读者进入不可访问页面。

### 构建失败或页面空白

先查看终端中的第一条错误，定位到文件和行号。把完整错误、刚刚修改的 Markdown 文件和 `git diff` 一起交给 AI 分析；不要只说“页面坏了”。修正后重新运行 `pnpm check` 和 `pnpm build`。
