> 首页已经能把文章露出来了。
> 下一步，是让用户点击文章之后，进入一个真正适合阅读的页面：能渲染 Markdown，能显示目录，也能继续跳到其他文章。

---

上一篇我们做了首页体验。

首页的任务是让用户知道：

- 有哪些文章
- 可以按哪些阶段读
- 可以按哪些工具查
- 点击哪里能继续阅读

但首页只是入口。

真正的内容体验发生在文章页。

所以这一篇要做阶段四里的关键一步：

> **做文章阅读页。**

如果文章页做不好，首页再漂亮也只是一个导航壳。

内容站最终还是要让用户舒服地读完一篇文章。

---

## 文章页要解决什么问题

文章页不是简单地把一大段文字显示出来。

它至少要解决 5 个问题：

| 问题 | 文章页要做的事 |
|------|----------------|
| 文章从哪里来 | 根据 URL 找到对应文章 |
| 正文怎么显示 | 把 Markdown 渲染成 HTML |
| 长文怎么导航 | 自动生成文章目录 |
| 用户怎么换文章 | 保留阶段、工具和最近文章入口 |
| 页面怎么保持稳定 | 宽度、代码块、表格和图片不能撑破布局 |

这几个问题都很实际。

尤其是长文。

如果没有目录，用户读到中间很容易失去方向。

如果没有侧栏，用户读完一篇也不知道下一篇去哪。

所以文章页不是"显示正文"这么简单。

它是内容站的核心阅读空间。

---

## 第一步：用 URL 表示当前文章

首页点击文章后，需要进入对应文章页。

本站使用的是 query 参数：

```text
/?article=write-first-code-with-ai
```

这里的 `write-first-code-with-ai` 就是文章的 `slug`。

slug 可以理解为文章的稳定地址名。

它比中文标题更适合放进 URL。

比如：

| 标题 | slug |
|------|------|
| 用 AI 写第一行代码 | `write-first-code-with-ai` |
| 搭项目基础：让空文件夹变成能运行的网页项目 | `setup-project-foundation` |
| 做首页体验：让文章、阶段和工具索引在一个页面里站稳 | `build-homepage-experience` |

这样做的好处是：

1. 用户可以复制文章链接
2. 刷新页面后仍然能打开同一篇文章
3. 首页和侧栏都可以用同一套跳转逻辑
4. 上一篇 / 下一篇也可以根据文章顺序计算

这就是为什么文章页第一步不是写样式，而是把 URL 和文章数据连起来。

---

## 第二步：根据 slug 找到文章

有了 URL 里的 `article` 参数，代码就可以从文章列表里找对应文章。

逻辑大概是：

```ts
const params = new URLSearchParams(window.location.search);
const selectedSlug = params.get('article');
const selectedArticle = articles.find((article) => article.slug === selectedSlug);
```

如果找到了，就显示文章页。

如果没有找到，就可以显示空状态，或者回到首页。

这里有一个细节：

> 文章页不应该依赖标题查找文章。

标题是给用户看的，后续可能会调整。

slug 才是稳定标识。

这和前面讲阶段数据时的原则一样：

- `title` 给用户看
- `slug` 给代码和 URL 用

把这两件事分开，项目会更稳。

---

## 第三步：用 Markdown 写正文

本站的文章正文不是直接写在 React 组件里，而是写成 Markdown 文件。

这样更适合持续写作。

一篇文章大概长这样：

```md
---
title: 用 AI 写第一行代码
stage: 对话篇
tools:
  - ai编程助手
  - ai IDE/编辑器
updatedAt: 2026-07-03
---

> 这一篇讲如何把一句模糊需求，变成 AI 能执行的清楚指令。

## 为什么第一行代码不重要

真正重要的是你能不能说清楚要做什么。
```

前面的 frontmatter 负责元数据。

后面的 Markdown 负责正文。

这样有几个好处：

1. 写文章不用碰复杂组件
2. 标题、列表、表格、代码块都更自然
3. 文章可以被统一解析和渲染
4. 后续新增文章成本很低

对内容站来说，Markdown 是一个很合适的起点。

---

## 第四步：渲染 Markdown

Markdown 不能直接显示成最终网页。

它需要被解析成 HTML 结构。

React 项目里可以使用类似 `react-markdown` 的库。

渲染思路大概是：

```tsx
<ReactMarkdown remarkPlugins={[remarkGfm]}>
  {article.body}
</ReactMarkdown>
```

这里的 `remarkGfm` 用来支持 GitHub 风格 Markdown。

比如：

- 表格
- 任务列表
- 删除线
- 更接近常见 Markdown 的语法表现

这一步完成后，文章里的：

```md
## 小结
```

就能变成页面上的二级标题。

文章里的表格、代码块、引用，也能按对应样式显示出来。

---

## 第五步：给正文一个稳定容器

Markdown 渲染出来以后，还需要样式。

否则正文会很粗糙。

文章正文容器要重点处理这些元素：

| 元素 | 要注意什么 |
|------|------------|
| 标题 | 层级清楚，间距稳定 |
| 段落 | 行高舒服，不要太挤 |
| 链接 | 颜色明显，符合全站风格 |
| 引用 | 和正文区分开，但不要太重 |
| 表格 | 能换行，不要撑破页面 |
| 代码块 | 可读、可复制、不要横向炸开 |
| 图片 | 最大宽度不能超过正文容器 |

这一点非常重要。

因为文章页最容易出问题的不是普通段落，而是代码块、表格和长链接。

如果不处理，它们很容易把正文卡片撑宽。

所以文章页 CSS 要有一个底线：

> **任何正文内容都不能破坏阅读宽度。**

---

## 第六步：自动生成文章目录

技术文章通常比较长。

所以文章页需要文章目录。

目录可以从正文里的标题提取。

比如 Markdown 里有：

```md
## 第一步：用 URL 表示当前文章
## 第二步：根据 slug 找到文章
### 为什么不用标题查找
```

页面就可以生成：

- 第一步：用 URL 表示当前文章
- 第二步：根据 slug 找到文章
  - 为什么不用标题查找

目录不需要作者手写。

它应该从 Markdown 标题自动计算。

这样一篇文章只要正常使用 `##` 和 `###`，目录就会自动出现。

这比手动维护目录更可靠。

手动目录最常见的问题是：

- 改了标题但忘记改目录
- 删除小节但目录还在
- 新增小节但目录漏掉

自动目录可以避免这些问题。

---

## 第七步：目录点击要能跳转

只有目录还不够。

目录里的每一项应该能点击跳转到正文对应位置。

这就需要给标题生成稳定的 `id`。

例如：

```html
<h2 id="第一步-用-url-表示当前文章">
  第一步：用 URL 表示当前文章
</h2>
```

然后目录链接指向：

```html
<a href="#第一步-用-url-表示当前文章">
  第一步：用 URL 表示当前文章
</a>
```

这样用户读长文时，可以快速跳到想看的部分。

参考成熟技术博客时，你会发现文章目录通常是阅读页非常重要的一部分。

它不是装饰，而是长文导航。

---

## 第八步：左右侧栏不要消失

文章页的中间是正文。

但左右两侧也很重要。

本站文章页保持了这样的结构：

| 位置 | 模块 |
|------|------|
| 左侧 | 学习档案、文章目录 |
| 中间 | 文章正文 |
| 右侧 | Learning Stages、Tool Index、最近文章 |

这让用户在阅读时仍然知道：

- 当前网站有怎样的学习路径
- 这篇文章在整体内容里的位置
- 读完以后还可以去哪里

侧栏最好在桌面端保持 sticky。

也就是页面往下滚动时，文章目录和右侧索引不会马上消失。

这样读长文时更方便。

但在手机屏幕上，不应该强行三栏。

移动端应该自然变成单列布局。

---

## 第九步：上一篇和下一篇

当文章有了顺序以后，就可以增加上一篇 / 下一篇。

它们可以根据 `articles` 数组的顺序计算。

比如当前文章是第 8 篇：

- 第 7 篇就是上一篇
- 第 9 篇就是下一篇

这比手动在每篇文章里写链接更省心。

文章页适合在两个地方放导航：

1. 标题右侧：让用户一进来就知道前后关系
2. 正文底部：读完以后自然进入下一篇

这会让学习路线更连续。

尤其是阶段四这种实战记录，很适合让用户按顺序读下去。

---

## 如何让 AI 帮你做文章阅读页

你可以这样描述任务：

```text
请完成 M4：文章阅读页。

目标：
点击首页文章后，进入独立文章页，渲染 Markdown 正文并显示文章目录。

要求：
1. 使用 URL query 参数 article 表示当前文章 slug
2. 根据 slug 从文章注册表里找到对应文章
3. 使用 Markdown 渲染正文
4. 支持表格、代码块、引用、列表和图片
5. 根据 h2 / h3 自动生成文章目录
6. 目录点击后能跳转到正文对应标题
7. 文章页保留学习档案、Learning Stages、Tool Index 和最近文章
8. 添加上一篇 / 下一篇导航
9. 确保正文宽度稳定，代码块和表格不能撑破布局
10. 完成后运行 npm run build 验证
```

这里的重点是：

> 文章页既要能读，也要能导航。

只渲染 Markdown 还不够。

只有把目录、侧栏、前后篇、宽度稳定一起做好，文章页才算真正可用。

---

## 验收文章阅读页

文章页完成后，可以这样检查：

1. 首页点击文章标题是否进入文章页？
2. 首页点击继续阅读是否进入文章页？
3. 直接打开 `/?article=xxx` 是否能显示对应文章？
4. Markdown 标题、段落、表格、代码块是否正常渲染？
5. 左侧文章目录是否自动生成？
6. 点击目录是否能跳到对应小节？
7. 右侧 Learning Stages 和 Tool Index 是否还能跳转？
8. 上一篇 / 下一篇是否按文章顺序工作？
9. 表格、代码块、长链接是否不会撑破正文宽度？
10. `npm run build` 是否通过？

这些检查比单纯看页面好不好看更重要。

因为阅读页是内容站最核心的地方。

它一旦不稳，后面写再多文章都会受影响。

---

## 常见问题

### 1. 为什么不直接把文章写进组件？

因为文章会越来越多。

如果都写进 React 组件，后续维护会很痛苦。

Markdown 更适合内容写作。

组件更适合页面结构和交互。

两者分开，项目更清楚。

### 2. 为什么文章目录只取 h2 和 h3？

第一版取 h2 和 h3 就够了。

h1 通常是文章标题，不需要进目录。

h4 以下层级太深，目录会变复杂。

技术文章里，二级和三级标题通常已经能表达结构。

### 3. 为什么用 query 参数，不用真正的路由？

第一版用 query 参数更简单。

它不需要引入完整路由系统，也能满足：

- 可复制链接
- 刷新不丢文章
- 点击文章跳转
- 根据 slug 显示内容

等项目变复杂以后，再考虑正式路由也不晚。

---

## 小结

这一篇做的是文章阅读页。

它把首页的"继续阅读"真正接到了内容体验上。

你需要记住这条线：

1. URL 里的 `article` 参数表示当前文章
2. 用 slug 从文章列表里找到正文
3. 文章用 Markdown 写，frontmatter 写元数据
4. 使用 Markdown 渲染器把正文显示出来
5. 从 h2 / h3 自动生成文章目录
6. 目录点击能跳转到正文小节
7. 文章页保留左右侧栏和前后篇导航
8. 代码块、表格、图片都不能破坏正文宽度

做到这里，网站就真正具备了内容站的核心能力：

> **用户能从首页进入文章，并在文章页里稳定地读完内容。**
