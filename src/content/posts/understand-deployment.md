> 发布篇的第一步，不是马上点部署按钮，而是先理解：部署到底把什么东西，放到了哪里，为什么别人能通过链接访问。

---

上一篇我们结束了调试篇。

我们学会了遇到报错时怎么整理现场、怎么看构建失败、怎么查白屏、怎么排查布局、怎么读 Git 状态，也学会了把一次 Debug 变成复盘。

现在项目已经能写、能读、能调试。

下一步就来到第六个阶段：

> **发布篇：让世界看到。**

这一步听起来很兴奋。

因为它意味着：

> 你的网页不再只是本地文件夹里的东西，而是可以通过一个链接被别人打开。

但在正式部署之前，先别急着选平台。

先搞清楚一个基础问题：

> **部署到底是在做什么？**

---

## 本地能打开，不等于已经发布

现在这个项目可以在本地运行。

比如开发服务器地址是：

```text
http://127.0.0.1:5177/
```

这个地址只能在你的电脑上访问。

`127.0.0.1` 的意思就是本机。

别人就算拿到这个链接，也打不开你的网页。

因为这个地址指向的是他自己的电脑，不是你的电脑。

所以本地运行和发布上线是两件事：

| 状态 | 谁能访问 |
|------|----------|
| 本地运行 | 主要是你自己 |
| 部署上线 | 任何拿到链接的人 |

发布的目标就是：

> 把本地项目变成一个公网可访问的网站。

---

## 部署不是把整个项目文件夹丢出去

很多新手会以为部署就是：

> 把整个项目文件夹上传到网上。

其实不是。

对于 Vite + React 这类前端项目，真正发布出去的通常不是源码本身，而是构建后的静态文件。

也就是运行：

```bash
npm run build
```

之后生成的：

```text
dist/
```

这个 `dist` 文件夹里通常包含：

- `index.html`
- 打包后的 JavaScript
- 打包后的 CSS
- 图片、字体等静态资源

浏览器真正需要的是这些文件。

它不需要你的 `src` 源码，也不需要你的 TypeScript 文件。

所以部署的核心可以理解成：

> **把构建产物放到一个公网服务器上。**

---

## npm run build 在发布里扮演什么角色

前面我们反复运行：

```bash
npm run build
```

在开发阶段，它主要用于检查项目是否能成功打包。

到了发布阶段，它还有另一个作用：

> **生成可以部署的生产文件。**

开发时，Vite dev server 会帮你热更新、显示错误、快速刷新。

但真正上线时，用户不会访问你的开发服务器。

用户访问的是构建后的静态文件。

所以发布前一定要确认：

1. `npm run build` 能通过
2. `dist` 文件夹能生成
3. 生成后的页面能正常打开
4. 文章、样式、图片路径都正常

如果构建都过不了，就不要急着部署。

先回到调试篇，把错误修好。

---

## 静态网站是什么意思

本站目前是一个静态网站。

静态网站不是说它不能交互。

它的意思是：

> 用户访问网站时，服务器主要返回已经构建好的 HTML、CSS、JavaScript 和资源文件。

它不需要自己写后端服务器。

也不需要数据库。

本站的文章数据在构建时已经打包进前端代码里。

用户打开页面后，浏览器加载 JavaScript，再根据 URL 显示对应文章。

这就是为什么它适合部署到：

- GitHub Pages
- Vercel
- Netlify

这些平台都很适合托管静态前端项目。

---

## GitHub Pages、Vercel、Netlify 的区别

这三个名字经常一起出现。

新手不用一开始就深入比较。

先记住它们都能做一件事：

> 帮你把静态网站发布到公网。

简单理解：

| 平台 | 适合场景 |
|------|----------|
| GitHub Pages | 项目已经放在 GitHub，想要一个免费静态站 |
| Vercel | 前端项目部署体验好，自动构建方便 |
| Netlify | 静态站和表单等功能支持成熟 |

对于本站这种 Vite + React 内容站，三者都可以。

第一版不需要纠结最完美选择。

更重要的是先理解发布流程：

1. 代码提交到 Git
2. 推送到远程仓库
3. 平台读取代码
4. 平台运行构建命令
5. 平台发布 `dist`
6. 你得到一个公网链接

这是发布的主线。

---

## 发布前要检查什么

发布前不要只看首页能不能打开。

要做一次小型检查。

至少确认：

1. `npm run build` 通过
2. 首页能正常显示
3. 文章页能通过 `?article=slug` 打开
4. 上一篇 / 下一篇能跳转
5. Learning Stages 能跳转
6. Tool Index 能跳转
7. 图片路径正常
8. 页面没有明显横向溢出
9. Git 状态清楚
10. 准备好提交和推送

发布不是开发结束的魔法按钮。

它更像是一次正式交付前的检查。

如果本地已经乱了，部署平台只会把混乱暴露出来。

---

## URL 和路由要特别注意

本站使用的是 query 参数文章页：

```text
/?article=debug-read-git-status
```

这对静态部署比较友好。

因为所有页面都从 `index.html` 进入，然后由前端根据 query 参数显示文章。

如果你使用真正的路径路由，比如：

```text
/articles/debug-read-git-status
```

静态部署时就要额外配置 fallback。

否则刷新页面可能出现 404。

所以第一版用 query 参数有一个好处：

> 部署配置更简单。

等后续项目更成熟，再考虑正式路由也不晚。

---

## 图片和资源路径要注意

发布后最常见的问题之一是：

> 本地图片正常，上线后图片不显示。

原因通常是路径问题。

在 Vite 项目里，放在 `public` 里的资源可以用绝对路径引用：

```md
![构建截图](/assets/m1-build-success.svg)
```

构建后它会对应到网站根路径下的：

```text
/assets/m1-build-success.svg
```

发布前要检查：

- 图片文件是否真的在 `public/assets`
- Markdown 里引用路径是否正确
- 文件名大小写是否一致
- 部署后的 base path 是否影响资源路径

如果平台把网站部署到子路径，资源路径可能要额外处理。

第一版可以优先选择根路径部署，减少复杂度。

## 让 AI 帮你准备发布检查

你可以这样问 AI：

```text
请帮我做发布前检查。

项目是 Vite + React + TypeScript 静态内容站。

请检查：
1. npm run build 是否通过
2. dist 是否生成
3. 首页和文章页 URL 是否正常
4. 图片资源路径是否可能有问题
5. query 参数文章页是否适合静态部署
6. git status 是否清楚
7. 如果要部署到 GitHub Pages / Vercel / Netlify，需要准备哪些步骤

不要直接部署，先给我检查清单和风险点。
```

这里最后一句很重要：

> 不要直接部署，先给我检查清单和风险点。

发布前先检查，比直接推上去再修更稳。

---

## 发布不是一次性的

很多人以为发布上线是终点。

其实不是。

发布之后，项目才开始进入另一个阶段：

- 发现线上问题
- 修复后重新部署
- 新增文章
- 改样式
- 改导航
- 改性能
- 收集反馈
- 持续迭代

所以发布不是"最后一步"。

它更像是：

> 让项目第一次走出本地电脑。

从那以后，每一次提交、构建和部署，都会形成一个小循环。

这就是持续迭代。

---

## 常见问题

### 1. 我需要买服务器吗？

第一版不需要。

本站这种静态前端项目，可以先用 GitHub Pages、Vercel 或 Netlify。

它们足够完成学习和展示。

### 2. dist 文件夹要提交到 Git 吗？

通常不需要。

更常见的做法是提交源码，让部署平台自动运行 `npm run build`。

不过 GitHub Pages 的某些配置方式也可能需要发布构建产物。

具体取决于部署方案。

第一步先理解流程，不急着定死。

### 3. 构建通过就一定能上线成功吗？

不一定。

构建通过只是前提。

上线还可能遇到：

- 资源路径问题
- base path 问题
- 平台构建配置问题
- Node 版本问题
- 仓库权限问题

但构建不通过，基本就不要继续部署。

---

## 小结

发布篇的第一步，是理解部署。

你需要记住这条线：

1. 本地地址 `127.0.0.1` 只给自己访问
2. 发布是让别人通过公网链接访问网站
3. Vite 项目通常部署的是 `npm run build` 生成的 `dist`
4. 本站是静态内容站，适合 GitHub Pages、Vercel、Netlify
5. 发布前要检查构建、文章页、导航、图片和 Git 状态
6. query 参数文章页对静态部署比较友好
7. 发布不是终点，而是持续迭代的开始

部署不是神秘仪式。

它只是把本地项目的构建产物放到一个别人能访问的地方。

> **本地能跑，是给自己看；部署上线，是让世界看到。**
